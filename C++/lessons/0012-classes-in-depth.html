构造函数:对象被创建时的初始化

一个 class 里可以有一个跟类同名的特殊函数,它叫构造函数(constructor)。 每次你创建这个类的对象,它会自动跑一遍——专门用来给成员变量赋初值。你不用手动调用它,创建对象的那一刻 C++ 就替你调了。 

class Vec3
{
public:
    float x, y, z;

    // 构造函数:名字 == 类名,没有返回类型
    Vec3(float px, float py, float pz)
        : x(px), y(py), z(pz)   // ← 初始化列表:把成员一个个填好
    {
    }
};

Vec3 p(1.0, 2.0, 0.0);   // 创建对象 → 构造函数自动跑 → x=1, y=2, z=0

注意两个特征:没有返回类型(连 void 都不写),名字和类一模一样。看到这两点同时出现,就知道"这是构造函数"。

冒号后面那串 : x(px), y(py), z(pz) 叫成员初始化列表(member initializer list)。读法很简单:x(px) 就是"用 px 去初始化成员 x"——给刚诞生的对象定好初值。

析构函数与 this:对象的收尾与"我自己"

和构造函数成对的是析构函数(destructor):写法是 ~ 加类名,例如 ~Vec3()。 当对象销毁(离开作用域、被 delete)时它自动跑,用来做清理——关文件、释放内存。这正是第 8 课 RAII 的机制:谁创建、谁负责收尾,自动发生。 

class ScopedTimer
{
public:
    ScopedTimer()  { startClock(); }    // 构造:对象出生 → 开始计时
    ~ScopedTimer() { stopClock();  }    // 析构:对象销毁 → 停止计时
};

另一个高频词是 this。在成员函数里,this 是一个指向"当前这个对象自己"的指针(承接第 7 课的指针概念)。因为它是指针,所以访问成员要用 -> 而不是 .: 

class Vec3
{
public:
    float x, y, z;
    void scale(float s)
    {
        this->x *= s;   // this 指向调用 scale 的那个对象;-> 因为 this 是指针
        this->y *= s;
        this->z *= s;
    }
};

多数时候 this-> 可以省略(直接写 x *= s; 编译器懂),但你读真实代码会经常看到它,知道它的意思是"我这个对象的 x"就够了。

const 成员函数:一句"我保证不改你"

看这一行,注意函数末尾多了个 const: 

float length() const     // ← 结尾的 const:这个函数承诺"不修改对象"
{
    return sqrtf(x*x + y*y + z*z);   // 只读 x/y/z,不改它们
}

末尾的 const 是函数对调用者的承诺:"调我不会改变这个对象的任何成员。"这呼应第 3 课的 const(只读)和第 7 课的指针——一个 const 成员函数里,this 变成了"指向只读对象的指针",所以它碰不了成员。

读代码的好处:看到 length() const、getName() const,你立刻知道这是个"查询"函数,只读不写,放心调用;没有 const 的(如 scale()、normalize())则可能改对象。这是一条非常省事的速读线索。

继承 + virtual / override:派生你自己的 Actor

重头戏来了。继承(inheritance)让一个类"接手"另一个类的全部东西,再加上自己的。写法是冒号加 public 基类名: 

class AEnemy : public AActor   // AEnemy「是一个」AActor,并继承它的一切
{
    // 这里只写「敌人」额外/不同的部分,
    // AActor 已有的 Transform、组件、生命周期机制全部自动继承过来
};

读法:AEnemy 是一个 AActor(子类 / 派生类),AActor 是它的基类 / 父类。子类白拿父类的成员变量和函数,只需补上自己独有的差异。这就是"基于一个基础类型,派生出你自己的定制类"——和你在蓝图里基于 Actor 新建一个子 Blueprint 是同一回事。

但派生不只是"加东西",更重要的是重写父类的某段逻辑。父类把一个函数标成 virtual("虚函数",意思是"子类可以替换我"),子类用 override 写一个同名同签名的版本顶替它: 

class AActor          // —— 基类(引擎提供)——
{
public:
    virtual void Tick(float DeltaTime);   // virtual:声明「子类可以重写我每帧的逻辑」
};

class AEnemy : public AActor   // —— 你的派生类 ——
{
public:
    void Tick(float DeltaTime) override;   // override:用「我的」Tick 顶替父类那个
};

为什么这能成立?因为引擎内部是用一个父类指针/引用来持有你的对象的(回想第 7 课):它手里拿的是 AActor*,但实际指向的对象是 AEnemy。当它写 Actor->Tick(dt) 时,virtual 保证:运行时真正跑的是对象实际类型的那个 Tick——也就是你写的 AEnemy::Tick。这叫运行时多态(runtime polymorphism)。

这就是你会在 BeginPlay() / Tick() 上反复看到 override 的原因:引擎在某个时刻调 Tick(),而你把自己的逻辑塞进了那个被调用的位置。这跟你在子 Blueprint 里重写(Override)一个父类 Event / Function 是完全一样的机制。(参见 LearnCpp 25.2 对虚函数"解析到最派生版本"的说明。)

读法 父类 AActor 像 Unreal 的"通用 Actor 骨架":Transform、组件、何时 BeginPlay、何时 Tick,引擎都安排好了。你派生一个 AEnemy 并 override 它的 Tick(),等于说"引擎你照常运转,但'每帧干什么'这一步换成我的逻辑"——就是你在蓝图里 Override 父类那个 Event 时做的事。

你会在真实代码里看到(Unreal 风格示意)

把上面拼起来,真实的 Unreal 自定义 Actor 大致长这样(已简化,仅作示意): 

class AEnemy : public AActor
{
public:
    AEnemy()
    {
        PrimaryActorTick.bCanEverTick = true;   // 构造:把自己「立起来」、配好默认值
    }

    virtual void BeginPlay() override;   // 重写父类:游戏开始时引擎调一次

    // 重写父类的每帧入口:引擎每帧调它,你的逻辑在这里跑
    virtual void Tick(float DeltaTime) override;

    float GetHealth() const;   // const:只读查询,不改对象

private:
    float Health;
};

你读这段时该看到的信号:: public AActor(这是个派生 Actor)、构造函数里配默认值、Tick(...) override(这就是该 Actor 每帧真正干活的地方)、GetHealth() const(只读查询)。你不需要会写,但这几个标记一眼扫过,就知道"逻辑在 Tick 里,我要改行为就改它"。

读代码易错点

这几个坑专门绊读代码的人,记住就能少栽:

  • 忘了构造/析构"何时"被调用。 它们没有显式调用点——构造在对象创建那一刻自动跑,析构在对象销毁那一刻自动跑。在代码里找不到 obj.Vec3() 这种调用是正常的,别去找。
  • 把 override 当成"新函数"。 带 override 的函数不是新增功能,而是替换了父类里那个同名虚函数。要理解它,得先去父类看 virtual 版本的"合同"是什么。
  • 漏看 virtual。 父类函数前那个 virtual 容易被忽略,但它是多态的开关:有它,基类指针->Tick() 才会跑到子类版本;没它,跑的就是父类那个死板版本。
  • 看到 -> 慌张。 用 -> 只说明左边是个指针(第 7 课):Actor->Tick(dt) = "对 Actor 指向的那个对象调 Tick"。等价于 (*Actor).Tick(dt),只是更顺手。
  • 把末尾 const 和开头 const 混为一谈。 开头的 const(如 const char *name)修饰参数;结尾的 const 修饰整个成员函数,承诺"不改对象"。位置不同,意思不同。
