内存与生命周期(读者视角)

读代码时,你要能一眼看出:这个对象活在哪儿、活多久、谁负责清掉它。

本课目标(一个收获) 在阅读代码时认出对象"活在哪、活多久"——栈上(自动)对象 vs 堆上 new 出来的对象;并理解 new/delete 和智能指针 unique_ptr/shared_ptr 出现时分别意味着什么(所有权)。注意:现阶段是读懂别人怎么管内存,不是要你独立管。

热身 · 回忆第 7 课(别翻笔记)

先凭记忆答一题,把上次"指针 / ->"的东西捞回来——主动回忆比重看一遍记得牢。

两块地盘:栈 stack 与堆 heap

C++ 里每个对象都得"住"在某个地方。读代码时,你只需要分清两块地盘:

栈(stack):函数运行时临时用的一块空间。你在函数里直接写出来的变量,默认就住在栈上。它的规矩很死板但很省心——离开它所在的 { } 作用域,它自动就没了,不用任何人动手清理。

堆(heap):一块"按需申请、长期持有"的大空间。只有当代码里出现 new(或包了它的工具)时,才会在堆上分配一块内存。它不会自动消失——必须有人显式地、或借助工具,把它释放掉。

栈上对象:出了 { } 就自动销毁

看这段。p 是直接写出来的变量,住在栈上:

void simulate()
{
    Particle p;            // p 在栈上诞生
    p.mass = 2.0f;
    p.update();
}                            // 走到这个 } → p 自动销毁,内存自动还回去

你不用写任何清理代码。函数一结束,p 就随作用域一起蒸发了。绝大多数局部变量都是这样——读到它们,你心里只需标记"短命、自动管理"。

new / delete:堆上手动管理(老代码常见)

当代码需要一个对象"活得比当前函数久",就会用 new 在堆上分配。new 返回的是一个指针(第 7 课那个 *),指向堆上那块内存。用完必须 delete,否则那块内存就一直占着不还。 

Particle* p = new Particle();   // 在堆上分配,p 指向它
p->mass = 2.0f;                      // 用 -> 访问(第 7 课)
p->update();

delete p;                            // 必须手动释放,否则泄漏
p = nullptr;                        // 好习惯:删完置空,避免悬空

读这种代码时,你要做的是给每个 new 找到配对的 delete。配上了,内存被正确回收;配不上,就是下面要讲的麻烦。

读代码易错点:泄漏 / 悬空 / 释放后使用

这三个是手动管内存最经典的坑。你现阶段不必会修,但要能在读代码时认出危险信号。

① 内存泄漏(leak):new 了,却没有任何路径走到对应的 delete。内存一直占着、再也拿不回来。信号:看到 new 后,函数里某条分支(比如提前 return)绕过了 delete。

Cache* c = new Cache();
if (!loadFile()) return;        // 提前 return → 下面的 delete 永远到不了 → 泄漏
delete c;

② 悬空指针(dangling pointer):对象已经被 delete 了,但还有指针指着那块"已经不属于你"的内存。指针本身看起来没问题,指向的东西却没了。

③ 释放后使用(use-after-free):顺着上面那个悬空指针又去读写它。这是 bug 重灾区——结果可能是垃圾值、可能崩溃、也可能"碰巧"正常一阵子再炸。

delete p;          // 这块内存还回去了
p->update();         // ✗ use-after-free:p 已悬空,这里在用一块不归你的内存
读代码时的警惕清单 看到 new → 找它的 delete;看到 delete → 想"这之后还有谁在用这个指针吗";看到一个指针 → 问"它指的东西现在还活着吗"。这三问能帮你嗅出大半内存类 bug。

现代 C++:用智能指针表达"所有权"

为了不再手动配对 new/delete,现代 C++ 用智能指针:它们是包着裸指针的小对象,会在自己(栈上)销毁时自动把堆上的东西删掉。读代码时,它们最大的价值是把"谁负责销毁"写在了类型里。

std::unique_ptr(独占所有权):这块数据归它独占,它一离开作用域就把数据清掉。通常用 std::make_unique 创建,你看不到 delete——因为不需要写。

auto p = std::make_unique<Particle>();  // p 独占这个 Particle
p->mass = 2.0f;
p->update();
// 离开作用域 → p 自动销毁 → 它持有的 Particle 也自动删掉,无需 delete

std::shared_ptr(共享所有权):多处可以共同持有同一块数据,内部维护一个引用计数。每多一个 shared_ptr 指它,计数 +1;每销毁一个,计数 −1。计数归零时,数据才被删掉。

auto mesh = std::make_shared<Mesh>();   // 计数 = 1
auto alias = mesh;                          // 又一个持有者 → 计数 = 2
// 两个都销毁后 → 计数 = 0 → Mesh 才真正被释放

所以读到智能指针时,你不用再去配 delete;你要读的是所有权:unique_ptr 说"只有我一个主人",shared_ptr 说"我们一起,最后一个走的关灯"。

一句话:RAII 智能指针背后的思想叫 RAII——对象自己管理自己的资源。资源(堆内存)的生命周期绑定到一个栈上对象的生命周期:对象一销毁,资源就释放。这就是为什么现代代码里你常常看不到 delete 却没有泄漏。

你会在真实代码里看到

同一个意图,老代码和现代代码长这样——能认出两者、并知道"现代版不需要手动 delete",就达标了:

// —— 老代码:手动 new / delete(危险但常见)——
Texture* tex = new Texture("smoke.exr");
render(tex);
delete tex;                                   // 漏写这行 → 泄漏

// —— 现代代码:unique_ptr 表达独占,自动释放 ——
auto tex2 = std::make_unique<Texture>("smoke.exr");
render(tex2.get());                            // .get() 取出里面的裸指针给旧接口用
// 不写 delete:tex2 出作用域时自动清理
