引用 references —— 给数据起个别名

签名里那个不起眼的 &,决定了函数是在复制一份还是在动你的原件。读懂它,你就读懂了真实代码里 90% 的参数。

本课目标(一个收获) 读懂签名里的 T& 和 const T&:理解"传引用"让函数能改调用方的数据,而 const& 是"只读、不复制";并明白为什么真实引擎代码里参数几乎全是 const&。

热身 · 回忆第 5 课(别翻笔记)

先凭记忆答一题,把上次的循环捞回来——这种"主动回忆"比重看一遍记得牢得多。

引用是什么:同一份数据的"第二个名字"

一个引用(reference)就是某个已经存在的变量的别名。它不是新盒子、不存新数据——它只是给同一个盒子又贴了一张名牌。写法是在类型后面加 &: 

float density = 3.0;        // 真正的盒子,名叫 density
float& alias = density;       // alias 是 density 的别名,不是新盒子

alias = 9.0;                  // 改 alias 就是在改 density
// 现在 density 也是 9.0 —— 它俩自始至终是同一个盒子

引用有两条铁律,读代码时要默认它们成立:

  • 必须在出生时就绑定。 你不能写 float& alias; 然后晚点再说它指谁——声明的那一刻就得给它一个变量去绑。
  • 绑了就不能改绑。 之后无论怎么给 alias 赋值,都是在改它绑的那个原件,而不是让它去当别人的别名。这点跟下一课的"指针"正好相反,先记住。

核心区别:按值传递 vs 按引用传递

这是本课的心脏。函数怎么"拿到"你给的参数,有两种根本不同的方式。

① 按值传递(pass by value)—— 复制一份

参数类型没有 & 时,函数拿到的是你那个值的一份拷贝。它在拷贝上随便改,你的原件纹丝不动:

void tryToModify(float x)   // x 是「调用方那个值」的一份复制品
{
    x = 100.0;             // 只改了复制品
}

float mass = 3.0;
tryToModify(mass);
// mass 还是 3.0 —— 函数动的是拷贝,不是 mass 本人

② 按引用传递(pass by reference)—— 操作原件

参数类型带 & 时,函数拿到的是你那个变量本人的别名。它改参数,就是改你的原件:

void modify(float& x)        // x 是「调用方那个变量」的别名
{
    x = 100.0;             // 直接改原件
}

float mass = 3.0;
modify(mass);
// mass 现在是 100.0 —— 函数透过别名改掉了 mass 本人

就差一个 &,结果天壤之别。读真实代码时,看到参数上的 &,你就要警觉:这个函数可能会改我传进去的东西。这也是它存在的意义——很多函数(尤其名字像 compute…、fill…、get…(out))就是靠引用参数把结果"写回"给调用方的。

对照 HLSL / Blueprint 你在 HLSL 写过 out/inout 参数:函数能把结果写回调用方的变量——那正是 C++ 的 T&。Blueprint 函数的输入引脚也有一个 "Pass-by-Reference" 勾选项,勾上后函数里改它就改到了外面的变量。没有 & 的普通参数则相当于不勾——函数只拿到一份拷贝。

const T&:只读引用 —— 既不复制,又不让你改

这是真实代码里出现频率最高的参数写法。它把上面两种好处合并了:

  • & 让它不复制——直接用原件,省掉拷贝大对象的开销;
  • const 给它上锁——函数只能读、不能写原件,所以调用方很安全。
float computeArea(const Geometry& geo)   // 只读地借用 geo
{
    return geo.surfaceArea();   // 能读它的数据
    // geo.clear();  ← 编译器会拒绝:const 不准改
}

读这一行你脑子里要翻译成:"我借你的 geo 看看,保证不复制、也保证不动它。" 正因为这个承诺,你把任何东西传进去都放心。LearnCpp 12.6 直接给出一条经验法则:对于复制代价高的类型,默认用 const T& 传。

为什么真实代码里参数几乎全是 const&? 三个原因叠加:

  • 省钱。 复制一个 std::string、一个向量数组、一份几何,都要真实地搬数据;const& 零拷贝。
  • 安全。 const 让编译器替你把关,函数碰不了你的原件,读代码时你能立刻排除"它会不会偷偷改我数据"的担忧。
  • 通用。 const T& 还能接收临时值和字面量(比如直接传一个表达式的结果),而非 const 的 T& 不行——这点 LearnCpp 12.4 讲得很细。

你会在真实代码里看到

下面是 Unreal 风格的一段真实签名(引擎里这种写法到处都是)。别管细节,只盯着参数上的 const … &:

// 计算一个 actor 的包围盒(Unreal 风格)
bool ComputeBounds(const AActor& actor, FBox& box)
{
    box = actor.GetComponentsBoundingBox();   // 读 actor(只读),把结果写进 box
    return box.IsValid;
}

这一行签名里两个 & 各司其职,读懂它你就读懂了引擎代码的惯例:

  • const AActor& actor —— 输入:整个 actor,只读借用,不复制(大对象,复制会很贵)。
  • FBox& box —— 输出:没有 const,所以函数会写它;调用方传一个空盒子进去,函数把算好的包围盒填回来。返回值 bool 只表示成功/失败。

这种模式在引擎里成对出现得极其频繁。再看一个:

// 把一个世界坐标点找最近的样条点(UE 风格)
void FindClosest(const FVector& worldPos, FVector& outClosest)
{
    outClosest = project(worldPos);   // 读 worldPos,写 outClosest
}

命名约定帮你二次确认:带 out 前缀、又没有 const 的引用参数,基本就是函数用来回填结果的口子。

读代码易错点

这几个是阅读引用时最常栽的坑,逐条记住:

  • 把 & 看漏了。 void f(Mesh m) 和 void f(Mesh& m) 行为完全不同:前者改不到你的原件,后者能改。读签名时务必把 & 当成关键信息,别一眼扫过去。
  • 把声明里的 & 和取地址 & 搞混。 写在类型后面的 &(如 float& x)是"引用类型";写在变量前面的 &(如 &mass)是"取它的地址"——那是第 7 课指针的东西,这里先分清位置。
  • 以为 const T& 会复制。 不会。它和 T& 一样是零拷贝的别名,const 只是加了"不准改"的锁,跟"复不复制"无关。
  • 返回引用的陷阱(先眼熟)。 函数也可以返回引用,写成 T& getElement(...),意思是"把内部某个真实数据的别名给你,你能直接改它"。但有个致命坑:绝不能返回函数内局部变量的引用——那个变量在函数结束时就销毁了,返回的别名会指向一块已经失效的内存(悬空引用)。读代码时看到返回 &,要留意它引用的东西"活得够不够久"。

性能账:引用 vs 拷贝

为什么这件事在 VFX 里格外要紧?因为你的数据常常很大。一个 float 复制起来不值一提,但一个上百万元素的数组、一份网格、一条长字符串,按值传一次就要实打实搬一遍内存。

  • 小而便宜的类型(int、float、bool、FVector):按值传就好,复制成本几乎为零,反而更直观。
  • 大或复制昂贵的类型(TArray、FString、网格、矩阵):用 const T& 传,只读时零拷贝;需要函数写回时用 T&。

所以当你在一段每帧都跑的热代码里看到 const FMeshData& 而不是 FMeshData,那不是随手写的——是作者在有意避免复制大对象。读懂这个意图,你就开始像写代码的人一样思考了。
