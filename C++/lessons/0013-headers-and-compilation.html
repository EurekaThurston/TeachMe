前面你都在读单个函数、单个类。这一课往后退一步,看清整个项目:几十个 .h 和 .cpp 是怎么变成一个能跑的程序的。

为什么代码要分成两种文件

你打开一个真实的 C++ 工程(比如 Houdini 的 HDK、某个渲染器的源码),会看到成对出现的文件:Vector3.h 和 Vector3.cpp、Camera.h 和 Camera.cpp……为什么不写在一个文件里?

分工很简单,记住这一句:.h 放声明(declaration),.cpp 放定义(definition)。

  • 声明 declaration:这个函数 / 类长什么样——名字、参数、返回类型。就是你第 1 课读的那个函数签名,只是末尾加个分号、没有 { } 实现。
  • 定义 definition:具体怎么实现——花括号里那一堆真正干活的代码。
VFX 类比 .h 像一个节点 / HDA 的接口:参数面板(对外长什么样,有哪些输入输出端口);.cpp 像它内部的实现网络。别人调用你的节点时,只需要看参数面板(.h),根本不用关心里面那张网怎么连(.cpp)。

一个贯穿全课的配对例子:Vector3

先看头文件。它只承诺有这些东西,但一行实现都没有:

// ===== Vector3.h —— 声明(对外的接口) =====
#pragma once

struct Vector3 {
    float x, y, z;

    float length() const;          // 声明:有这个函数,但没写实现
    Vector3 normalized() const;   // 声明:同上
};

float dot(const Vector3& a, const Vector3& b);  // 声明,注意结尾的分号

注意:每行末尾是分号 ;,而不是 { }。这就是"声明":只说签名,不说怎么做。现在看配对的源文件,它把实现补上: 

// ===== Vector3.cpp —— 定义(具体实现) =====
#include "Vector3.h"   // 先把上面那份声明"贴"进来
#include <cmath>        // 标准库的 sqrt 在这里

float Vector3::length() const {
    return std::sqrt(x*x + y*y + z*z);   // 真正干活的代码
}

Vector3 Vector3::normalized() const {
    float len = length();
    return { x/len, y/len, z/len };
}

float dot(const Vector3& a, const Vector3& b) {
    return a.x*b.x + a.y*b.y + a.z*b.z;
}

那个 Vector3:: 前缀(下一课 14 讲 ::)意思是"这是属于 Vector3 的 length"。.cpp 里写实现时,必须告诉编译器它对应 .h 里的哪个声明。声明和定义必须一字不差地匹配(类型、参数都得对上),否则编译器认为你说的是两个不同的东西。

#include 到底在做什么

#include "Vector3.h" 不是什么神秘的"导入"。它是一条预处理器(preprocessor)指令,字面意思就是:在真正编译之前,把 Vector3.h 这个文件的全部内容,原封不动地复制粘贴到这一行的位置。

预处理器是编译之前的一道"文本替换"工序。以 # 开头的几句都归它管,先认识三个最常见的:

写法它在编译前做的事
#include "x.h"把 x.h 的内容整段贴到此处
#define PI 3.14把后文出现的 PI 文本替换成 3.14
#pragma once保证这个头文件只被贴进来一次(下文讲)
VFX 类比 #include "Vector3.h" 像在你的场景里引用进一个现成的 HDA 定义:你没有重写它的内部网络,只是把它的"接口契约"拉进来,这样你的代码就知道 Vector3、dot() 这些名字代表什么、能怎么用。

为什么需要 #pragma once / include guard

既然 #include 是"复制粘贴",就有个麻烦:如果 A.h 包含了 Vector3.h,你的 main.cpp 又同时包含了 A.h 和 Vector3.h,那 Vector3 的定义会被贴两遍。同一个 struct Vector3 出现两次,编译器会报"重复定义"。

解决办法就是头文件第一行那句 #pragma once:它告诉预处理器"这个文件只贴一次,重复的 #include 直接忽略"。你也会看到老式写法,叫 include guard(包含保护),效果一样: 

// 老式 include guard —— 和 #pragma once 等价
#ifndef VECTOR3_H      // 如果还没定义过 VECTOR3_H 这个标记……
#define VECTOR3_H      // ……就定义它(立个旗子)

struct Vector3 { float x, y, z; };

#endif                 // 第二次贴进来时旗子已立,整段被跳过

看到任意一个头文件顶上有 #pragma once 或这种 #ifndef / #define / #endif 包夹,你就知道:这是在防止重复包含,不是什么业务逻辑,扫一眼即可。

你会在真实代码里看到

真实工程里 .cpp 顶部往往有一长串 #include。这是一段 HDK(Houdini Development Kit)风格的头部,扫一眼能认出层次就够了:

// ===== SOP_MyDeform.cpp 的开头 =====
#include "SOP_MyDeform.h"   // 1) 配对的自家头文件,通常放最前

#include <SOP/SOP_Node.h>     // 2) HDK 提供的节点基类
#include <GEO/GEO_Point.h>    //    几何点的接口
#include <UT/UT_Vector3.h>    //    Houdini 自己的向量类型

#include <vector>             // 3) C++ 标准库
#include <cmath>

两种括号的区别:"双引号" 一般是你自己项目里的头文件;<尖括号> 一般是库或系统提供的头文件(标准库、HDK 等)。读代码时,这串 #include 就是这个文件的"依赖清单"——它用到了哪些外部东西,一目了然。

关键:编译 → 链接,是两步

从"一堆 .cpp"到"一个能跑的程序",中间有两个独立阶段。搞清这两步,你才能看懂构建报错出在哪。

① 编译(compile):每个 .cpp 单独处理。 编译器一次只看一个 .cpp(连同它 #include 进来的所有声明),把它翻译成一个目标文件(object file,.o 或 .obj)。一个 .cpp 加上它包含的头文件,合起来叫一个编译单元(translation unit)。这一步里,编译器只要看到声明就满意了——它知道 dot() 长什么样,就允许你调用,哪怕实现还在别的文件里。

② 链接(link):把所有目标文件拼到一起。 编译完每个 .cpp 各得一个 .o。链接器(linker)负责把它们缝合成最终的可执行文件 / 库:谁调用了 dot(),就去所有 .o 里找到 dot() 的定义,把这根线接上。

VFX 类比 编译像分别检查每一个节点内部接得通不通(单个节点自己的问题);链接像最后把所有节点连成一张能跑的大网——某个输入端口期待有人喂数据,链接器就负责找到那根输出线接上。少了哪根线,网就跑不起来。

读代码易错点:链接错误 ≠ 编译错误

这是本课最值钱的一节。两类报错出现在不同阶段,意味着完全不同的毛病:

编译错误(compiler error):语法 / 类型问题,出在单个文件内。 比如少了分号、类型对不上、用了没声明过的名字。报错会明确指到某个文件的某一行。

// 典型编译错误长这样:
Vector3.cpp:7:12: error: 'sqrt' was not declared in this scope
// → 在第 7 行,某个名字没声明(可能忘了 #include <cmath>)

链接错误(linker error):声明了、却找不到定义。 编译每个文件都过了(因为声明都在),但链接器去拼的时候,发现某个函数谁都没实现。报错信息里通常没有具体行号,而是一串符号名:

// 典型链接错误长这样(措辞按编译器不同):
undefined reference to `dot(Vector3 const&, Vector3 const&)'   // GCC/Clang
unresolved external symbol "float dot(...)"                    // MSVC
// → 声明有(.h 里写了),定义没有:可能 Vector3.cpp 没被加进构建

看到 undefined reference / unresolved external 这两个关键词,立刻反应:这是链接错误,不是语法错误。问题不在你正盯着的那行代码,而在"某个声明没有对应的定义被编译进来"——通常是某个 .cpp 漏加进了构建,或者库没链上。盯着 .h 改语法是没用的。

三个最常踩的坑

  • 把链接错误当语法错误。 看到 unresolved external 就去检查分号、类型——白费力气。它说的是"定义没找到",该去查构建里有没有漏掉那个 .cpp 或库。
  • 改了 .cpp 没更新 .h(反之亦然)。 比如给 dot 加了第三个参数,只改了 .cpp 的定义,忘了改 .h 的声明。两边对不上,调用方按旧签名调用 → 链接时找不到匹配的新定义。改签名要两边一起改。
  • 循环包含(circular include)。 A.h 包含 B.h,B.h 又包含 A.h,两个头文件互相等对方先定义,转不出来。常见解法是前向声明(forward declaration)。

前向声明,一句话先眼熟

如果某个文件只是用到某个类型的名字(比如声明一个 Camera* 指针、或一个返回 Camera 的函数),并不需要知道它内部长什么样,就可以只写一行"前向声明"告诉编译器"有这么个类型,细节回头再说": 

struct Camera;   // 前向声明:只承诺"有 Camera 这个类型",不含成员

Vector3 worldToScreen(const Camera& cam, const Vector3& p);  // 用得到名字就够了

这样就不必 #include "Camera.h",既能打破循环包含,也能减少不必要的依赖。第 14 课会再细讲名字与作用域。
