改并编译 —— 在 Unreal 里让改动跑起来

最后一课。前面你学会了"读";现在合上这门课的闭环——改一小处、编译、在编辑器里亲眼看到它生效。

热身 · 回忆第 17 课(别翻笔记)

先凭记忆答一题,把上一课读真实 Unreal C++ 源码的东西捞回来——这一课就建立在它之上。

这一课的形状:一个闭环,不是一堆语法

前 17 课都在练"读"。这一课不教新语法,而是教一个动作流程:把你已经读懂的代码动一下,然后让 Unreal 把改动编译进去、在视口里跑起来。整门课的目标——"读懂并能改真实 C++"——就靠这一步合上。

闭环只有四步,记住它:

步骤你做什么结果
① 读懂找到要改的那一行,确认它干嘛知道改哪、改成啥
② 改一小处改一个默认值,或加一个 if源码变了,但还没生效
③ 编译Live Coding 或重新 buildC++ 被翻译成机器码并装回引擎
④ 验证在编辑器/视口里看效果确认改动真的生效了

为什么"改"和"生效"是两步?因为 C++ 是提前编译的语言(这正是第 13 课讲的:源码 .h/.cpp 要先经过编译 + 链接才变成引擎能运行的机器码)。你在文本编辑器里改字符,引擎里跑的还是上一次编译出来的旧机器码——直到你重新编译。这跟你在 Blueprint 里连根线就立刻能玩、或在材质里调个数就实时刷新不一样:C++ 的改动必须先过编译这道关。

我们要改的真实代码:一个 Actor

下面是一个最常见、最有代表性的 Unreal C++ Actor:它每帧让自己绕 Z 轴转一点。这种"会自己动的物件"在 VFX/场景搭建里到处都是。它和第 17 课一样分成头文件 .h(声明)和源文件 .cpp(实现)两份。[1]

头文件 SpinningActor.h(精简)

// SpinningActor.h —— 声明这个 Actor 类
#pragma once

#include "CoreMinimal.h"
#include "GameFramework/Actor.h"
#include "SpinningActor.generated.h"   // 自动生成,必须放最后

UCLASS()
class MYGAME_API ASpinningActor : public AActor
{
    GENERATED_BODY()

public:
    ASpinningActor();

    // 每秒转多少度,可在编辑器面板里调
    UPROPERTY(EditAnywhere, Category = "Spin")
    float RotationSpeed = 90.0f;

protected:
    virtual void BeginPlay() override;

public:
    virtual void Tick(float DeltaTime) override;
};

源文件 SpinningActor.cpp(精简)

// SpinningActor.cpp —— 实现每个方法
#include "SpinningActor.h"

// 构造函数:Actor 被创建时跑一次
ASpinningActor::ASpinningActor()
{
    // 打开逐帧 Tick(默认是关的)
    PrimaryActorTick.bCanEverTick = true;
}

void ASpinningActor::BeginPlay()
{
    Super::BeginPlay();   // 先让父类做它那一半
}

// 每帧调用一次,DeltaTime 是上一帧到这一帧的秒数
void ASpinningActor::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    FRotator delta(0.0f, RotationSpeed * DeltaTime, 0.0f);
    AddActorLocalRotation(delta);
}

你应该已经认得这里几乎所有东西了(第 17 课的"放大"逐块讲过):class ... : public AActor 是继承(第 12 课);ASpinningActor() 是构造函数(第 12 课);Tick(float DeltaTime) 是第 1 课的函数签名;RotationSpeed * DeltaTime 是第 4 课的表达式。新面孔只有几个 Unreal 专属的宏:UCLASS()、UPROPERTY(...)、GENERATED_BODY()——把它们当成"贴在类/变量上的标签",作用是让 Unreal 认得这个类、并把 RotationSpeed 暴露到编辑器面板里。EditAnywhere 就是"这个值允许在编辑器里改"。

改动 A · 改一个默认 UPROPERTY 值(最安全)

最安全的第一个改动:让它转快一点。在头文件里把默认转速从 90 改成 180。这是一处单字符级的改动,不会动到任何结构。 

// 改之前
float RotationSpeed = 90.0f;

// 改之后 ——— 只改了这一个数字
float RotationSpeed = 180.0f;

就这一行。注意:因为它写在 UPROPERTY(EditAnywhere) 上,已经摆进场景的那个 Actor 可能仍保留它当初的值(编辑器把改过的实例值存下来了);这个新默认值对之后新拖进场景的实例生效。这是 Unreal 的正常行为,先眼熟,后面"验证"一节会再说。

改动 B · 在 Tick 里加一个 if 守卫(读代码者最该会的)

第二个改动更能体现"读懂才能改":给 Tick 加一个守卫(guard),转速接近 0 时干脆别算旋转,省掉无意义的每帧操作。"守卫"就是函数开头一个 if,条件不满足就提前 return 退出——这正是第 2 课讲的"return 立刻离场"。 

void ASpinningActor::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    if (RotationSpeed == 0.0f)   // ← 新增:转速为 0 就不用算了
        return;                  //    提前退出,后面两行不执行

    FRotator delta(0.0f, RotationSpeed * DeltaTime, 0.0f);
    AddActorLocalRotation(delta);
}

你能读懂它因为你学过:== 是"是否相等"的两个等号(第 2 课,别写成一个 =);if (...) return; 是条件成立就立刻结束函数(第 2 课);它加在 Super::Tick(DeltaTime) 之后,所以父类的 Tick 仍然照常跑。这就是"读代码"直接变现成"安全地改代码":你知道这一行加在哪不会破坏别的逻辑。

编译让它生效:Live Coding vs 完整重新编译

改完源码,引擎里跑的还是旧机器码。把改动"装进去"有两条路,你必须分清什么时候用哪条。

路线一 · Live Coding(编辑器开着就能编,最快)

Unreal 5 自带 Live Coding,默认开启(在 Editor Preferences > General > Live Coding 里能看到开关)。[1] 用法只有一步:

  • 保存你改过的 .h/.cpp(不存的话改动不会被编译)。
  • 焦点在 Unreal 编辑器(或你的 IDE)上时,按 Ctrl + Alt + F11 触发一次 Live Coding 编译。[1]
  • 右下角弹出一个编译状态通知;成功的话,编辑器不用重启,改动立刻在当前会话里生效。你甚至可以在编译时继续操作编辑器。[1]

上面的改动 A、B 都属于 Live Coding 能搞定的小改动:改一个 .h 里的默认值、在已有函数体里加几行逻辑。官方文档明确说:在 .h 头文件里改默认值,Live Coding 能看到变化生效。[1]

路线二 · 完整重新编译(改了结构就必须走这条)

有些改动 Live Coding 处理不了,硬按 Ctrl+Alt+F11 可能行为异常甚至让编辑器崩溃。[1] 这时候要做完整重新编译:

  • 关掉 Unreal 编辑器。
  • 在 Visual Studio / Rider 里,把配置设成 Development Editor,右键你的项目选 Build(或 Rebuild 全量重编),让 UnrealBuildTool 重新编译并链接整个模块。[2]
  • (可选)也可以直接在 Unreal 编辑器工具栏点 Compile 按钮做一次常规编译。[2]
  • 编译完再打开编辑器,改动就装好了。

另外:如果你新增了文件或新的类,改的不只是代码内容、而是项目结构,那还要先重新生成项目文件——右键 .uproject 选 Generate Visual Studio project files(或在 IDE 的 Tools 菜单里 Refresh Visual Studio Project),让 IDE 认得新文件,然后再 build。[2]

读代码易错点 · 报错与那个 UE 大坑

读代码易错点
  • 改了 UCLASS/UPROPERTY 的结构,常常要完整重编,不是 Live Coding。 改一个默认值没问题;但新增/删除一个 UPROPERTY、改它的类型、加一个新的 UFUNCTION、加一整个新类,属于"结构性改动"。它会让 Unreal 的代码生成器(就是那个 .generated.h)重新生成新代码,Live Coding 这种增量补丁应付不来——官方说大改动(新函数、新变量集、大重构)在 Live Coding 下"行为不可预测",通常直接崩。[1] 正确做法:关编辑器 → 在 IDE 里完整 Build。这就接回第 13 课的编译/链接模型:头文件里的声明变了,所有 #include 它的编译单元都得重新编译、重新链接,半路打补丁是不够的。
  • 报错先看第一条,不是最后一条。 C++ 一个错误常引发一长串后续报错。从最上面那条开始读,它通常是真正的病根;修掉它,后面一片往往跟着消失。
  • 看懂报错三件套:文件、行号、信息。 比如 SpinningActor.cpp(24): error C2065: 'RotatonSpeed': undeclared identifier —— 文件 SpinningActor.cpp、第 24 行、"用了一个没声明过的名字"。八成是把 RotationSpeed 拼成了 RotatonSpeed。双击报错能直接跳到那一行。
  • ; expected / "missing ';'" 往往错在上一行。 编译器是读到下一行才发现少了分号,所以它报的行号常常比真正出错的地方晚一行。看到这种错,先检查它上面那一行结尾。
  • == 写成 = 不一定报错,但行为全变。 if (RotationSpeed = 0.0f) 是赋值再判断(第 2 课),会把转速悄悄改成 0、且条件恒为假;编译器可能只给个 warning。读/改 if 时务必确认是两个等号。
  • "链接错误"(LNK)和"编译错误"(C2xxx)不是一回事。 第 13 课:编译错误是某个 .cpp 自己语法不对;链接错误(LNK2019: unresolved external symbol)是各文件单独都编过了、但拼起来时找不到某个函数的实现——常见于"在 .h 声明了函数却忘了在 .cpp 写它的实现"。

在编辑器里验证(闭环的最后一步)

编译成功不等于改对了。一定要亲眼验证:

  • 改动 A(默认转速):往场景里新拖一个 ASpinningActor,点 Play,它应该转得更快了。注意已经在场景里的旧实例可能仍保留旧值——这正是上面说的 EditAnywhere 实例覆盖行为,不是 bug。也可以直接在它的 Details 面板里看 RotationSpeed 这一栏现在的默认是不是 180。
  • 改动 B(if 守卫):在 Details 面板把某个实例的 RotationSpeed 设成 0,Play,它应该纹丝不动(走了你新加的 return);设成非 0,它照常转。两种情况都对上了,才算这一处改动真的生效。

看到效果的那一刻,你就完整跑通了"读懂 → 改 → 编译 → 验证"。这正是这门课从第 1 课到现在一直瞄准的终点。
