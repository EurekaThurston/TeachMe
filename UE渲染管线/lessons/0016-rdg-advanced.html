Part 3 · 渲染线程 · Lesson 0016

RDG 进阶：资源生命周期、屏障自动化、异步计算、Pass 剔除

0015 讲了 RDG 的三步模型和基本 API。这节课进入编译期：RDG 编译器如何自动管理资源生命周期与内存别名、如何推导屏障、如何把标记了 AsyncCompute 的 Pass 调度到独立队列、如何剔除无用 Pass。这几项能力叠加在一起，解释了为什么"迁移到 RDG"在 4.26 之后是渲染性能优化的标准动作。

时长 ~25 分钟 前置 0015（RDG 入门）、0002（异步计算）、0003（资源屏障） 产出 能解释 RDG 如何自动管理资源生命周期/屏障、做内存别名、调度异步计算、剔除无用 Pass

把 0015 的三步模型记在脑子里：Recording → Compile → Execute。这节课聚焦 Compile 阶段——RDG 编译器在 Execute() 内部做了哪四件高价值的事，以及你作为 Pass 编写者如何利用它们。

1资源生命周期与内存别名（Resource Aliasing）

编译阶段第一件事：对图中每一个 FRDGTexture/FRDGBuffer，确定它的存活区间——从第一个写它的 Pass，到最后一个读它的 Pass。

Pass A写 Tex1
›
Pass B读 Tex1
写 Tex2
›
Pass C读 Tex2
写 Tex3
›
Pass D读 Tex3

Tex1 的生命周期：[PassA, PassB]；Tex2 的生命周期：[PassB, PassC]。两者不重叠——RDG 让它们共用同一块物理显存（内存别名）。

在上面这个例子里，Tex1 在 Pass B 读完后就"死了"，Tex2 的分配可以复用那块显存——这叫内存别名（Resource Aliasing），在 glossary ⑤ 有定义。对于帧内大量临时纹理（SSAO 中间图、模糊 Buffer、TAA 历史等），别名优化能把峰值显存开销压低 20–40%。

瞬态资源池（Transient Allocator）

RDG 内部对瞬态资源（即"只在本帧内存活、不跨帧"的资源）使用一个专属分配器。相比通用 GRenderTargetPool，瞬态分配器能做更激进的别名，因为 RDG 知道整帧所有资源的全局依赖图。GRenderTargetPool 是旧有的运行时池，4.26 中与 RDG 并存；新 Pass 优先用 RDG 瞬态资源。

2屏障自动化

回顾 0003：在 D3D12/Vulkan 下，每次一个资源从"渲染目标"切换到"着色器只读"都要显式插 ResourceBarrier（状态转换），否则 GPU 缓存不一致、读到旧数据。手写这些屏障是错误高发区。

RDG 的屏障自动化逻辑：

  1. 每个 Pass 的 BEGIN_SHADER_PARAMETER_STRUCT 告诉 RDG：这个资源作为 RTV、UAV 还是 SRV 使用（对应 D3D12 的不同资源状态）。
  2. 编译时，RDG 对相邻 Pass 之间的每对（资源，状态变化）生成一次状态转换命令。
  3. RDG 还会合并（batch）多个 Pass 边界处的屏障，让多个资源转换尽量打包进一次 API 调用——D3D12 的 ResourceBarrier 支持数组参数，批量提交显著减少命令开销。
// 典型的读写状态变化：Pass B 把 Tex1 从 RTV 状态切到 SRV 状态
// 你不写任何 Barrier 代码——RDG 根据 Param 声明自动插入

BEGIN_SHADER_PARAMETER_STRUCT(FPassBParams, )
    // 这里标注 SRV，RDG 知道 Pass B 读它
    SHADER_PARAMETER_RDG_TEXTURE(Texture2D, Tex1)
    // 这里标注 RTV，RDG 知道 Pass B 写它
    RENDER_TARGET_BINDING_SLOTS()
END_SHADER_PARAMETER_STRUCT()
Pass 里绕过 RDG 直接用 RHI 操作资源——屏障将丢失

如果你在 AddPass 的 lambda 里，用 RHICmdList.TransitionResource 手动转换一个 RDG 管理的资源状态，会破坏 RDG 的状态追踪，导致之后的自动屏障失效。RDG 管理的资源必须完全走 RDG 声明路径——不要绕过它手动插屏障。若确实需要特殊屏障，用 GraphBuilder.UseExternalAccessMode 或 Pass Flags 通知 RDG。

3异步计算调度

给 AddPass 的 ERDGPassFlags 加上 AsyncCompute：

GraphBuilder.AddPass(
    RDG_EVENT_NAME("SSAO_AsyncCompute"),
    Params,
    ERDGPassFlags::AsyncCompute,   // ← 这一行
    [Params](FRHICommandList& RHICmdList) { /* ... */ });

RDG 编译时识别到这个标记，把该 Pass 的命令路由到独立的计算队列（Async Compute Queue）。计算队列与图形队列在物理上是 GPU 内部不同的引擎，可以真正并行——在 GCN/RDNA/Maxwell 以上的 GPU 上，图形光栅化的空隙可以被计算工作填满（呼应 0002 讲的"白嫖性能"）。

RDG 还自动插入计算队列与图形队列之间的同步围栏：如果后续图形 Pass 依赖 AsyncCompute Pass 的输出，RDG 在图形队列上插一个等待，确保计算结果可见后才继续。

典型 AsyncCompute 候选 Pass

SSAO（纯计算、不依赖光栅结果）、GTAO、光照剔除的 Light Grid 计算、部分 RayTracing Dispatch，以及延迟光照里的 Clustered 计算——这些 Pass 与同一帧的 GBuffer 写入没有数据依赖，可以并发。4.26 源码里搜 ERDGPassFlags::AsyncCompute 即可找到引擎已标记的 Pass。

4Pass 剔除（Culling）

如果一个 Pass 的所有输出资源在之后的任何 Pass 里都没有被读取，RDG 编译时会把它从执行队列里删除——它的 lambda 不运行，资源不分配。

典型触发场景：

  • 用 r.SSR 0 关闭 SSR 时，依赖 SSR 输出的 Pass 链整条被剔除。
  • 编辑器里切换 Debug View Mode，大量正常渲染 Pass 的输出不再被后处理读取。
  • 某些 Pass 只在特定 CVars 或 Feature 开启时有意义，关闭时 RDG 自动不执行。

强制保留一个 Pass（即使其输出没被引用）可以用 ERDGPassFlags::NeverCull，或把其输出资源标为 Extracted（GraphBuilder.QueueTextureExtraction），告诉 RDG "我要在 Execute 外面用这个结果"。

检索练习

RDG 能对生命周期不重叠的两个瞬态纹理做内存别名，根本原因是什么？

解释 ERDGPassFlags::AsyncCompute 对 GPU 调度的影响，包括：路由到哪个队列、与图形队列如何同步、哪类 Pass 适合标记。

标记 AsyncCompute 的 Pass 在 Execute 时，RDG 把它的命令录制到独立的计算队列而不是图形队列。计算队列与图形队列在 GPU 内部并行（独立引擎），可同时运行，填满光栅化空隙。

同步：若后续图形 Pass 依赖该 AsyncCompute Pass 的输出，RDG 在图形队列插入等待计算队列围栏的命令，确保结果可见后才继续。围栏由 RDG 自动插入，程序员不需手动写。

适合标记的 Pass：纯计算、与同帧 GBuffer 写入没有数据依赖的工作——如 SSAO、GTAO、Light Grid 计算、部分 RayTracing Dispatch。不适合：依赖当前帧光栅阶段写入结果、或输出要立刻给光栅 Pass 读的 Pass。

带走这几条

  • 内存别名：生命周期不重叠的瞬态资源共用同一块物理显存——前提是延迟分配（FRDGTextureRef 是虚拟句柄）。
  • 屏障自动化：Pass 参数结构体声明资源访问类型（RTV/UAV/SRV）→ RDG 编译时推导状态转换 → 批量打包生成 D3D12 ResourceBarrier；绕过 RDG 手动操作 RDG 资源会破坏追踪。
  • 异步计算：ERDGPassFlags::AsyncCompute 把 Pass 路由到计算队列并行执行；RDG 自动插队列间围栏保证同步；适合纯计算、无光栅依赖的 Pass。
  • Pass 剔除：输出无引用的 Pass 自动不执行；用 NeverCull 或 QueueTextureExtraction 强制保留。
  • 调试：r.RDG.Debug（屏障日志）、r.RDG.CullPasses 0（禁剔除）、r.RDG.AsyncCompute 0（禁异步）、RDG_GPU_STAT_SCOPE（GPU 时间线标注）。
