多线程渲染：帧流水线与"渲染线程落后一帧"

UE 4.26 不是单线程顺序画帧的——三条 CPU 侧线程加 GPU，同时各跑不同的帧。这节课拆透这个错位：为什么这样设计、由什么机制保证不出乱、以及 stat unit 的四个数字分别在量谁。

时长 ~20 分钟 前置 0004（CPU/GPU 协作、流水线与延迟）、0005（一帧的生命周期总览） 产出 能解释 GameThread / RenderThread / RHIThread / GPU 的流水线关系、各落后一帧的原因，并正确解读 stat unit 的 Game / Draw / GPU / RHIT 四个数字

0004 证明了"流水线是提高吞吐的正确姿势"，0005 画出了四角色接力的全景图。这节课回答那张图里被一笔带过的问题：游戏线程在帧 N、渲染线程在帧 N-1、GPU 在帧 N-2 ——这件事到底是怎么实现的？谁在"锁"谁、锁在哪里？ 弄清这个机制，stat unit 的每一行才有意义，帧率瓶颈才能被准确定位。

1三线程 + GPU：四条并行轨道

UE 4.26 在 PC 默认开启三条 CPU 线程各司其职，加上 GPU 作为独立执行单元，共四条轨道同时推进。它们不是顺序排队，而是流水线错位并行的——某一帧在游戏线程上是"帧 N"，在渲染线程上同时处理的是"帧 N-1"，在 RHI 线程是"帧 N-2"，GPU 则在画"帧 N-3"（实际错位因配置略有不同，最常见是 1–2 帧差）。

游戏线程
GameThread处理 帧 N：Tick Actor/Physics/Animation，更新 UWorld 状态，通过 ENQUEUE_RENDER_COMMAND 把变化投递给渲染线程。由 FTaskGraphInterface 驱动。
渲染线程
RenderThread处理 帧 N-1：消费游戏线程上一帧的入队，跑可见性剔除、组织 Pass、用 RDG 录制 RHI 命令列表。写 FRHICommandList，不直接调图形 API。
RHI 线程
RHIThread (GRHIThread)处理 帧 N-2：FRHICommandListExecutor 把渲染线程录的平台无关命令翻译成 D3D12 调用并提交 GPU 队列。
GPU处理 帧 N-3（约）：执行命令列表——光栅化、Pass、后处理——最后 Present 上屏。

2帧流水线：错位的同步栅栏

四轨道并行，若无同步则游戏线程会无限领先渲染线程，内存里堆积无穷多帧的待处理数据。UE 用两道栅栏把领先幅度锁死：

栅栏类型控制什么
FFrameEndSync帧同步栅栏游戏线程帧末，检查渲染线程是否已消化到"当前帧 − 最大落后帧数"；若未消化则游戏线程阻塞等待。
FRenderCommandFence渲染命令围栏游戏线程往渲染线程投递命令后，可选择插入围栏等待某批命令执行完毕（BeginFence / Wait），用于需要同步的单次操作。

决定"游戏线程最多领先几帧"的是控制台变量 r.OneFrameThreadLag（默认值 1，即开启）。开启时游戏线程最多领先渲染线程 1 帧；若强制关闭（设为 0），游戏线程每帧都要等渲染线程追上，帧率受两线程中较慢者瓶颈，等于退化成单流水线，吞吐显著下降。

为什么"落后一帧"是吞吐最优解

0004 课推导过：CPU 和 GPU 同时跑不同帧，两者的忙等时间被重叠掉，帧率由最慢的那条轨道决定，而不是各段串行相加。r.OneFrameThreadLag=1 就是这个原理的直接开关——用 1 帧的输入延迟代价，换两条 CPU 线程并行满跑的吞吐。对于 60 Hz 游戏，这 1 帧约 16 ms 的延迟通常远低于玩家感知阈值；电竞低延迟场景可通过其他机制（如 NVIDIA Reflex）补偿。

3任务图驱动：FTaskGraphInterface

三条线程之所以能有条不紊地并行，是因为它们都跑在 UE 的任务图（Task Graph）系统上，而不是各自持有一个原始 OS 线程并靠睡眠/唤醒协调。FTaskGraphInterface 是任务图的全局接口，负责把"任务节点"分发给底层线程池中的工作线程，并管理节点间的依赖（前置完成才触发后续）。

游戏线程和渲染线程是任务图里两个命名线程（Named Thread），它们各有专属队列；RHI 线程同样如此。ENQUEUE_RENDER_COMMAND 宏本质上是把一个 lambda 包装成任务节点投进渲染线程队列——任务图保证它在渲染线程上顺序执行，游戏线程无需知道底层细节。

关键洞见 · 渲染线程"阻塞"不是真 OS 阻塞

FFrameEndSync 让游戏线程等待时，实际上游戏线程在任务图里自旋或让出，而不是调 OS 的 Sleep——这样一旦渲染线程的任务完成，游戏线程能以极低延迟被唤醒，避免 OS 调度抖动。理解这一点能解释为什么 UE 的线程开销比"朴素多线程"低得多。

4RHI 线程：GRHIThread 与 Bypass 模式

渲染线程录制的 FRHICommandList 是平台无关的——它存的是"绑定这个 PSO""SetRenderTarget 到这张纹理""DrawIndexed 1200 个顶点"这类抽象指令。真正的 D3D12 / Vulkan / Metal 调用由 RHI 线程完成：FRHICommandListExecutor 在 RHI 线程上消费命令列表，调用具体平台的实现（如 D3D12RHI 模块），把命令写入 D3D12 ID3D12GraphicsCommandList，最后提交到 GPU 队列。

RHI 线程由全局布尔 GRHIThread（以及 GRHISupportsParallelRHIExecute）标记是否启用。若平台不支持或主动关闭（控制台变量 r.RHICmdBypass 1），则走 bypass 模式：渲染线程直接在自身线程上执行 RHI 命令，没有单独 RHI 线程，命令列表被跳过。Bypass 便于调试（单线程易断点），但代价是渲染线程和 RHI 录制串行，帧率下降。

Bypass 模式是调试工具，不是性能优化

r.RHICmdBypass 1 会让渲染线程承担所有 RHI 翻译工作，帧时间通常明显上升。只在排查"是否某个 RHI 命令顺序问题导致 Bug"时使用，正式发布绝不开启。

5帧错位全景：N / N-1 / N-2 / N-3

把上面几节合并成一张时间轴图，是理解 stat unit 的关键。下图横轴是时间（每格约一帧），纵轴是四条轨道：

GameThread 帧 N：Tick → 变化入队 → FFrameEndSync 检查 →（若渲染线程未追上则等待）
RenderThread 帧 N-1：收上帧入队 → InitViews → RDG → 录 FRHICommandList → 触发 RHI 线程消费
RHIThread 帧 N-2：FRHICommandListExecutor 翻译 → D3D12 命令列表 → ExecuteCommandList 提交 GPU 队列
GPU 帧 N-3（约）：执行 D3D12 命令列表 → 各 Pass → Present

6stat unit：四个数字，四条轨道

在 UE 编辑器或开发版 build 中输入 stat unit，可以看到四行计时：

stat unit 行量的是对应轨道瓶颈特征
Game游戏线程当前帧 Tick 耗时GameThreadGame 高 → Gameplay / Physics / Animation 太慢
Draw渲染线程当前帧录制耗时RenderThreadDraw 高 → 可见性剔除 / RDG 录制 / 绘制命令数太多
GPUGPU 执行当前帧所有命令的耗时GPUGPU 高 → 着色器 / 像素数 / 带宽瓶颈
RHITRHI 线程翻译耗时RHIThreadRHIT 高 → draw call 太多或平台 RHI 驱动开销大

帧率由四条线中最慢的那条决定——这是流水线的铁律（同 CPU 流水线的最长级理论）。若 GPU = 20 ms、其余 < 10 ms，则帧率上限约 50 fps，CPU 轨道再快也毫无意义，要优化就从 GPU 入手。反之若 Game = 25 ms 但 GPU = 8 ms，GPU 在等游戏线程，优化 Gameplay Tick 才有收益。

Draw 高 ≠ GPU 慢

Draw（渲染线程）和 GPU 是两条独立轨道。Draw 高意味着 CPU 侧的渲染组织太慢（比如数万个 FMeshDrawCommand 录制），GPU 未必是瓶颈。反过来 GPU 高时 Draw 可以很低。把这两个数混淆是初学者最常见的性能诊断错误。

检索练习

r.OneFrameThreadLag 1（默认开）对渲染管线的主要作用是什么？

不看笔记，说出 stat unit 的 Game / Draw / GPU / RHIT 四个数字各对应哪条轨道、以及如何用它们定位帧率瓶颈。
Game = 游戏线程（Tick 耗时）；Draw = 渲染线程（录制 RHI 命令列表耗时）；GPU = GPU 执行耗时；RHIT = RHI 线程翻译耗时。帧率由四者中最大值决定。定位瓶颈：看哪个数字最大，那条轨道是瓶颈；Game 高查 Gameplay/物理/动画，Draw 高查绘制命令数量/剔除效率，GPU 高查着色器复杂度/像素数/带宽，RHIT 高查 draw call 数量或驱动开销。特别注意：Draw 高和 GPU 高是两件不同的事，不要混淆 CPU 侧录制和 GPU 侧执行。

带走这几条

  • UE 4.26 有三条 CPU 侧线程：GameThread（Tick）→ RenderThread（录制）→ RHIThread（翻译提交）+ GPU，四者流水线并行、各处理不同帧号。
  • 帧同步由 FFrameEndSync 和 FRenderCommandFence 控制；r.OneFrameThreadLag=1（默认）允许游戏线程最多领先渲染线程一帧，换吞吐代价是输入延迟。
  • 三线程和 GPU 都跑在 FTaskGraphInterface 任务图上；命令投递用 ENQUEUE_RENDER_COMMAND。
  • RHI 线程（GRHIThread）靠 FRHICommandListExecutor 把平台无关命令翻成 D3D12 调用；r.RHICmdBypass 1 退化到渲染线程直接执行，仅用于调试。
  • stat unit 的 Game / Draw / GPU / RHIT 对应四条轨道，帧率由最慢一条决定；Draw 高和 GPU 高是两个独立问题。
