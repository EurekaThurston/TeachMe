GPU 是一台什么样的并行机器

上一课讲了"管线做什么"，这课讲它"为什么能那么快"。理解 GPU 的并行模型，你才能解释：为什么材质里加个分支会变慢、为什么 Draw Call 要合批、为什么 UE 要把光照剔除放进计算着色器、为什么异步计算能"白嫖"性能。

时长 ~15 分钟 前置 0001 硬件管线 产出 能用"延迟隐藏 + SIMT"解释 GPU 性能的来源与三类常见浪费

CPU 和 GPU 是为两个相反目标设计的。CPU 想让单个任务尽快完成（低延迟）；GPU 不在乎单个任务多慢，它要让海量任务的总吞吐最大。这一句对立，决定了 GPU 几乎所有的设计取舍——也决定了你该怎么喂它。

1两种哲学：延迟优化 vs 吞吐优化

CPU（延迟优化）GPU（吞吐优化）
核心数少而强（几~几十个）极多而简（数千 ALU）
面对内存延迟砸大量晶体管做缓存、乱序执行、分支预测去避免/隐藏等待几乎不做这些，靠切换到别的并行任务来掩盖等待
擅长串行、分支密集、低延迟的控制逻辑同一段代码在海量数据上并行跑（着色、计算）
核心机制 · 用并行隐藏延迟

访问显存要几百个时钟周期。CPU 怕这个延迟，所以建多级缓存去躲。GPU 不躲——当一组线程因为等内存而卡住时，硬件立刻切换到另一组就绪的线程继续算，等内存回来再切回去。只要在飞的线程足够多，所有等待都被别人的计算填满，执行单元永不空闲。这就是 GPU 高吞吐的根本秘密：不是更快，而是永不闲着。

2SIMT：32 个线程绑在一起走

GPU 不是一个线程一个线程地调度。它把着色器实例（一个顶点、一个片元、一个计算线程）32 个打成一组锁步执行——NVIDIA 叫 warp，AMD 叫 wavefront（64 个）。1 一组里所有线程在同一时刻执行同一条指令，只是各自的数据不同（Single Instruction, Multiple Threads）。

所以"GPU 有几千个核"要打个折看：真正独立的调度单位是 warp，不是单个线程。32 个线程共享一个程序计数器。

这带来一个你必须刻进脑子的后果——分支分歧（divergence）：

if (worldNormal.y > 0.5) {
    color = ExpensiveSnow();   // warp 里一半线程走这
} else {
    color = ExpensiveRock();   // 另一半走这
}

材质里一个看似无害的分支

因为 32 个线程必须步调一致，硬件没法让一半跑雪、一半跑岩石。它的做法是：两条分支都执行，用掩码让"雪线程"在跑岩石代码时丢弃结果，反之亦然。结果——这个像素的成本约等于 雪 + 岩石，分支根本没省。这就是为什么材质里的动态分支常常不省反亏，以及为什么 UE 用 static switch / shader permutation 在编译期把分支消掉，而不是留到运行时。

关键洞见 · "免费"的并行有边界

同一 warp 内分歧才有惩罚。如果整个 warp 的线程都走同一条分支（比如一大片连续像素都是天空），就没有惩罚——这叫相干（coherent）执行。屏幕上相邻像素往往落在同一 warp，所以"大块同质区域便宜、细碎杂乱区域贵"。这条直觉能解释很多性能现象。

3占用率：决定延迟能不能被藏住

第 1 节说"靠切换到别的线程隐藏延迟"——能切换的前提是真的有足够多线程同时驻留。一个 SM/CU（流多处理器）上同时活着的 warp 数与硬件上限之比，就叫占用率（occupancy）。

占用率不是越满越好，但太低就藏不住延迟。限制它的是资源压力：

  • 寄存器：每个线程用的寄存器越多，能同时驻留的线程就越少。一个写得臃肿、变量爆多的复杂材质着色器，会因为寄存器压力把占用率压低，反而变慢。
  • 共享内存 / LDS：计算着色器里 thread group 共享的片上内存，用得越多，能并发的 group 越少。

这就是为什么 UE 的 r.Shaders.Optimize、shader 复杂度视图（材质的 Shader Complexity 着色）很重要——它们间接反映寄存器/指令开销，进而影响占用率与最终耗时。

显存 VRAM大带宽·高延迟

GPU 的显存带宽是 CPU 内存的好几倍，但单次访问延迟更高。要吃满带宽，访问必须合并（coalesced）：一个 warp 的 32 个线程如果读的是连续地址，硬件能合成一两次大事务；如果读得四处乱跳，就退化成几十次小事务，带宽利用率崩塌。这条直接解释了：纹理为什么要用 mipmap 和块压缩（提升缓存命中与合并）、为什么随机访问大缓冲很贵。

5计算着色器：跳出图形管线的并行

顶点/像素着色器被绑在图形管线的固定流程里。计算着色器（compute shader）则直接把 GPU 当成通用并行处理器：你自己定义线程组织（thread → group → grid），自由读写缓冲，不经过光栅化。UE 用它做大量"非画三角形"的活：

  • 光照剔除：把光源分配进屏幕 3D 网格（Light Grid，见 0024）——天然是网格状并行任务。
  • 后处理：Bloom 下采样、TAA、直方图统计曝光。
  • GPU 剔除 / GPU Scene：在 GPU 上算可见性、做实例化（0020）。
异步计算 · 为什么能"白嫖"性能

图形队列在跑某个 Pass 时，GPU 的计算单元常常没吃满（比如光栅化阶段卡在固定功能硬件上）。异步计算把计算着色器丢到一条独立队列，让它和图形队列同时跑，正好填进那些空闲的执行单元——总时间几乎不增加，等于免费做完。UE 4.26 的 RDG 会自动调度异步计算并插好围栏（见 0016）。你现在就理解了 RDG 那个卖点的物理基础。

检索练习

材质里一个 if 让同一 warp 的线程走向不同分支，最可能的结果是什么？

GPU 几乎不靠大缓存，那它如何掩盖访问显存的几百周期延迟？这对"占用率"提出了什么要求？
GPU 靠超额并行掩盖延迟：当一个 warp 因等显存而停顿，调度器立刻切换到另一个就绪 warp 继续算，等数据回来再切回。只要同时驻留的 warp 足够多，所有停顿都被别人的计算填满，执行单元不空转。前提是占用率要够——而占用率受每线程寄存器数和共享内存用量限制。所以一个寄存器爆多的臃肿着色器会压低占用率、削弱延迟隐藏能力，即使指令数不变也可能更慢。

带走这几条

  • GPU 是吞吐优化的：不靠跑得快，靠"永不闲着"——用海量并行线程掩盖显存延迟。
  • 调度单位是 warp/wavefront（32/64 线程锁步）。同 warp 内分支分歧会让两条路径都执行——运行时分支常常不省。
  • 占用率决定延迟能否被藏住；它被寄存器/共享内存压力限制——臃肿着色器会变慢。
  • 显存带宽大但延迟高；访问要合并才吃得满带宽。
  • 计算着色器 + 异步计算能填满图形管线的空闲单元——这是 RDG 自动异步调度的物理基础。
