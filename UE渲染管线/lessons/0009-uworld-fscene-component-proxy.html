场景的两份表达：UWorld→FScene、Component→Proxy

UE 为何要把场景维护两份、两个对象对应同一个图元？这节课拆清六个核心类的职责与所属线程，搞透这张"游戏态 vs 渲染态"的对照表——Part 2 其余课全依赖它。

时长 ~22 分钟 前置 0005（两份场景概念）、0008（Tick 与帧末推送） 产出 能区分 UWorld/FScene、UPrimitiveComponent/FPrimitiveSceneProxy/FPrimitiveSceneInfo 的职责与所属线程

0005 用一句话概括了"两份场景"；0008 展示了游戏线程侧数据如何产生。这节课把中间那道墙拆细：游戏侧的 UWorld 和渲染侧的 FScene 如何分工，一个 UStaticMeshComponent 在游戏侧和渲染侧各对应什么对象，以及三个名字相近但职责迥异的"Primitive 类"如何协作。

1UWorld 与 FScene：场景的两条线

UWorld 是游戏线程的世界容器，持有所有 Level、Actor 列表以及物理场景（FPhysScene）。它有一个成员：

// UWorld（World.h）
FSceneInterface* Scene;   // 实际类型是 FScene（渲染线程侧）

FScene（Renderer/Private/Scene.cpp / ScenePrivate.h）是渲染线程的场景，持有所有可渲染对象的渲染侧数据结构，包括：

  • TArray<FPrimitiveSceneInfo*> Primitives——所有已注册图元的渲染侧记账对象
  • TArray<FLightSceneInfo*> Lights——所有灯光的渲染侧记账对象
  • 各种加速结构（八叉树等）供可见性剔除使用

FScene 只能在渲染线程上读写。游戏线程通过 ENQUEUE_RENDER_COMMAND（见 0010）向它注册/更新/移除图元，绝不直接操作。

为什么要两份，不能共享一份？

游戏线程与渲染线程并行运行、各差一帧（见 0005 流水线图）。若渲染线程直接读 UWorld，游戏线程在并行 Tick、改 Actor 位置，结果是不可预测的数据竞争。两份场景以单向命令队列解耦，是整个 CPU 侧架构的根本约束。

2三个易混的 Primitive 类

围绕一个可渲染图元，有三个类，各归不同所有者、不同线程：

类名 所属线程 所有者 生命周期 职责
UPrimitiveComponent 游戏线程 Actor（UObject 体系） 与 Actor 共存亡 Gameplay 可读写的属性（位置、材质引用、可见性…）；是 Tick 和蓝图操作的目标
FPrimitiveSceneProxy 渲染线程（只读） 渲染线程（由组件创建，渲染线程管理） 组件注册到 FScene 时创建，反注册时销毁 渲染所需数据的不可变镜像；按组件类型各有子类（如 FStaticMeshSceneProxy）；提供 GetDynamicMeshElements / DrawStaticElements
FPrimitiveSceneInfo 渲染线程 FScene 随图元在 FScene 注册/反注册 渲染器内部记账：存可见性 ID、光照缓存、阴影所需数据、到代理的反向引用等；非面向组件逻辑，而是面向渲染调度
关键洞见 · Proxy 是"渲染时需要的所有数据的自包含副本"

FPrimitiveSceneProxy 构造时从 UPrimitiveComponent 拷贝渲染所需的全部数据（顶点工厂引用、材质实例、包围盒、LOD 偏差…），此后游戏线程改 Component 时，渲染线程仍读稳定的副本。若游戏线程改了 Component 上的材质，必须通过 MarkRenderStateDirty → 帧末推送 → 重建 Proxy（或增量更新 Proxy），不能绕过。

3FPrimitiveSceneProxy 的子类与 Mesh 提交接口

每种几何类型有自己的 Proxy 子类，在 UPrimitiveComponent::CreateSceneProxy() 里按类型创建：

Component 类型Proxy 子类文件
UStaticMeshComponentFStaticMeshSceneProxyStaticMeshRender.cpp
USkeletalMeshComponentFSkeletalMeshSceneProxySkeletalMeshRender.cpp
ULandscapeComponentFLandscapeComponentSceneProxyLandscapeRender.cpp

Proxy 向渲染器提交几何有两条路径：

  • DrawStaticElements(FStaticPrimitiveDrawInterface*)：注册时调用一次，把不变的几何缓存为 FStaticMesh（进 FMeshBatch 静态路径，见 0018）。适合静态网格。
  • GetDynamicMeshElements(Views, ViewFamily, VisibilityMap, Collector)：每帧 InitViews 里调用，动态生成 FMeshBatch（适合骨骼网格、粒子等每帧变化的几何）。
游戏线程UPrimitiveComponent（UObject 生命周期）· Tick 改属性 · MarkRenderStateDirty
渲染线程FPrimitiveSceneProxy（只读镜像，按类型子类化）· FPrimitiveSceneInfo（FScene 内部记账）· 向 InitViews 提供 FMeshBatch

4灯光的对应关系

灯光同样有两份表达，逻辑完全对称：

  • ULightComponent（游戏线程）→ CreateSceneProxy() → FLightSceneProxy（渲染线程只读镜像，LightSceneProxy.h）
  • FLightSceneInfo（渲染器内部记账，由 FScene 持有，LightSceneInfo.h）

与 Primitive 三件套完全一一对应——理解了 Primitive 侧，灯光侧无需额外记忆。

检索练习

下列关于 FPrimitiveSceneInfo 的描述，哪一条最准确？

从 UStaticMeshComponent 开始，顺着线程边界列出它在游戏侧和渲染侧的完整对应关系（对象名 + 所属线程 + 职责一句话）。
游戏线程：UStaticMeshComponent（UObject，Actor 子对象）——Gameplay 可读写的全部属性（位置、材质引用、可见性、碰撞等）。
渲染线程（镜像）：FStaticMeshSceneProxy（FPrimitiveSceneProxy 子类，StaticMeshRender.cpp）——渲染所需数据的只读副本；实现 DrawStaticElements（缓存静态路径）和 GetDynamicMeshElements（动态提交 FMeshBatch）。
渲染线程（记账）：FPrimitiveSceneInfo（FScene 持有）——存可见性 ID、阴影/光照缓存、指向 Proxy 的指针；是渲染调度的内部索引，非组件镜像。

带走这几条

  • UWorld 是游戏线程的世界；FScene（Scene.cpp）是渲染线程的世界，持有 FPrimitiveSceneInfo / FLightSceneInfo 数组，只能在渲染线程上操作。
  • 一个图元有三个对象：UPrimitiveComponent（游戏态，可读写）、FPrimitiveSceneProxy（渲染态只读镜像，按类型子类化）、FPrimitiveSceneInfo（渲染器内部记账）。
  • FPrimitiveSceneProxy 通过 DrawStaticElements（一次缓存）或 GetDynamicMeshElements（每帧动态）向渲染器提交 FMeshBatch。
  • 灯光完全对称：ULightComponent → FLightSceneProxy → FLightSceneInfo。
  • 两份场景是强制设计，目的是让两条线程并行运行时零数据竞争。
