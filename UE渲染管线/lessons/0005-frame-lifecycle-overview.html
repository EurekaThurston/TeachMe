一帧的生命周期：三线程渲染架构总览

这节课是整门课程的脊柱。我们用一张图把"从 FEngineLoop::Tick 的第一行，到像素被 Present 上屏"的全过程串起来——之后每一节课，都是在放大这张图里的某一段。

时长 ~18 分钟 前置 Part 0 全部（尤其 0004 流水线） 产出 能凭记忆画出三线程 + GPU 的一帧全景，并说出每段的入口与职责

UE 一帧看似有几百个步骤，但骨架只有四个角色，按固定顺序接力：游戏线程决定"这帧场景是什么样"，渲染线程把它翻译成"该画哪些 Pass、怎么画"，RHI 线程把指令翻译成 D3D12 调用，GPU 真正画并 Present 上屏。Part 0 学的命令列表、围栏、CPU/GPU-bound，全在这张图里各就各位。

1四个角色，一条接力线

游戏线程决定场景内容 · FEngineLoop::Tick → World/Actor Tick → 更新变换/动画 → 把变化入队给渲染线程
渲染线程组织怎么画 · 消费命令 → 可见性/剔除 → 构建 RDG → 逐 Pass 录制 RHI 命令
RHI 线程翻译成图形 API · 把平台无关的 RHI 命令翻成 D3D12 命令列表 → 提交队列
GPU执行 + 上屏 · 跑光栅化管线 → 写各缓冲 → 后处理 → Present
核心切分 · 为什么是"两份场景"

游戏线程持有 UWorld / AActor / UPrimitiveComponent（可被 Gameplay 任意修改）；渲染线程持有它们的只读镜像 FScene / FPrimitiveSceneProxy。渲染从不直接碰游戏对象——否则两线程并行时必然数据竞争。两份表达之间靠 ENQUEUE_RENDER_COMMAND 单向同步。这条"游戏态 vs 渲染态"的分界，是理解整个 CPU 侧的总钥匙（Part 2 全在讲它）。

2游戏线程：从 Tick 到入队

一切始于引擎主循环 FEngineLoop::Tick()。1 它驱动 UWorld::Tick，逐个 Tick Actor 与 Component——移动、动画、物理、Gameplay 逻辑都在这里改变世界状态。Tick 结束后，引擎把这一帧的变化（新的变换、新生成/销毁的图元、可见性变更）通过渲染命令投递给渲染线程，然后调用 UGameViewportClient::Draw 一路触发 FRendererModule::BeginRenderingViewFamily，把"渲染这一帧"这件事踢给渲染线程。

代码锚点（先记位置，Part 2 精读）：FEngineLoop::Tick 在 LaunchEngineLoop.cpp；UWorld::Tick 在 LevelTick.cpp；入队宏 ENQUEUE_RENDER_COMMAND 在 RenderingThread.h。 
// 游戏线程把"修改渲染状态"打包成 lambda 投给渲染线程

3渲染线程：场景渲染器的总调度

渲染线程拿到 ViewFamily 后，创建一个场景渲染器。PC 延迟着色路径用的是 FDeferredShadingSceneRenderer，它的 Render() 函数就是整条 GPU 管线的总编排者。2 它大致分两段：

代码锚点：FDeferredShadingSceneRenderer::Render 在 DeferredShadingRenderer.cpp；可见性在 InitViews（SceneVisibility.cpp）。0014 会逐段精读这个函数。
  1. InitViews（准备）：视锥/距离/遮挡剔除算出每个视图能看见哪些图元（Part 4），为各 Pass 生成 FMeshDrawCommand，建立动态阴影等。这一段决定"画什么"。
  2. Render（执行）：按延迟着色顺序，用 RDG 把每个 Pass 声明进依赖图，再编译执行（Part 3 / Part 5 / Part 6）。这一段决定"怎么画、画成什么"。

在 4.26 里，Render() 内部越来越多地用 RDG 来组织 Pass：不立即执行，而是先把整帧 Pass 录成依赖图，让 RDG 自动管理资源生命周期、屏障与异步计算（你在 0003/0002 已理解它要自动化的是什么）。RDG 是 Part 3 的重头戏。

5上屏：像素如何到达玩家

后处理链的最后一步把图像写进交换链的后缓冲，然后 Present 把它提交到显示队列。显示控制器按刷新率把前缓冲逐行扫描输出到显示器——这一刻，像素真正被玩家看到（细节见 0038/0039）。至此，从游戏线程那一帧的 Tick，到玩家眼睛，闭环完成。

检索练习

渲染线程为什么操作 FScene/FPrimitiveSceneProxy 这套镜像，而不直接读 UWorld/UPrimitiveComponent？

不看图，按顺序说出 UE 一帧延迟着色的宏观 Pass 序列（从深度到上屏），并指出每个 Pass 内部与 0001 硬件管线的关系。
顺序：Depth Prepass（只写深度，建 Early-Z）→ Base Pass（写 GBuffer）→ 光照剔除（Light Grid）→ 阴影（ShadowDepth）→ 延迟光照 → 屏幕空间（SSAO/SSR/SSGI）→ 半透明 → 后处理（TAA/Bloom/Tonemap）→ Present 上屏。关系：每个 Pass 内部都是把 0001 的固定硬件管线 IA→VS→光栅化→PS→OM 用一组特定的着色器 + 渲染目标再跑一遍——例如 Base Pass 用材质着色器把数据写进多个 GBuffer（MRT），阴影 Pass 用极简着色器只写深度。管线不变，变的是"绑哪个 PSO、读写哪些缓冲"。

带走这几条

  • 一帧四角色接力：游戏线程（画什么）→渲染线程（怎么画）→RHI 线程（翻成 D3D12）→GPU（画+Present）；三级流水各落后一帧。
  • CPU 侧的总钥匙是两份场景：游戏态 UWorld/Component ↔ 渲染态 FScene/Proxy，靠 ENQUEUE_RENDER_COMMAND 单向同步。
  • 渲染编排的核心函数是 FDeferredShadingSceneRenderer::Render()：先 InitViews（剔除、生成绘制命令）再 Render（用 RDG 逐 Pass 执行）。
  • GPU 一帧的宏观 Pass 序列 = 本课程 Part 4–6 的章节顺序；每个 Pass 内部都是 0001 硬件管线的一次复用。
