延迟着色总览与 GBuffer 布局

延迟着色是 UE 4.26 PC 路径的核心承诺：把"几何有哪些像素"与"像素怎么着色"拆成两个独立的 Pass，让光照成本彻底与场景复杂度解耦。这节课从动机出发，精确到每个通道存什么。

时长 ~18 分钟 前置 0001（MRT、OM 阶段）、0005（帧全景） 产出 能解释延迟着色的动机，并说出 UE 4.26 GBuffer 各通道存什么

前向渲染里，每个可见片元的像素着色器必须立刻对所有影响它的光源求和——光越多、片元越多，开销直接叠加。延迟着色把"求和"这步搬到一个专门的屏幕空间 Pass：BasePass 只负责把材质属性写进一组纹理（GBuffer），光照 Pass 再读这些纹理逐光源叠加。几何复杂度决定写的代价，光源数量决定读的代价，两者不再耦合。你会在 0001 里学到的 MRT 机制（SV_Target0~N 同时写出）正是让这套思路成立的硬件基础。

1前向 vs 延迟：成本模型对比

设场景有 G 个可见片元、L 盏动态光：

路径着色调用次数什么时候变慢
前向渲染G × L（最坏情况每片元循环所有光）光源多 + 复杂场景几何双重叠加
延迟着色G（写 GBuffer）+ L × 受影响像素（读 GBuffer 着色）光源多时只多花"受影响像素×L"；几何复杂不加重光照
延迟的代价

延迟着色不是免费的：GBuffer 占显存带宽（读+写），半透明物体无法直接延迟着色（需前向路径或单独处理），MSAA 代价高昂（每采样点都要存 GBuffer）。UE 4.26 针对这些限制分别有对策——0029 讲半透明，抗锯齿依赖 TAA 而非 MSAA。

2UE 4.26 GBuffer 完整布局

UE 4.26 的 GBuffer 由以下渲染目标组成。除 SceneDepth 外，其余均通过 BasePass 的 MRT 一次写出。在着色器侧，DeferredShadingCommon.ush 里的 FGBufferData 结构体是所有编码/解码逻辑的统一入口。

缓冲名格式（典型）存储内容
SceneColorR11G11B10F 或 RGBA16FHDR 场景颜色（光照前为清空或 Emissive；光照后累积直接光；最终供后处理）
SceneDepthD32F / D24S832-bit 线性深度。Depth Prepass（0022）先填充；BasePass 用 Early-Z 复用
GBufferARGBA8RGB = WorldNormal（编码为 oct 或 signed 格式）；A = Per-Object GBuffer 数据（如自定义深度等标记）
GBufferBRGBA8R = Metallic；G = Specular；B = Roughness；A[4:0] = ShadingModelID；A[5] = SelectiveOutputMask
GBufferCRGBA8 (sRGB)RGB = BaseColor；A = GBuffer AO（预烘焙或 SSAO 结果）
GBufferDRGBA8各 ShadingModel 自定义数据（如 Subsurface Profile 用 R 存 Profile ID；Clear Coat 用 R=ClearCoat, G=Roughness）
GBufferERGBA8预计算阴影因子（Precomputed Shadow Factors）；可选，按需分配
VelocityRG16F每像素屏幕空间运动向量，供 TAA（0032）和运动模糊（0035）使用；由 r.BasePassOutputsVelocity 控制是否在 BasePass 内输出
ShadingModelID 的作用

GBufferB 的高四位存 ShadingModelID（最多 16 种模型）。光照 Pass 读出这个 ID 后，用 switch 跳转到对应的 BRDF 实现（ShadingModels.ush）——这就是"一个 GBuffer 支持多种材质模型"的机制。UE 4.26 内置模型包括 DefaultLit（0）、Subsurface（1）、PreintegratedSkin（2）、ClearCoat（3）、SubsurfaceProfile（4）、TwoSidedFoliage（5）等。

3着色器侧：FGBufferData 与编码/解码

所有对 GBuffer 的读写，都经由 Engine/Shaders/Private/DeferredShadingCommon.ush 里两个函数完成：

// DeferredShadingCommon.ush（节选，架构示意）
struct FGBufferData
{
    float3 WorldNormal;       // GBufferA.rgb 解码
    float  Metallic;          // GBufferB.r
    float  Specular;          // GBufferB.g
    float  Roughness;         // GBufferB.b
    float3 BaseColor;         // GBufferC.rgb (linear)
    float  GBufferAO;         // GBufferC.a
    uint   ShadingModelID;    // GBufferB.a 低 4 bit
    uint   SelectiveOutputMask;
    float4 CustomData;        // GBufferD
    float4 PrecomputedShadowFactors; // GBufferE
};

// BasePass 写：把材质输出填进 FGBufferData，再编码进 MRT
EncodeGBuffer(GBuffer, OutTarget0, OutTarget1, ...);

// 光照 Pass 读：从 MRT 采样，解包成 FGBufferData 供 BRDF
FGBufferData GBuffer = DecodeGBufferData(
    GBufferATexture, GBufferBTexture, GBufferCTexture,
    GBufferDTexture, GBufferETexture,
    SceneDepthTexture, ScreenUV);

CPU 侧，GBuffer 纹理集合由 FSceneTextures（SceneTextures.h）管理，通过 FSceneTexturesUniformParameters 以 Uniform Buffer 的形式绑定到着色器——光照 Pass 直接读这个 UB，无需手动绑定每张纹理。

4法线编码细节

GBufferA.rgb 用 8-bit 精度存世界法线，直接存 [-1,1]³ 会损失精度。UE 4.26 默认用压缩-符号编码：把 [-1,1] 映射到 [0,1] 再写入 RGBA8（即 N * 0.5 + 0.5），解码时反向。对于需要更高精度的路径（如开启 r.GBuffer 精度升级），可换用 RGBA16F。GBufferA.a 存每对象数据（如选择遮罩、自定义深度 stencil 标记），与法线并不干扰。

UE5 超纲提示

UE5 引入 Substrate 材质系统，彻底重写了 GBuffer 布局与 ShadingModel 分发逻辑；Lumen 也绕开了传统 GBuffer 读取路径。这些均超出本课程范围——本课所有内容严格锁定 UE 4.26。

不看表，按顺序说出 UE 4.26 的 GBuffer 通道及其存储内容，并解释为什么 SceneDepth 由 Depth Prepass（而非 BasePass）填充。
通道顺序：SceneColor（HDR 颜色）、SceneDepth（32-bit 线性深度）、GBufferA（WorldNormal + per-object）、GBufferB（Metallic/Specular/Roughness + ShadingModelID）、GBufferC（BaseColor + GBuffer AO）、GBufferD（自定义数据）、GBufferE（预计算阴影）、Velocity（运动向量）。SceneDepth 由 Depth Prepass（0022）先填充是为了建立 Early-Z——让 BasePass 的像素着色器只在最前可见片元上运行，避免对被遮挡片元执行昂贵的材质着色；这是整条延迟路径的性能基础。

带走这几条

  • 延迟着色把"几何→GBuffer"（BasePass）与"GBuffer→光照"（Lighting Pass）分离，光照成本与几何复杂度解耦——核心动机。
  • UE 4.26 GBuffer 有 8 个主要通道：SceneColor、SceneDepth、GBufferA（法线）、GBufferB（Metallic/Specular/Roughness/ShadingModelID）、GBufferC（BaseColor/AO）、GBufferD（自定义）、GBufferE（阴影因子）、Velocity。
  • FGBufferData + EncodeGBuffer / DecodeGBufferData 是着色器侧读写 GBuffer 的统一接口，定义在 DeferredShadingCommon.ush。
  • CPU 侧 FSceneTextures 管理 GBuffer 资源，通过 FSceneTexturesUniformParameters 绑定到光照着色器。
  • 延迟的缺陷：显存带宽高、半透明需单独处理、MSAA 昂贵——UE 4.26 用 TAA 替代 MSAA。
