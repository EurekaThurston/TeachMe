后处理管线概述

场景颜色从 HDR 线性出发，经 TAA、景深、动态模糊、泛光、曝光、色调映射一路走到 LDR——这节课把整条后处理链的结构和 RDG 接线方式搞清楚。

时长 ~18 分钟 前置 0009（RDG 基础）、0030（延迟着色完成前的场景颜色） 产出 能说出后处理链的整体顺序、每个 Pass 的输入/输出类型、FPostProcessSettings 如何决定哪些 Pass 运行

延迟着色结束后，SceneColor 是一张 HDR 线性颜色缓冲——它存储的值可以超过 1.0，颜色空间是线性 sRGB，没有 gamma，没有曝光压缩，不能直接送到显示器。后处理链的职责就是把这张缓冲变成一张能被显示器正确呈现的 LDR（或 HDR 输出）图像，同时叠加所有美术驱动的视觉效果。整条链在 UE 4.26 里被组织成一个 RDG Pass 序列，入口是 PostProcessing.cpp 里的 AddPostProcessingPasses。

1场景颜色的生命周期：从 HDR 到 LDR

理解后处理链的前提是搞清楚 SceneColor 的"格式旅程"。它在不同阶段使用不同的像素格式和值域：

阶段格式（典型）值域颜色空间
Base Pass 输出 / 光照后PF_FloatRGBA（R16G16B16A16F）[0, ∞)线性 sRGB
TAA / 大部分后处理中间步PF_FloatRGBA 或 PF_FloatR11G11B10[0, ∞)线性 sRGB
色调映射输出PF_B8G8R8A8 或 PF_A2B10G10R10[0, 1]sRGB（含 gamma）
UI 合成后 / 送 Present同上（取决于交换链格式）[0, 1]sRGB / Rec.709

PF_FloatR11G11B10 是 RGB 各 11/11/10 bit 的无符号浮点格式，UE 4.26 在内存受限时可选用它替代 R16G16B16A16F 存场景颜色，省约一半显存，但无法存负值和透明度。

关键约束：在色调映射之前，所有 Pass 都工作在线性 HDR 域。这意味着泛光的亮度加法、曝光乘法、SSAO 乘法，全部在线性空间进行，结果物理正确。色调映射是这条链里唯一一个把值域从 HDR 压缩到 [0,1] 的 Pass，它也是线性→gamma 的转换点。

为什么 TAA 在最前面

TAA 需要读取 GBuffer 的速度通道（Velocity），该通道在 Base Pass 写入、在后处理开始时还"新鲜"。而且 TAA 的核心工作是对齐当前帧和历史帧，如果先跑泛光，泛光产生的低频光晕会污染历史帧重投影，产生鬼影和闪烁。所以 TAA 是后处理链里第一个写 SceneColor 的 Pass。

例外：运动模糊在某些配置下可以在 TAA 之前或之后——UE 4.26 默认把它放在 TAA 之后、泛光之前，以避免运动模糊消耗未经 TAA 处理的含锯齿图像。

3AddPostProcessingPasses：RDG 接线的总入口

Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp 里的 AddPostProcessingPasses 函数是整条后处理链的组织者。它的职责不是执行任何着色工作，而是按条件注册 RDG Pass：

// PostProcessing.cpp — 结构示意（非原文逐字）
void AddPostProcessingPasses(
    FRDGBuilder& GraphBuilder,
    const FViewInfo& View,
    const FPostprocessContext& Context)
{
    // 1. TAA（若 AA 模式 == TAA）
    if (View.AntiAliasingMethod == AAM_TemporalAA)
        SceneColor = AddTemporalAAPass(GraphBuilder, View, ...);

    // 2. 景深（若 FPostProcessSettings::bDOF）
    if (IsDOFEnabled(View))
        SceneColor = AddDiaphragmDOFPass(GraphBuilder, View, SceneColor, ...);

    // 3. 动态模糊
    if (IsMotionBlurEnabled(View))
        SceneColor = AddMotionBlurPass(GraphBuilder, View, SceneColor, ...);

    // 4. 泛光下采样 + 上采样
    FRDGTextureRef BloomOutput = AddBloomPass(GraphBuilder, View, SceneColor);

    // 5. 眼部适应（直方图 → 曝光值）
    FRDGBufferRef EyeAdaptation = AddEyeAdaptationPass(GraphBuilder, View, SceneColor);

    // 6. 色调映射 + LUT：HDR → LDR
    SceneColor = AddTonemapPass(GraphBuilder, View, SceneColor,
                                   BloomOutput, EyeAdaptation, ...);
    // SceneColor 现在是 LDR sRGB
}

每一步返回的 SceneColor 是 FRDGTextureRef——一个 RDG 虚拟纹理句柄，没有实际分配显存，直到 RDG 编译执行时才确定别名与分配。

注意：这里并不是真正按顺序执行，而是在 RDG Builder 里录制 Pass 依赖图。所有 AddXxxPass 调用结束后，调用 GraphBuilder.Execute() 才会按拓扑序真正执行。RDG 会自动：

  • 在 Pass 之间插入 资源屏障（Resource Barrier），把 SceneColor 从写目标切换为着色器可读；
  • 对生命周期不重叠的中间缓冲做内存别名（Aliasing），节省峰值显存；
  • 把不依赖图形管线的 Pass（如曝光直方图）调度到异步计算（Async Compute）队列。

4FPostProcessSettings：数据驱动哪些 Pass 运行

FPostProcessSettings 是 Scene.h 中定义的一个大结构体，聚合了所有后处理效果的参数。它由 APostProcessVolume 或相机组件的 PostProcessSettings 字段填写，在渲染线程通过 FSceneView 传入。

字段（选取）控制的效果典型范围

AddPostProcessingPasses 在注册每个 Pass 之前，都会读取 View.FinalPostProcessSettings（已经过所有 PostProcessVolume 的混合和优先级叠加后的最终值）做条件判断。例如 BloomIntensity == 0 时泛光 Pass 整体跳过，避免一点开销都不浪费。

关键洞见 · Pass 跳过是编译期决策

RDG 里的"不注册 Pass"和"注册 Pass 但 PS 里做 discard"代价不同。前者在 RDG 编译期就消失，相关的资源分配、屏障、命令录制全部不存在；后者仍然有 draw call 和着色器开销。UE 4.26 后处理设计的正确方式：在 AddPostProcessingPasses 里用 C++ 条件跳过注册，而不是在 HLSL 里用 if 分支。

5SceneColor 的"Ping-Pong"模式与 RDG 别名

后处理链里每个 Pass 读当前 SceneColor、写新的 SceneColor，形成一种"乒乓"（Ping-Pong）缓冲模式——每步产出一个新的 FRDGTextureRef，旧的在之后的 Pass 不再写入。RDG 能检测到哪些中间纹理的生命周期不重叠，并把它们映射到同一块显存——这就是内存别名。

典型情况：TAA 输出、景深中间缓冲、泛光各层级下采样，在同一帧里可以共享显存，因为它们不同时存活。对于一帧后处理链来说，显存峰值可能只需要 2–3 张全分辨率 HDR 纹理，而不是每个 Pass 独占一张。

RDG Transient Resources vs. External Resources

用 GraphBuilder.CreateTexture() 创建的是瞬态（Transient）纹理，生命周期由 RDG 管理，可参与别名。而历史帧缓冲（如 TAA 历史）是外部（External）资源，用 GraphBuilder.RegisterExternalTexture() 注入图中，不参与别名——因为它需要跨帧持续存在。

6后处理与主渲染器的接缝

AddPostProcessingPasses 的调用点在 DeferredShadingRenderer.cpp 的 FDeferredShadingSceneRenderer::Render 函数里，位于所有几何 Pass、光照 Pass、半透明 Pass、体积雾之后。调用时传入的 SceneColor 已经是包含了直接光照 + 间接光照 + 半透明的合并结果。

后处理链结束后，SceneColor 变为 LDR sRGB，交给 UI/HUD 合成 Pass（Slate 渲染的 UI 叠加在最顶层），然后通过 RHI Present 提交到交换链。这也是后续 0036 课（RHI Thread 与 RHICommandList）的起点。

ScreenPercentage 与后处理分辨率

ScreenPercentage（r.ScreenPercentage）控制场景颜色的内部渲染分辨率。设为 50% 意味着 SceneColor 宽高各是最终输出的一半，大部分后处理 Pass（TAA、景深、动态模糊、泛光）都在这个低分辨率下工作，节省带宽。空间上采样（Primary Screen Percentage Upscale）在色调映射附近把图像放大到原始分辨率。这个上采样 Pass 的位置很关键——它之前的 Pass 省带宽，之后的 Pass（UI 合成、HUD）在全分辨率工作，保证 UI 清晰度。

检索练习

UE 4.26 后处理链中，哪个 Pass 是 HDR → LDR 的转换点？在此之前，场景颜色处于什么状态？

不看图，说出 UE 4.26 后处理链从 TAA 到 Present 的 Pass 顺序及每步的输入/输出格式变化。
  1. TAA：输入 HDR SceneColor + 速度缓冲（GBuffer Velocity）+ 历史帧纹理 → 输出去锯齿的 HDR SceneColor（FloatRGBA）。
  2. SSR（可选）：输入 SceneColor + GBuffer 法线/深度 → 输出叠加了屏幕空间反射的 HDR SceneColor。
  3. 景深（可选）：输入 HDR SceneColor + 深度 → 输出散焦模糊的 HDR SceneColor。
  4. 动态模糊（可选）：输入 HDR SceneColor + 速度缓冲 → 输出运动拖尾的 HDR SceneColor。
  5. 泛光：下采样金字塔提取高亮 → Gaussian 上采样 → 输出泛光纹理（HDR），不直接写 SceneColor，而是在色调映射时作为 add 混入。
  6. 自动曝光：从 SceneColor 构建亮度直方图 → 输出曝光值缓冲（1×1 R32F）。
  7. 色调映射：输入 HDR SceneColor + 泛光 + 曝光值 + 颜色分级 LUT → 输出 LDR sRGB（B8G8R8A8）。
  8. UI 合成：UI/HUD 叠加在 LDR SceneColor 上 → 提交 RHI Present。

带走这几条

  • 后处理链入口是 PostProcessing.cpp 的 AddPostProcessingPasses，它负责按条件向 RDG 注册 Pass，不直接执行。
  • SceneColor 全程保持 HDR 线性浮点格式，直到色调映射 Pass 才压缩到 LDR [0,1]。
  • Pass 顺序：TAA → 景深 → 动态模糊 → 泛光 → 自动曝光 → 色调映射 → UI → Present。
  • FPostProcessSettings 控制哪些 Pass 运行，条件判断在 C++ 注册阶段完成，False 则整个 Pass 不进 RDG 图。
  • RDG 的内存别名让中间缓冲共享显存；TAA 历史缓冲是 External Resource，不参与别名，跨帧存活。
  • ScreenPercentage 决定大部分后处理在低分辨率工作，空间上采样在色调映射附近把分辨率恢复。
