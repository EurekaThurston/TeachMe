总调度：FDeferredShadingSceneRenderer::Render() 全景

FDeferredShadingSceneRenderer::Render() 是整条 GPU 管线的总编排者——它不直接画一个像素，它决定"以什么顺序、用什么参数"把每一个 Pass 送上 GPU。理解它的宏观结构，等于掌握了后面七个 Part 的目录索引。

时长 ~22 分钟 前置 0005（帧全景）、0013（渲染入口） 产出 能复述 Render() 的宏观执行顺序，并知道每段对应后面哪个 Part

Render(FRHICommandListImmediate&) 定义在 Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp，在 UE 4.26 源码里超过 1000 行。你不需要现在读完它——你需要的是知道它的骨架：哪几段、以什么顺序、调什么函数。这节课就是那张骨架图。

2第一段：InitViews ——"画什么"

InitViews(FRHICommandListImmediate&, ...) 定义在 SceneVisibility.cpp。它是 Render() 里最先运行、也是 CPU 开销最重的一段，负责回答"这帧要画哪些物体"。主要工作：

  • 视锥剔除（Frustum Culling）：快速排除不在视锥内的图元。
  • 遮挡剔除（Occlusion Culling）：HZB/硬件遮挡查询（从上帧结果读取）。
  • GPU Scene 更新：把当帧新增/修改的图元变换上传到 GPUScene 缓冲（GPUScene.cpp）。
  • 生成 FMeshDrawCommand：为每个可见图元的每个 Pass 调用对应的 FMeshPassProcessor，把 FMeshBatch 转成缓存好的 FMeshDrawCommand（Part 4 精读）。
  • 动态阴影初始化：为这一帧的光源确定阴影 Cascade 分配。
InitViews 完成后，整帧"画什么"已完全确定

从 InitViews 返回时，每个 Pass 的 FMeshDrawCommand 列表已就绪，GPU Scene 已同步。后续所有 Pass 只是消费这些命令——它们不会再改变"画什么"的决策。这条分界线是理解 UE 渲染 CPU 开销分布的关键：InitViews 通常占 CPU 渲染侧 30–50% 的耗时。

3RDG 在 Render() 里的位置

在 UE 4.26 中，Render() 函数体早期就构造了 RDG 构建器：

// DeferredShadingRenderer.cpp — 简化示意
void FDeferredShadingSceneRenderer::Render(FRHICommandListImmediate& RHICmdList)
{
    InitViews(RHICmdList, /* ... */);

    // 4.26 起越来越多 Pass 迁移到 RDG
    FRDGBuilder GraphBuilder(RHICmdList);

    RenderPrePassAndVelocity(GraphBuilder, /* ... */);
    RenderBasePass(GraphBuilder, /* ... */);
    ComputeLightGrid(GraphBuilder, /* ... */);
    RenderShadowDepthMaps(GraphBuilder, /* ... */);
    RenderLights(GraphBuilder, /* ... */);
    RenderTranslucency(GraphBuilder, /* ... */);
    AddPostProcessingPasses(GraphBuilder, /* ... */);

    GraphBuilder.Execute();   // 编译图、分配资源、执行所有 Pass
}

注意 FRDGBuilder GraphBuilder(RHICmdList) 构造在函数靠前处，多数 Pass 以 GraphBuilder 为参数调用——它们只是向图里 声明 Pass，并不立即执行。直到 GraphBuilder.Execute()，整帧的 RDG 图才被编译、优化、按依赖序执行（见 0015/0016）。

5几个重要的全局状态

在进入各 Pass 之前，Render() 还设置几个全局影响后续的状态：

  • GIsRenderingThreadSuspended：调试用标志，正常渲染时为 false。
  • bIsWireframe / ViewModeIndex：由编辑器视口决定，影响各 Pass 是否跳过或改写着色器。
  • FGlobalDynamicIndexBuffer / FGlobalDynamicVertexBuffer：帧内临时几何（粒子、UI、Debug 图元）的内存池，在帧末 Reset。

顺序：① InitViews（SceneVisibility.cpp，最先执行）→ ② FRDGBuilder GraphBuilder(RHICmdList)（RDG 构建器构造）→ ③ RenderPrePassAndVelocity → ④ RenderBasePass → ⑤ BeginOcclusionTests（遮挡查询）→ ⑥ ComputeLightGrid → ⑦ RenderShadowDepthMaps → ⑧ RenderLights → ⑨ RenderReflections（SSR）→ ⑩ RenderTranslucency → ⑪ AddPostProcessingPasses → ⑫ GraphBuilder.Execute()（在函数末尾，编译并执行整帧 RDG 图）。

带走这几条

  • FDeferredShadingSceneRenderer::Render()（DeferredShadingRenderer.cpp）是整条 GPU 管线的总编排——它不画像素，它决定顺序与参数。
  • 第一段 InitViews 决定"画什么"（剔除 + FMeshDrawCommand 生成），之后所有 Pass 只是消费结果。
  • 4.26 起大量 Pass 迁移到 FRDGBuilder：先声明、后 Execute()——整帧一次性编译执行（0015 展开）。
  • 这张宏观顺序表是课程 Part 4–7 的目录：每一行对应一节或几节课，方向迷失时回来对照。
