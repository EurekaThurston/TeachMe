GPU 执行时间线与扫描输出

GPU 消费命令队列、Signal 围栏、DXGI 把后缓冲送到显示控制器，显示控制器按刷新率逐行扫描——这条链的每一环都有固定时序，也各自引入延迟。本课把这条链从头到尾走完，闭合整个课程的大循环：从 0008 的 CPU Tick，到光子击中玩家眼睛。

时长 ~25 分钟 前置 0008（EngineLoop Tick）、0036（RHI 线程）、0038（交换链与 Present） 产出 能画出一帧从 CPU Tick 到像素发光的完整时间线；理解 GPU Fence 机制、扫描输出时序、撕裂成因、VRR 原理和输入-到-光子延迟的各组成部分

整条课程从"GPU 有哪些硬件阶段"讲到"RHI 线程把命令提交给 D3D12 队列"。但 Present 之后 GPU 还要实际执行命令，执行完了 DXGI 才把缓冲交给显示控制器，显示控制器还要等下一个 VBlank 才能扫描。这最后一英里，也是帧率和延迟数字的真正来源。

1GPU 命令队列的执行模型

CPU 提交的 D3D12 命令列表进入 GPU 的命令队列（ID3D12CommandQueue）后，GPU 异步地、串行地（在该队列内）消费它们。这里有几个关键特性：

  • 命令列表是原子单元：队列内的命令列表按提交顺序执行，不会乱序——但不同命令列表内部的 Draw Call 可以被 GPU 内部并行（固定管线的各阶段在不同 Shader Core 上流水）。
  • GPU 不回报进度：CPU 无法得知 GPU 当前执行到哪条命令——只能用围栏（Fence）确认"GPU 已执行到某个提交点"。
  • 多个队列可并行：D3D12 有图形队列、计算队列、复制队列。UE 的 RDG 异步 Compute Pass 利用计算队列与图形队列并行（0016 已述），它们之间用队列间 Fence 同步。

2GPU Fence：CPU 与 GPU 的同步原语

ID3D12Fence 是 CPU 与 GPU 之间（也可以是 GPU 队列间）的同步原语。使用模式：

// D3D12RHI 层的简化示意

// 1. CPU 把"到达这个 Signal 时请把 Fence 值设为 N"插入命令队列
CommandQueue->Signal(Fence, FenceValue);

// 2. GPU 执行完 Signal 之前所有命令后，异步地把 Fence 值置为 N

// 3. CPU 等待（可选：阻塞或轮询）
if (Fence->GetCompletedValue() < FenceValue)
{
    Fence->SetEventOnCompletion(FenceValue, WaitEvent);
    WaitForSingleObject(WaitEvent, INFINITE); // CPU 睡眠直到 GPU 完成
}

GPU Fence 等待范式（D3D12 标准用法）

UE 4.26 的 D3D12RHI 在以下场景使用 Fence：

  • 后缓冲复用保护（0038 已述）：Present 后对命令队列 Signal，下帧开始前等待该 Fence 确认 GPU 已完成对该后缓冲的所有写入。
  • 命令分配器复用：确认 GPU 已执行完对应命令列表后，才允许 Reset 并复用命令分配器（FD3D12CommandAllocatorManager）。
  • 异步 Compute 同步：图形队列等待计算队列完成 SSAO/SSGI 等异步 Pass 后，才读取其结果缓冲。
GetCompletedValue() — GPU 时间线的"里程碑"

ID3D12Fence::GetCompletedValue() 返回 GPU 最后一次 Signal 写入的值。CPU 用它判断"GPU 走到哪了"。这是 CPU 与 GPU 之间仅有的同步语义——CPU 无法拿到 GPU 当前指令地址，只能靠预先插入的 Signal 标记划分已完成区间。设计上和 Git 的 "tagged commit" 很像：你不知道仓库正在写第几行代码，但你知道某个 tag 已经出现了。

3DXGI Present 队列与 VBlank

调用 IDXGISwapChain::Present() 后，该后缓冲进入 DXGI 的 Present 队列（也叫"显示队列"）。DXGI 在下一个 VBlank 信号到来时，把队列头部的缓冲"翻"到前台（即把显示控制器的扫描指针切换到该缓冲）。

4扫描输出时序详解

显示控制器（Display Controller，GPU 的一个固定功能单元）按显示器的刷新时序把前缓冲逐行读出：

  • 有效扫描区（Active Scan）：每 1/刷新率 秒，显示控制器从左上到右下逐行扫描前缓冲，把每个像素的 RGB 值发送给显示面板（信号编码后经 DP/HDMI 传输）。
  • 垂直消隐区（VBlank / Vertical Blanking Interval）：扫描完最后一行后，有一段极短的消隐期（几百微秒到数毫秒）——这是翻转缓冲的安全窗口。VSync 就是让 Present 对齐这个窗口。
  • 帧到达时刻 vs. 扫描开始时刻：即使 Present 在 VBlank 前一瞬完成，第一行像素也要等到下一个活跃扫描期才会真正显示——这意味着"上屏"本身就至少有 0–16.67ms（@60Hz）的显示器扫描延迟。
关键洞见 · 撕裂的物理成因

若 Present（缓冲翻转）发生在活跃扫描期中段，显示控制器上半段读的是旧缓冲数据，下半段读的是新缓冲数据——同一帧画面上出现水平"撕裂线"。VSync（SyncInterval=1）通过把翻转锁定到 VBlank 消隐期来避免这一现象。

5VRR：G-Sync 与 FreeSync

可变刷新率（Variable Refresh Rate, VRR）技术（NVIDIA G-Sync、AMD FreeSync，底层协议为 VESA Adaptive-Sync）允许显示器的刷新率动态跟随游戏的帧率，而不是固定在 60Hz。工作原理：

  • GPU 在 Present 时通知显示器"我有一帧准备好了"，显示器按需触发扫描，而不是每 16.67ms 固定扫描一次。
  • 在帧率范围内（通常 48–144Hz），每帧结束立即显示，无需等待下一个固定 VBlank——兼得无撕裂和低延迟。
  • 帧率超出 VRR 上限时，退化到固定刷新率；低于下限时用"LFC（低帧率补偿）"重复帧避免闪烁。
UE 4.26 与 VRR：引擎侧没有特殊适配

UE 4.26 的 Present 路径本身不包含 VRR 专用代码——VRR 由 DXGI 翻转模型 + 驱动 + 显示器硬件自动协商。开发者需要做的是：① 关闭 r.VSync（SyncInterval=0，让驱动控制刷新时机）；② 若需要 Tearing 标志兼容，在 Present Flags 里加 DXGI_PRESENT_ALLOW_TEARING（G-Sync Compatible / FreeSync Premium 需要）。UE 4.26 的 r.D3D12.AllowTearing 控制台变量管理这个标志。注意：UE5 的 Reflex/VRR 深度集成不属于 4.26 范围，不在本课讨论。

6帧节奏（Frame Pacing）

即使平均帧率达标，帧间时间不均匀（一帧 8ms、下一帧 24ms 交替）会让玩家感受到明显卡顿（"感知帧率"远低于平均帧率）。这就是帧节奏问题。

根源在于 CPU/GPU 流水线的三级滞后（0005/0006）：游戏线程、渲染线程、GPU 各落后一帧，任何一级出现抖动都会向下游传播并放大。常见抖动来源：

  • 游戏线程上的垃圾收集 GC 停顿。
  • 渲染线程遭遇 PSO 缓存未命中（大量阻塞）。
  • GPU 某帧 DrawCall 骤增（如 LOD 级别跳变、阴影图重建）。
  • VSync 模式下 GPU 帧时间略超 16.67ms 导致整帧"错过" VBlank，下帧等待时间变成 33.33ms。
关键洞见 · "VSync 截止"引发的延迟翻倍

60Hz VSync 下，若某帧 GPU 帧时间是 17ms（仅多 0.3ms），该帧会错过当前 VBlank，必须等下一个——延迟从 16.7ms 跳到 33.3ms。玩家感受到明显卡顿。解决方案：确保 GPU 帧时间有余量（≤14ms），或使用 VRR 消除"截止"效应，或使用 NVIDIA Reflex 等框架优化 Present 时机。

7输入-到-光子延迟：完整分解

从玩家按下按键到眼睛看到画面变化，延迟由以下部分累加：

阶段典型延迟（@60fps）UE 内对应的位置
操作系统输入采样延迟1–4msWindows 消息队列 → UE 输入系统
游戏线程 Tick（处理输入 → 更新 Actor 状态）0–16.7ms（等下一帧 Tick）FEngineLoop::Tick()（0008）
渲染线程延迟（落后游戏线程约 1 帧）~16.7msENQUEUE_RENDER_COMMAND 队列（0012）

检索练习

开启 VSync（r.VSync=1，SyncInterval=1）时，某帧 GPU 帧时间为 17ms（60Hz，周期 16.67ms）——该帧的实际显示延迟最接近哪个值？

GPU Fence 的使用方式是什么？UE D3D12RHI 中哪两个场景强制等待 Fence？

GPU Fence 使用步骤：① CPU 调用 ID3D12CommandQueue::Signal(fence, value)，在命令队列里插入一个"到达此处时把 fence 值写为 value"的标记；② GPU 异步执行到该标记后，把 fence 对象的值更新为 value；③ CPU 调用 fence->GetCompletedValue() 轮询，或用 SetEventOnCompletion 让 CPU 睡眠等待，任选其一。

UE D3D12RHI 中两个典型等待场景：① 后缓冲复用保护——每帧开始时等待上一轮使用同一后缓冲的 GPU Fence，确认 GPU 已完成对该缓冲的写入，才允许 CPU 重新指定它为本帧渲染目标；② 命令分配器复用——FD3D12CommandAllocatorManager 等待对应 GPU Fence 确认分配器对应的命令列表已执行完毕，才调用 ID3D12CommandAllocator::Reset()。

列出"输入-到-光子"延迟的 5 个主要组成部分，并指出哪一部分在 VSync 关闭 + 高帧率下缩减最显著。

五个主要组成部分：① OS 输入采样延迟（1–4ms）；② 游戏线程 Tick 延迟（0–16.7ms，等待下一帧 Tick 到来）；③ 渲染线程滞后（约 1 帧，~16.7ms）；④ GPU 执行延迟（2–10ms，取决于帧工作量和命令队列积压深度）；⑤ Present + VBlank 等待（0–16.7ms，VSync 开时最坏情况为完整刷新周期）。

VSync 关闭 + 高帧率下缩减最显著的是 ⑤（Present 等待）和 ② （帧率越高，Tick 间隔越短，输入等待越少）。VSync 关闭让 Present 无需等 VBlank，消除了最坏 16.7ms 的等待；帧率从 60fps 升到 240fps 时，游戏线程 Tick 间隔从 16.7ms 缩到 4.2ms，输入被更快处理。

带走这几条——也是整个课程的总结

  • GPU 以命令队列异步消费命令列表；CPU 用 ID3D12Fence::Signal/Wait 确认 GPU 到达某个提交点——这是 CPU 与 GPU 同步的唯一语义。
  • Present 把后缓冲推入 DXGI Present 队列；显示控制器在下一个 VBlank 翻转到前台扫描输出——这是"像素真正被看到"的最后一步。
  • 撕裂 = 翻转发生在活跃扫描期中段；VSync（SyncInterval=1）把翻转锁到 VBlank 消隐期避免撕裂，但 GPU 帧时间超过刷新周期哪怕一点点，延迟就翻倍（VSync 截止效应）。
  • VRR（G-Sync/FreeSync）让显示器按需触发扫描，同时消除撕裂和 VSync 截止效应；UE 4.26 通过 r.D3D12.AllowTearing 配合 DXGI 翻转模型支持 VRR，无需引擎特殊适配。
  • 输入-到-光子延迟 = OS采样 + Tick等待 + 渲染线程滞后 + GPU执行 + Present等待 + 扫描延迟；每一环都有工程根因，也都有对应的优化手段——这条分解是性能调优的地图。
  • 大循环闭合：从 0008 的 FEngineLoop::Tick() 到本课的光子——整条管线是一个确定性的数据流，没有黑盒，每个延迟都有出处。
