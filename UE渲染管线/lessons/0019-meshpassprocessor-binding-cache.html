Pass 处理器与着色器绑定、绘制命令缓存

FMeshPassProcessor 是 FMeshBatch 到 FMeshDrawCommand 转换的执行者：它选定着色器排列、构建完整绑定、排序命令、支撑并行构建。这节课把这个"转换核心"拆开讲透。

时长 ~22 分钟 前置 0018（Mesh Drawing Pipeline 全貌）、0003（PSO 与描述符） 产出 能说出 FMeshPassProcessor 如何选着色器并生成 FMeshDrawCommand、绑定如何存储、命令如何排序与并行构建

0018 建立了整条流水线的骨架，这节课聚焦在"骨架的心脏"——FMeshPassProcessor::AddMeshBatch 内部。它要完成三件事：①根据材质和顶点工厂选定一套着色器（VS/PS 排列）；②把所有运行时参数打包进 FMeshDrawShaderBindings；③把最终的 FMeshDrawCommand 按排序键插入队列。并行构建机制让这三件事可以多线程同时进行。

1着色器选择：材质 × 顶点工厂 → 排列

UE 的着色器系统预编译了大量排列（Permutation）：同一段材质代码在不同顶点工厂（静态网格 vs 骨骼网格 vs 实例化）、不同 Pass（DepthPass vs BasePass）、不同特性开关（是否有方向光、是否开 GI…）下会编译成不同的 HLSL 二进制。FMeshPassProcessor 的 AddMeshBatch 实现在运行时根据当前上下文，从已编译的排列库里查找（而非编译）正确的 VS/PS 组合。

// FBasePassMeshProcessor::AddMeshBatch 内部（简化流程）
bool TryAddMeshBatch(const FMeshBatch& Batch, ...)
{
    // 1. 从 MaterialRenderProxy 取着色器排列键
    FMaterialShaderTypes ShaderTypes;
    ShaderTypes.AddShaderType<TBasePassVS<LightMapPolicyType>>();
    ShaderTypes.AddShaderType<TBasePassPS<LightMapPolicyType>>();

    // 2. 结合顶点工厂，取出编译好的着色器对象
    FMaterialShaders Shaders;
    if (!Material.TryGetShaders(ShaderTypes, Batch.VertexFactory->GetType(), Shaders))
        return false; // 排列不存在，跳过

    // 3. 构建 FMeshDrawCommand
    BuildMeshDrawCommands(Batch, Shaders, ...);
    return true;
}

着色器查找的逻辑骨架：排列键 = 材质 + 顶点工厂类型 + Pass 特性开关

2顶点工厂的作用：同材质适配不同几何

Vertex Factory 抽象了"顶点数据从哪来、怎么送给 VS"。对于同一个材质（同一段 PS 逻辑），静态网格、骨骼网格、实例化网格各自有不同的顶点输入布局，需要不同的 VS 排列——VS 里的骨骼蒙皮逻辑只出现在 FSkeletalMeshVertexFactory 的排列里。

FLocalVertexFactory静态网格：位置/法线/UV 直接读，无蒙皮
FSkeletalMeshVertexFactory骨骼网格：在 VS 里做骨骼权重蒙皮，影响位置/法线
FInstancedStaticMeshVertexFactory实例化网格：每实例读不同变换，一次 DrawInstanced

同一个材质 M 应用到三种几何上，FMeshPassProcessor 会查找三种 VS 排列（TBasePassVS<..., FLocalVertexFactory>、TBasePassVS<..., FSkeletalMeshVertexFactory>……），PS 可以共用同一排列。这正是"同材质适配不同几何来源"的实现机制。

3FMeshDrawShaderBindings：绑定即描述符表

着色器确定后，BuildMeshDrawCommands 把所有运行时参数收集进 FMeshDrawShaderBindings。它在逻辑上等价于 0003 讲的描述符堆/根签名——记录每个着色器阶段（VS/PS）需要绑定的资源句柄：

绑定类型典型内容D3D12 对应
Uniform Buffer（常量缓冲）变换矩阵、材质参数、View 参数CBV / Root Constant
SRV（Shader Resource View）贴图、结构化缓冲Descriptor Table SRV
Sampler纹理采样状态Descriptor Table Sampler
UAV（Unordered Access View）可写缓冲（较少见于网格 Pass）Descriptor Table UAV

所有这些句柄在 BuildMeshDrawCommands 里通过 ShaderBindings.Add(ShaderParam, Resource) 逐条填入，最终打包进 FMeshDrawCommand。提交时只需遍历这张表做绑定，不需要再次查找材质或调用任何 UObject。

为什么把绑定存进命令而不是每帧重新查找

资源句柄查找涉及字符串查找、哈希表等操作，每帧对每个网格重复执行代价不菲。一旦绑定存进 FMeshDrawCommand，提交时只是遍历一个连续数组——缓存友好，完全没有查找开销。这和 0003 讲的"预构建描述符表"思路完全一致。

4FMeshDrawCommandSortKey：排序与状态切换最小化

FMeshDrawCommand 附带一个 64 位排序键 FMeshDrawCommandSortKey，控制命令在 Pass 内的提交顺序。排序键的字段按优先级从高到低包含：

  • PSO 哈希（最高优先级）：相同 PSO 的命令聚在一起，避免频繁切换管线状态对象——这是 GPU 最昂贵的状态切换。
  • 着色器绑定哈希：相同着色器+纹理组合的命令连续提交，减少描述符表切换。
  • 深度/距离（BasePass 里次要，DepthPass 里由近到远以利用 Early-Z）。

收集完一个 Pass 的所有命令后，FMeshDrawCommandPassSetupTask 对命令列表排序，再走 SubmitMeshDrawCommands。排序这一步的 CPU 开销换来的是 GPU 端大量减少的状态切换——通常是值得的。

5并行构建：FParallelMeshDrawCommandPass

构建 FMeshDrawCommand 的工作（调用 AddMeshBatch、收集绑定）可以在多个工作线程上并行完成。FParallelMeshDrawCommandPass 把图元列表切分成多个块，每个工作线程处理自己的块，最终合并。

6PSO 收集：与 0003 的关联

在 BuildMeshDrawCommands 执行期间，每个唯一的 PSO 配置（着色器+混合+光栅状态）都会被登记进 PSO 缓存（PipelineStateCache）。如果缓存命中，直接取回已编译的 GPU Pipeline；如果未命中，则触发一次耗时的编译——也就是你在游戏加载或首次渲染新材质时可能看到的卡顿。这是 0003 讲的 PSO 缓存机制在 Mesh Draw Pipeline 里的落地点。

PSO 缺失卡顿的根源

PSO 编译是阻塞的，耗时可达几十到几百毫秒。UE 4.26 通过 PSO 预缓存（r.ShaderPipelineCache.Enabled）在加载期提前编译已知 PSO 来缓解。但材质的每一种排列 × 每一种顶点工厂 × 每一个 Pass，都是一个潜在的 PSO——组合爆炸是真实工程挑战。

检索练习

FMeshDrawCommandSortKey 的首要排序字段是什么，目的是什么？

说出 FMeshPassProcessor 的完整工作流：从接收 FMeshBatch 到产出 FMeshDrawCommand，涉及哪四个关键步骤？
① 着色器排列查找：根据材质 + 顶点工厂类型 + Pass 特性开关，从预编译排列库查找正确的 VS/PS 对。
② 绑定构建（BuildMeshDrawCommands）：把所有运行时参数（常量缓冲、纹理 SRV、采样器）填入 FMeshDrawShaderBindings。
③ PSO 收集：把当前着色器+混合+光栅状态组合登记进 PSO 缓存，未命中则触发编译。
④ 命令入队 + 排序键赋值：把完整的 FMeshDrawCommand 附上 FMeshDrawCommandSortKey 插入 Pass 命令列表，后续排序 + 提交。

带走这几条

  • FMeshPassProcessor::AddMeshBatch → BuildMeshDrawCommands 是着色器选择 + 绑定构建的执行位置；着色器排列键 = 材质 + 顶点工厂类型 + Pass 特性。
  • 顶点工厂让同一材质的 PS 适配不同几何来源（静态/骨骼/实例化），只需切换 VS 排列。
  • FMeshDrawShaderBindings 是"命令自带的描述符表"——所有参数打包进命令，提交时无需再查找。
  • FMeshDrawCommandSortKey 按 PSO 优先排序，减少管线状态切换；FParallelMeshDrawCommandPass 多线程并行构建命令。
  • PSO 在此被收集进缓存——PSO 缺失会触发阻塞编译，是真实工程里性能卡顿的常见原因。
