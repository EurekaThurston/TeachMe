Part 0 · 基础 · Lesson 0003

图形 API 的契约：命令列表、PSO、描述符、屏障

CPU 从不直接命令 GPU 干活——它录命令，GPU 异步地放命令。图形 API（这里以 D3D12 为心智模型）就是这份契约。搞懂它的四个核心概念，UE 的 RHI 和 RDG 就只是"替你把这份契约填好"。

时长 ~16 分钟 前置 0001 / 0002 产出 能说清命令列表/PSO/描述符/屏障各解决什么问题，以及 UE 在哪一层接管它们

现代显式 API（D3D12 / Vulkan）的设计哲学是一句话：把同步和状态管理的责任，从驱动手里拿走、交给应用。这听起来是负担，却是巨大性能的来源——因为只有应用（UE）知道整帧的全局信息，能做出驱动猜不到的优化。代价是有人得把这份"契约"填对。在 UE 里，填契约的人就是 RHI 和 RDG。本课先认识契约本身。

为什么用 D3D12 当心智模型

UE 4.26 同时支持 D3D11（隐式、驱动替你管同步）和 D3D12（显式、你自己管）。本课程一律用显式模型思考，因为它把"到底谁在同步什么"摊开在明面上——而这正是 RDG 被发明出来要自动化的东西。D3D11 只是把这些步骤藏进了驱动，并没有消失。

1命令列表与队列：录制 / 提交 / 执行

CPU 不会调一个函数就让 GPU 立刻画。它把一连串指令（设 PSO、绑资源、Draw、Dispatch……）录进一个命令列表（command list），录完提交到一条命令队列，GPU 在自己的时间线上异步消费。

CPU 录制SetPSO → SetDescriptors → Draw → Draw → Barrier → Draw …（写进命令列表，此刻 GPU 还没动）
GPU 执行从队列里取出已提交的命令列表，按序执行——通常落后 CPU 一帧（见 0004）

两个关键性质，后面整门课反复用到：

  • 异步：提交 ≠ 执行完成。CPU 提交后继续往下跑，靠围栏（0004）才知道 GPU 何时做完。
  • 可并行录制：显式 API 允许多个线程各录一个命令列表，最后一起提交。UE 正是用这点把网格绘制命令的录制并行化（见 0018/0036），吃满多核 CPU。

2PSO：把"一大堆状态"冻成一个对象

画一个三角形要确定的状态多得吓人：用哪个 VS/PS、顶点布局、混合模式、深度测试方式、光栅化设置、渲染目标格式…… 老式 API（D3D11）让你一项一项分别设，到 Draw 那一刻驱动才把这些状态组合、验证、编译成 GPU 能用的形态——这个运行时编译会造成卡顿（hitch）。

PSO（Pipeline State Object，管线状态对象）的解法：把上面几乎所有状态预先打包成一个不可变对象，提前编译好。绘制时只需一句"绑这个 PSO"，零运行时拼装。

// 概念示意：PSO 把这些一次性冻结
PSO = {
  VS, PS,                  // 着色器字节码
  InputLayout,             // 顶点格式
  BlendState, DepthState,  // 混合 / 深度测试
  RasterizerState,
  RenderTargetFormats      // 输出目标格式
};
cmdList.SetPipelineState(PSO);   // 绘制时一句话搞定

PSO 把运行时的状态拼装与编译，前移到加载期

UE 的痛点 · PSO 缓存与卡顿

每一种"着色器 + 状态"的组合都是一个 PSO，必须提前编译。UE 的材质 shader permutation 爆炸就源于此：一个材质在不同光照/顶点工厂/Pass 下会生成大量变体，每个变体对应若干 PSO。若某 PSO 在玩家眼前第一次用到时才编译，就是一次卡顿。UE 因此有 PSO 预缓存（PSO caching）机制——这正是"shader 编译/PSO"成为 UE 性能话题的根因，0037 会深入。

3描述符与根签名：着色器怎么找到资源

着色器里写 Texture2D BaseColor;，可 GPU 怎么知道这张纹理在显存哪儿、什么格式、怎么采样？答案是描述符（descriptor）——一小块描述"某资源在哪、如何解释"的元数据。绑定资源，本质是把描述符摆到着色器能查到的位置。

资源显存里的纹理/缓冲
→
描述符"在哪+怎么读"
→
描述符堆一大张表
→
根签名着色器可见的槽位布局
→
着色器按槽位读到资源

根签名（root signature）定义了"这套着色器需要哪些资源槽位"——相当于函数签名。描述符堆是一大张连续的描述符表，绑定时把堆里的范围接到根签名的槽位上。UE 的 RHI 用 FRHIShaderResourceView、FRHIUniformBuffer 等抽象封装这一层；RDG 的 SHADER_PARAMETER_* 宏则让你声明式地填这些槽位（0015 会写到）。

4资源屏障：显式 API 里最容易错、也最关键的一环

同一块显存，这一刻是 Base Pass 的渲染目标（GPU 在写），下一刻要当作光照 Pass 的着色器输入（GPU 来读）。在显式 API 里，这个用途切换必须由你显式声明，叫资源屏障（resource barrier / transition）。

// 把 GBuffer 从"可写的渲染目标"切到"可读的着色器资源"
cmdList.ResourceBarrier(GBufferA,
    RENDER_TARGET -> PIXEL_SHADER_RESOURCE);

没有这条屏障，光照 Pass 可能读到尚未写完的数据

屏障干两件事：①同步——确保写操作真正完成、缓存刷新，读才能看到正确数据；②布局转换——某些硬件对"渲染目标"和"可采样纹理"用不同的内存排布，切换时要转换。屏障下错或漏下，会导致读到脏数据或性能悬崖（过度屏障让流水线频繁停顿）。

关键洞见 · 这就是 RDG 存在的全部理由

手动在几百个 Pass 之间下对每一条屏障，既易错又难优化——而且只有"知道整帧长什么样"才能下到最优（提前转换、合并屏障、避免停顿）。RDG 的做法：让你只声明每个 Pass 读写哪些资源，它从读写关系自动推导出全部屏障，并放在全局最优的位置。本课的"屏障"，就是 0016 里 RDG 替你自动化的那个东西。理解了手动版的痛，才懂 RDG 的价值。

5把四件事串起来：一次绘制的 API 视角

// 渲染线程录制命令列表（概念伪码）
cmd.ResourceBarrier(GBuffer, COMMON -> RENDER_TARGET); // ④ 屏障：准备写
cmd.SetRenderTargets(GBufferA, GBufferB, GBufferC);
cmd.SetPipelineState(BasePassPSO);                       // ② PSO：一次绑好所有状态
cmd.SetDescriptorHeaps(sceneTextures, samplers);           // ③ 描述符：资源接入槽位
for (draw : meshDrawCommands)                              // ① 命令列表：批量录 Draw
    cmd.DrawIndexed(draw.indexCount, draw.instanceCount);
cmd.ResourceBarrier(GBuffer, RENDER_TARGET -> SRV);     // ④ 屏障：转给光照读

UE 的 BasePass 在 RHI 层最终就坍缩成这种序列；RDG/RHI 替你生成它

检索练习

相比逐项设置渲染状态的旧式 API，PSO 把状态打包成不可变对象，最主要解决了什么问题？

为什么显式 API 把资源屏障的责任交给应用，反而能比驱动自动管理更快？这给 UE 带来了什么任务？
因为最优屏障决策需要全局信息——何时转换、能否提前转换以掩盖延迟、能否合并多条屏障——只有"知道整帧所有 Pass 如何读写资源"的应用才掌握这些信息，驱动只能保守地逐次猜测。把责任交给应用，就能下出驱动猜不到的更优同步。代价是 UE 必须有人填这份契约且不能填错：手动易错且难优化，于是 UE 用 RDG 从"每个 Pass 声明读写了哪些资源"自动推导并最优放置全部屏障（0016）。

带走这几条

  • 命令列表 + 队列：CPU 录制、GPU 异步执行；显式 API 允许多线程并行录制——UE 借此并行化绘制提交。
  • PSO：把渲染状态冻成不可变对象、提前编译，消除运行时拼装卡顿——UE shader permutation/PSO 缓存的根源。
  • 描述符 / 根签名 / 堆：着色器通过描述符找到资源；根签名是资源槽位的"函数签名"。
  • 资源屏障：显式声明资源用途切换，负责同步与布局转换——错/漏即脏数据或性能悬崖。
  • UE 用 RHI 抽象这份契约、用 RDG 自动生成屏障与调度——后面 Part 3 / Part 7 就是看它们怎么填。
