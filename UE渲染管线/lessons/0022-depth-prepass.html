Depth Prepass（Early-Z）

在 BasePass 运行任何像素着色器之前，UE 先跑一遍只写深度的 Prepass，把深度缓冲填满。代价：一次额外的几何遍历。收益：后续所有 Pass 的 Early-Z 几乎消灭 overdraw。

时长 ~14 分钟 前置 0001（Early-Z / overdraw / OM 阶段）、0014（FDeferredShadingSceneRenderer） 产出 能解释 Prepass 如何用 Early-Z 消除 overdraw，并知道 r.EarlyZPass 各档含义

0001 讲过 Early-Z：只要像素着色器不修改深度且不丢弃片元，GPU 可在 PS 之前做深度测试，把被遮挡片元直接丢掉。问题在于：第一个不透明 Pass 到来时深度缓冲是空的，Early-Z 无从比较。Depth Prepass 解决这个引导问题——先用极简着色器只写深度，建立完整深度缓冲，之后 BasePass 开启 Early-Z 就能对每个片元精确判断"它真的可见吗"，把不可见片元的昂贵材质着色器调用彻底跳过。

1代码入口：RenderPrePass

FDeferredShadingSceneRenderer::RenderPrePass（DepthRendering.cpp）是 Prepass 的调度函数，在 Render() 序列里位于 BasePass 之前。它向 RDG 注册一个写入 SceneDepth 的 Pass，然后遍历所有不透明图元，经由 FDepthPassMeshProcessor 生成 FMeshDrawCommand。

// DepthRendering.cpp（架构示意）
void FDeferredShadingSceneRenderer::RenderPrePass(
    FRDGBuilder& GraphBuilder,
    FRDGTextureRef SceneDepthTexture)
{
    // 为每个 View 录制 DepthOnly draw calls
    AddRenderTargetSetupPass(GraphBuilder, SceneDepthTexture);
    DrawPrePass(GraphBuilder, ...); // → FDepthPassMeshProcessor
}

FDepthPassMeshProcessor 给每个图元选择 DepthOnly 变体着色器——顶点着色器做最小变换（只输出 SV_Position），像素着色器为空或极简（opaque 物体甚至可以不挂 PS）。这是整条管线里最"轻"的绘制调用。

DepthOnly 着色器在哪里

Engine/Shaders/Private/DepthOnlyPixelShader.usf 是 Depth Prepass 的 PS；对于完全不透明、无裁剪需求的物体，PS 可直接为空（硬件不运行 PS，只写深度）。顶点侧使用通用 BasePassVertexShader.usf 的 DepthOnly 变体，经 Vertex Factory 读取几何数据——这样骨骼网格、静态网格都能走同一套 MeshProcessor，只换着色器变体。

2Early-Z 收益示意

Prepass只写 SceneDepth
无材质开销
›
SceneDepth完整深度缓冲
建立 Early-Z
›
BasePass片元 Early-Z 测试
被遮挡 → 丢弃
›
可见片元运行材质 PS
写 GBuffer

收益的量级取决于场景的 overdraw 程度。在物体密集、后向深度排序差的场景里，overdraw 可以轻松超过 4–8×——即同一像素被画 4–8 次，Prepass 把这 4–8 次材质 PS 调用降为 1 次。代价是所有不透明几何额外跑一遍顶点着色器，通常比材质 PS 便宜 1–2 个数量级。

3r.EarlyZPass 的各档含义

cvar r.EarlyZPass 控制 Prepass 覆盖哪些几何（RendererSettings.cpp）：

值含义适用场景
0关闭 Prepass，不运行 Early-Z（仅前向路径或极简场景调试用）几乎不用
1仅对设置了"遮挡体（occluder）"标记的不透明物体运行 Prepassoverdraw 中等、想节省 Prepass 本身开销时
2对所有不透明物体运行 Prepass（UE 4.26 默认）通用 PC 延迟着色路径
3对所有不透明物体运行 Prepass，并在 Prepass 之后紧接着做遮挡查询高度 occluder-优化的场景

关联 cvar：r.EarlyZPassMovable（是否把可移动物体也纳入 Prepass）。可移动物体的 Prepass 会增加每帧更新代价，但在城市场景等大量动态物体高度重叠时值得。

何时全 Prepass 划算

判断标准：当场景是 GPU-bound 且 PS 密集（复杂材质 + 高 overdraw）时，Prepass 总是划算的——额外的顶点开销远小于省掉的 PS 调用。当场景是 VS-bound 或几何极简时（如粒子系统主导的屏幕，几乎无不透明遮挡），Prepass 反而浪费了一次顶点遍历。用 r.EarlyZPass=1 搭配手动标记 occluder 是一种折中。

4Prepass 对 Occlusion Culling 的额外作用

Prepass 填充 SceneDepth 后，UE 随即用它构建 HZB（Hierarchical Z Buffer，层级深度缓冲）。HZB 在后续帧（和当前帧的遮挡查询路径）被用来进行 GPU 端遮挡剔除——判断一个包围盒是否完全被当前深度缓冲遮挡。SceneDepth 因此是 Prepass 的双重产出：Early-Z（同帧）+ HZB（下帧剔除）。

检索练习

r.EarlyZPass=2 时，Depth Prepass 的像素着色器对完全不透明物体做什么？

不看文章，解释：若 r.EarlyZPass=0（关闭 Prepass），BasePass 的 overdraw 会怎样？在什么硬件/场景条件下，关闭 Prepass 反而性能更好？
若关闭 Prepass，BasePass 开始时深度缓冲为空，Early-Z 无法在 PS 之前剔除被遮挡片元（只能在 PS 之后做深度测试），每个像素可能被多个重叠三角形的材质 PS 都运行一遍——overdraw 全部变成实际着色开销，GPU 时间与 overdraw 倍数成正比飙升。反而性能更好的条件：场景几乎无不透明重叠（outdoor 空旷场景、粒子主导）、几何极为简单（顶点 transform 也很贵时没必要跑两遍）、或当前是 VS-bound 而非 PS-bound。这时 Prepass 的顶点遍历开销大于其省掉的 PS 调用，关掉更合算。

带走这几条

  • Depth Prepass 用 FDepthPassMeshProcessor + DepthOnly 着色器只写 SceneDepth，PS 对不透明物体可为空——这是管线里开销最低的绘制 Pass。
  • Prepass 的核心收益：建立完整深度缓冲，使 BasePass 能全程利用 Early-Z 丢弃被遮挡片元，消除 overdraw。
  • r.EarlyZPass：0=关，1=仅遮挡体，2=全不透明（默认），3=全不透明+遮挡查询；配套 r.EarlyZPassMovable。
  • Prepass 还填充 HZB，为下帧 GPU 端遮挡剔除提供深度数据。
  • 全 Prepass 在 GPU-bound / 高 overdraw / PS 密集场景划算；VS-bound 或几何极简场景可酌情降档。
