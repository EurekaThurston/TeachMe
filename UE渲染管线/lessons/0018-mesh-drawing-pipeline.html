Mesh Drawing Pipeline：FMeshBatch → FMeshDrawCommand

UE 4.22 之后，"绘制一个网格"从即时发射命令改成了 retained-mode：代理提交 FMeshBatch 原料，Pass 处理器把它转成可缓存的 FMeshDrawCommand，最后批量提交到 RHI。这条流水线的设计动机是激进缓存与减少 CPU 录制成本。

时长 ~20 分钟 前置 0009（场景代理）、0014（渲染器总调度） 产出 能描述 FMeshBatch 经 FMeshPassProcessor 转成 FMeshDrawCommand、再 SubmitMeshDrawCommands 到 RHI 的全过程

在 4.22 之前，UE 用"Drawing Policy"模式：每帧逐个物体调用 SetShader、DrawMesh，CPU 录制开销随物体数量线性增长（呼应 0004 CPU-bound 讨论）。新的 Mesh Draw Pipeline 把"准备绑定"和"实际提交"分离：静态物体的绘制命令在 AddToScene 时就缓存好，每帧只需重用；动态物体才在每帧调用 GetDynamicMeshElements。结果是：一个拥有上千静态网格的场景，CPU 录制开销可以接近常数。

1三个核心数据结构

整条 pipeline 围绕三个结构体展开，弄清楚它们各自代表什么、在哪一步产生，就掌握了主干。

结构体产生时机内容可缓存？
FMeshBatch代理每帧提交（或 AddToScene）几何来源 + 材质 + LOD 等原料，不绑定 Pass否（描述几何，Pass 无关）

FRHICommandList 调用SubmitMeshDrawCommands具体的 DrawIndexedPrimitive 等 RHI 调用否（每帧重新提交）

3FMeshBatch：几何与材质的原料单

FMeshBatch 由 FPrimitiveSceneProxy 提交。它描述"这个图元是什么"，包含：

  • FMeshBatchElement：顶点缓冲、索引缓冲、实例数、PrimitiveUniformBuffer（变换矩阵所在的常量缓冲）。
  • MaterialRenderProxy：材质实例的渲染代理，指向编译好的着色器资源（但还没选 Pass 变体）。
  • VertexFactory：顶点工厂类型（FLocalVertexFactory、FSkeletalMeshVertexFactory…），决定 VS 怎么读顶点数据。
  • LOD 索引、投影阴影标志等元数据。

静态网格的 FMeshBatch 在 AddToScene 时收集（FStaticMeshBatch），不需要每帧重新产生。动态物体（骨骼网格、粒子、实例化组件）每帧调用 GetDynamicMeshElements 产生新的 FMeshBatch。

4EMeshPass：Pass 枚举

UE 4.26 用 EMeshPass::Type 枚举所有网格 Pass，每个 Pass 有自己的 FMeshPassProcessor：

// MeshPassProcessor.h（简化）
namespace EMeshPass
{
    enum Type
    {
        DepthPass,          // Depth Prepass
        BasePass,           // 写 GBuffer
        ShadowDepth,        // 阴影深度
        Translucency,       // 半透明
        Velocity,           // 速度缓冲
        // ... 共约 20 个
        Num
    };
}

EMeshPass 枚举——每个值对应一个 FMeshPassProcessor 子类

每个 Pass 都有一个对应的 FMeshPassProcessor 子类，例如 FDepthPassMeshProcessor、FBasePassMeshProcessor。这些处理器的 AddMeshBatch 负责把 FMeshBatch 转成对应 Pass 的 FMeshDrawCommand。

5FMeshDrawCommand：无状态的完整绘制描述

FMeshDrawCommand 是这条 pipeline 的核心产物，它是无状态的：不依赖任何全局渲染状态，完整描述了一次 draw call 所需的全部信息：

  • 着色器绑定（FMeshDrawShaderBindings）：每个着色器阶段（VS/PS）用到的所有参数——贴图、常量缓冲、SRV 等的句柄（对应 0003 的描述符/根签名）。
  • PSO 引用：对应的 FGraphicsPipelineStateInitializer（着色器+混合+光栅状态）。
  • 顶点流绑定：顶点缓冲、索引缓冲的偏移与步长。
  • 绘制参数：DrawIndexedPrimitive 需要的顶点数、实例数等。
为什么"无状态"是关键设计目标

传统即时模式下，着色器绑定散落在各处调用链中；一旦全局状态被意外修改，绘制结果就不可预测。FMeshDrawCommand 把所有绑定自包含，任何线程都可以安全地读取并提交——这是并行录制（FParallelMeshDrawCommandPass，见 0019）和跨帧缓存两个特性的共同前提。

6Retained Mode：静态缓存的工作方式

静态网格的 FMeshDrawCommand 在 AddToScene 阶段调用 CacheMeshDrawCommands 生成并缓存，存进 FCachedPassMeshDrawList（每个 Pass 一张列表）。每帧 InitViews 结束后，可见性位图告诉系统哪些静态网格可见，系统直接从缓存里取出命令，不需要重新调用 AddMeshBatch。

动态物体走另一路径：每帧 GetDynamicMeshElements → 实时经 FMeshPassProcessor::AddMeshBatch 转换 → 生成临时 FMeshDrawCommand → 提交。

7SubmitMeshDrawCommands：命令到 RHI

所有命令收集完毕后，SubmitMeshDrawCommands 遍历 FMeshDrawCommand 列表，将每条命令展开成一系列 FRHICommandList 调用：设置 PSO、绑定描述符、设置顶点流、最终调用 DrawIndexedPrimitive。这一步在渲染线程（或并行工作线程）上完成，RHI 线程再将其翻译为 D3D12 命令列表提交 GPU。

检索练习

静态网格的 FMeshDrawCommand 在哪个时机生成，并存放在哪里？

不看图，说出 FMeshBatch → FMeshDrawCommand → RHI 这条流水线的三个阶段：各阶段的输入/输出是什么，静态和动态物体在哪里分叉？
阶段一：FPrimitiveSceneProxy 提交 FMeshBatch——静态网格在 AddToScene 时提交（一次性），动态物体每帧调用 GetDynamicMeshElements 提交。
阶段二：每个 Pass 的 FMeshPassProcessor::AddMeshBatch 把 FMeshBatch 转成 FMeshDrawCommand（含 PSO、着色器绑定、顶点流、绘制参数）——静态网格的命令缓存进 FCachedPassMeshDrawList，动态物体的命令是临时的。
阶段三：SubmitMeshDrawCommands 把 FMeshDrawCommand 列表展开成 FRHICommandList 调用，RHI 线程翻译为 D3D12 命令提交 GPU。

带走这几条

  • 三段式流水：FMeshBatch（原料）→ FMeshPassProcessor（转换）→ FMeshDrawCommand（无状态绘制描述）→ SubmitMeshDrawCommands（RHI）。
  • FMeshDrawCommand 无状态、自包含，可缓存、可合批——这是新管线相比旧 Drawing Policy 的核心优势。
  • 静态网格走 CacheMeshDrawCommands 路径，AddToScene 时一次性缓存，每帧直接取用；动态物体每帧调用 GetDynamicMeshElements 实时生成。
  • EMeshPass 枚举定义了所有网格 Pass，每个 Pass 有独立的 FMeshPassProcessor 子类负责着色器选择和转换。
  • 设计动机根植于 0003/0004：降低 CPU 录制成本、避免 CPU-bound——场景越静态，缓存收益越大。
