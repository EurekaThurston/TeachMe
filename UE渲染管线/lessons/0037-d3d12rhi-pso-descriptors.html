D3D12 RHI 后端：PSO 缓存与描述符堆

RHI 线程调用的 GDynamicRHI 究竟在做什么？本课拆开 D3D12RHI 后端：FD3D12CommandContext 如何映射到 D3D12 命令列表、PSO 如何被缓存与重用、描述符堆（在线/离线）如何管理、资源屏障如何追踪——把 0003 讲的图形 API 抽象契约，在 UE 源码层面完整对应一遍。

时长 ~28 分钟 前置 0003（图形 API 契约）、0036（RHI 线程与命令列表） 产出 能说清 FD3D12CommandContext 的职责、PSO 缓存命中路径、描述符堆在线/离线分工、资源状态追踪机制，以及上传堆的常量缓冲分配策略

在 0003 你学到了"PSO、命令列表、描述符、屏障"是 D3D12 对应用程序的四个核心契约。在 0036 你看到 RHI 线程把 FRHICommand* 链表翻译成 GDynamicRHI 调用。本课就是那个翻译目的地内部——FD3D12DynamicRHI 的具体实现，以及它管理的缓存、堆、状态机。

1FD3D12CommandContext：一次绘制调用的承载者

FD3D12CommandContext（Engine/Source/Runtime/D3D12RHI/Private/D3D12CommandContext.h）是 D3D12RHI 的核心执行上下文，对应 D3D12 层的一个 ID3D12GraphicsCommandList。它的生命周期与一段 GPU 工作批次对齐：

FD3D12CommandContext 内部持有：

  • FD3D12CommandListHandle——封装 ID3D12GraphicsCommandList 及其对应的 ID3D12CommandAllocator。
  • 当前绑定的 PSO 句柄（用于检测 PSO 是否需要切换）。
  • 当前描述符堆绑定状态（在线堆的当前偏移）。
  • 资源状态缓存（FD3D12ResourceStateCache，见第 4 节）。
命令分配器池——为什么不能立即复用

D3D12 的 ID3D12CommandAllocator 在 GPU 执行完对应命令列表之前不能 Reset。FD3D12CommandAllocatorManager 维护一个池，按 GPU 完成的 Fence 值区分"可复用"和"尚在飞行中"的分配器，避免 CPU 重置一个 GPU 还在读的分配器而触发未定义行为。

2PSO 缓存：FD3D12PipelineStateCache

管线状态对象（PSO）的创建在 D3D12 里极其昂贵——驱动要编译着色器字节码到机器码，可能耗时数百毫秒。UE 4.26 的 FD3D12PipelineStateCache（D3D12RHI/Private/D3D12PipelineState.h）用多级缓存把这个代价摊薄：

缓存层级存储位置键命中时
运行时内存缓存 TMap<FD3D12GraphicsPipelineStateDesc, FD3D12PipelineState*> 完整 PSO 描述结构体（着色器哈希 + 渲染状态） 直接返回已有 ID3D12PipelineState*，零开销。
磁盘持久化缓存 项目 Saved 目录下的 .pso 缓存文件 同上（跨会话） 从文件加载 PSO blob，驱动用预编译 blob 跳过大部分编译工作。
驱动缓存 GPU 驱动内部（不透明） 着色器字节码 驱动命中缓存可进一步加速。

每次渲染线程调用 RHICreateGraphicsPipelineState() 时，UE 先查运行时内存缓存；未命中则尝试磁盘缓存；再未命中才调用 ID3D12Device::CreateGraphicsPipelineState。第一帧或全新材质组合时缓存冷启动，这正是 UE 的 PSO 预编译（ShaderCompileWorker 输出阶段）要提前做的事。

关键洞见 · PSO 抖动是卡帧元凶之一

若运行时遭遇大量缓存未命中（动态材质组合、首次加载关卡），驱动创建 PSO 会让 RHI 线程阻塞数十到数百毫秒，表现为明显的卡顿尖刺。解决路径：① UE 的 PSO 预编译 pipeline（将常见 PSO 在后台线程提前创建）；② 使用 r.ShaderPipelineCache.Enabled 1 持久化运行时缓存；③ 减少运行时材质排列组合数量。

3描述符堆：在线堆与离线堆

D3D12 通过描述符堆（Descriptor Heap）把资源绑定给着色器。UE 的 D3D12RHI 把描述符堆分成两类：

在线堆
(Online)绑定到 GPU 命令列表（SetDescriptorHeaps）；着色器从这里读取描述符。环形缓冲设计：每帧分配新偏移，GPU 用完后偏移归零复用。容量有限，必须节约。
离线堆
(Offline)CPU 侧的大型描述符存储池；描述符在创建资源（CreateTexture2D 等）时写入。绑定时从离线堆复制到在线堆（CopyDescriptors）。容量大，但 GPU 不能直接访问。

这个两级设计的动机：D3D12 规定 GPU 只能访问绑定到命令列表的堆，而绑定数量有限（通常一帧只 SetDescriptorHeaps 一次，否则有性能风险）。解法是让在线堆作为"暂存区"，每次 Draw 之前把需要的描述符从离线堆批量复制过来。

// D3D12DescriptorCache.cpp — 概念简化
void FD3D12DescriptorCache::SetSRVs(
    EShaderFrequency ShaderStage,
    FD3D12ShaderResourceViewCache& Cache)
{
    // 1. 在在线堆上分配 N 个连续槽位
    uint32 HeapOffset = OnlineHeap.AllocateSlots(Cache.NumSRVs);

    // 2. 批量从离线堆复制到在线堆
    Device->CopyDescriptors(
        OnlineHandle(HeapOffset), Cache.SRVHandles, Cache.NumSRVs);

    // 3. 设置根描述符表指向在线堆的该偏移
    CommandList->SetGraphicsRootDescriptorTable(
        SlotIndex, OnlineHandle(HeapOffset));
}

4资源状态追踪与屏障

D3D12 要求开发者手动管理每个资源的资源屏障（Resource Barrier / Transition）。UE D3D12RHI 用 FD3D12ResourceStateCache（或 FD3D12Resource 内部的状态字段）追踪每个子资源（mip level × array slice）的当前状态，在需要时插入 D3D12_RESOURCE_BARRIER_TYPE_TRANSITION。

状态追踪的两种层次：

  • 本地（per-CommandList）追踪：FD3D12CommandContext 记录在当前命令列表中已应用的状态转换，避免重复插屏障。
  • 全局（per-Resource）追踪：FD3D12Resource 存储资源当前在 GPU 时间线上的"已知状态"，在 Execute 后更新，供下一批命令列表参考。
状态追踪不是 RDG——两者互补

RDG（0015/0016）在 Pass 编译阶段就推导出整帧的屏障需求，在 ExecutePass 之间插入屏障。这已覆盖大多数现代路径。但仍有一些"手写 RHI"的老代码绕过 RDG 直接操作 FRHICommandListImmediate，那里靠 D3D12RHI 层的状态缓存作为兜底。两者并存，各有分工：RDG 是高层编排者，D3D12RHI 是低层保证者。

5根签名：着色器如何找到资源

D3D12 的根签名（Root Signature）定义着色器能访问哪些描述符表、根常量、根描述符，以及它们在"着色器可见资源绑定空间"里的槽位布局。UE 4.26 使用一个固定的全局根签名策略——大多数管线状态共享同一个根签名（或少数几个变体），避免频繁切换根签名带来的 GPU 刷新开销。

根签名在 FD3D12RootSignature（D3D12RHI/Private/D3D12RootSignature.h）中表达，描述了：

  • 每个着色器阶段（VS/PS/CS）的 SRV/UAV/CBV/Sampler 描述符表范围。
  • 根常量（32 位值，直接嵌在根签名里，无需描述符）。
  • 根 CBV（直接用 GPU 虚拟地址绑定，无需通过描述符堆）。

6上传堆与常量缓冲分配

CPU 需要每帧向 GPU 传递大量常量数据（变换矩阵、光源参数、材质参数……）。D3D12 的上传堆（Upload Heap，D3D12_HEAP_TYPE_UPLOAD）在 CPU/GPU 都可见，但不是 GPU 最快的读取位置——它是 CPU 写、GPU 读的暂存区。UE D3D12RHI 的常量缓冲策略：

常量大小策略UE 实现
小常量（≤64B） 根常量（Root Constants），直接嵌在根签名里，无内存分配。 SetGraphicsRootConstants
中等常量（64B–几KB） 上传堆的线性分配器（每帧一个大块，子分配）；帧末归零。 FD3D12FastConstantAllocator（概念名，实际实现见 D3D12RHI 相关分配器代码）
大/持久常量缓冲（材质参数等） 独立 ID3D12Resource，生命周期与材质实例绑定；用 Map/Unmap 更新。 FD3D12ConstantBuffer
为什么上传堆不用于渲染目标

上传堆的内存映射为 CPU/GPU 共享（Write-Combined），GPU 读取它时经过 PCIe/系统互联，带宽远低于 GPU 本地显存（Default Heap）。渲染目标、GBuffer、深度缓冲都必须放在 Default Heap。常量缓冲因为数据量相对小、且每帧必须由 CPU 更新，才用上传堆。

D3D12RHI 中"在线描述符堆"（Online Descriptor Heap）的作用是什么？

PSO 缓存未命中时 UE D3D12RHI 会发生什么？为什么首帧卡顿与此有关？

缓存未命中时，FD3D12PipelineStateCache 先检查磁盘缓存（.pso 文件）；若也未命中，则同步调用 ID3D12Device::CreateGraphicsPipelineState。这个调用会让 GPU 驱动对着色器字节码做最终优化编译（JIT），可能耗时数十至数百毫秒，期间 RHI 线程被阻塞，整个渲染管线停顿，表现为帧率尖刺。首帧或首次进入新关卡遭遇大量新 PSO，正是这类卡顿的根源。解决方法是 UE 的 PSO 预编译机制（ShaderPipelineCache），在后台线程提前构建并缓存常用 PSO。

根签名（Root Signature）在 UE D3D12RHI 中是如何使用的？为什么 UE 倾向于使用少量固定根签名？

根签名定义了着色器可见的绑定布局（描述符表、根常量、根 CBV）。切换根签名会触发 GPU 内部管线刷新，代价类似切换 PSO。UE 4.26 的 D3D12RHI 为大多数图形管线使用统一的全局根签名（或少数变体），把所有着色器阶段的 SRV/UAV/Sampler/CBV 都纳入这个固定布局。这样每帧几乎不需要切换根签名，只需在 Draw 前更新根描述符表的偏移（指向在线堆的新位置）即可。代价是根签名占用较多根参数槽位，但比频繁切换的性能损失小得多。

带走这几条

  • FD3D12CommandContext 持有一个 ID3D12GraphicsCommandList 和命令分配器句柄；它是 RHI 线程翻译 FRHICommand* 的落地点。
  • FD3D12PipelineStateCache 提供三级 PSO 缓存（内存 → 磁盘 → 驱动）；缓存未命中会阻塞 RHI 线程数十至数百毫秒，是首帧卡顿的主要原因。
  • 描述符堆分在线（GPU 可见，环形缓冲）和离线（CPU 管理，大容量）两层；Draw 前把需要的描述符从离线复制到在线，再通过根描述符表绑定。
  • 资源状态追踪（FD3D12Resource 内部状态字段 + 本地命令列表缓存）确保每次状态转换都有正确的 ResourceBarrier，与 RDG 上层屏障互补。
  • 常量缓冲按大小分策略：小量用根常量（零分配），中量用上传堆线性分配器，大/持久量用专用 FD3D12ConstantBuffer。
