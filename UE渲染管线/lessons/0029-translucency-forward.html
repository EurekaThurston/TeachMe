半透明渲染：Translucency 与前向着色

半透明物体不能写 GBuffer，改用前向着色；它在不透明渲染完成后、从后到前排序绘制，光照来自预积分体积与光照网格。

时长 ~22 分钟 前置 0023（Base Pass / GBuffer）、0024（Light Grid 光照网格）、0001（OM 混合） 产出 能解释半透明为何用前向着色、排序策略、取光机制与 Separate Translucency

延迟着色的基本假设是"不透明几何先写 GBuffer，再统一光照"，但半透明物体（玻璃、水、粒子、毛发）不能写 GBuffer——它们需要把背景透过来，若写 GBuffer 会覆盖背后物体的材质属性，破坏延迟光照。因此 UE 4.26 在不透明光照完成后，专门开一个前向着色 Pass 处理半透明：材质着色器直接在 PS 里采样灯光数据，一次 draw call 完成着色 + 混合。

2为什么半透明不能用延迟着色

延迟着色的光照 Pass 读取 GBuffer（不透明几何在 Base Pass 写入的）来计算光照。若半透明物体也写 GBuffer，它会覆盖背后不透明物体的 GBuffer 数据，后续光照算的是半透明物体的材质，背景的颜色彻底丢失。

此外，混合（Blend）（呼应 0001 的 OM 阶段）要求：新写入的颜色 = src·α + dst·(1−α)，其中 dst 是已有的 SceneColor 背景色。GBuffer 里没有"最终颜色"，只有材质属性，所以混合根本无法在延迟阶段进行——必须在最终颜色已存在时（SceneColor 已渲染完不透明物体后）才能执行。

3从后到前排序

半透明 Pass 必须从后到前（Back-to-Front, painter's algorithm）排序绘制：先画最远的，再把近处的混合覆盖上去，保证前景半透明正确地"盖在"背景上。若顺序颠倒，背景的半透明会把前景混合掉，结果错误。

UE 4.26 按图元（Primitive）包围盒中心的相机深度排序，这是对每个可绘制对象粒度的排序，不是每个三角形——对于有内部交叉的复杂网格，可能仍有排序错误（这是前向半透明的已知局限）。

非顺序无关（NOT Order-Independent）

UE 4.26 的半透明不是 OIT（Order-Independent Transparency）。它严格依赖从后到前的绘制顺序。粒子系统通常每粒子排序，网格则按中心点。对于自相交或互相穿插的半透明网格，排序错误是不可避免的视觉 artifact。UE 有 TranslucentSortPolicy 可以调整排序轴，但根本上不能解决 OIT 问题。

4取光：半透明光照体积 + 光照网格

前向着色的半透明材质在 PS 里需要知道"我受哪些光影响、各多少能量"，但延迟光照的结果已经合并进了 SceneColor，无法再分离出每盏灯的贡献。UE 4.26 用两个机制为半透明提供光照数据：

4.1 半透明光照体积（Translucency Lighting Volume）

在不透明光照阶段，UE 把整个场景的光照预积分进一个3D 纹理（Froxel 体积）：相机视锥被切成一个个小 froxel（视锥空间体素），每个 froxel 存储该体积内的光照 SH 系数（方向性辐照度，类似光照探针）。半透明 PS 按世界坐标在体积中插值采样，得到所在位置的近似光照。

4.2 光照网格（Light Grid，呼应 0024）

对于精度要求更高的半透明物体（如大型半透明网格），也可以从 0024 的 Light Grid 中取回影响该 froxel 的动态灯光列表，在 PS 里逐灯逐像素计算，方法与不透明的前向着色相同。控制着色器里通过 GetForwardLighting() 访问。

光照体积 vs 光照网格：精度与成本的权衡

光照体积（体素化 SH）更快，适合粒子、烟雾等大量半透明几何——每个 froxel 只查一次 3D 纹理。光照网格精确，但每像素要迭代灯光列表。材质里通过 Translucency Lighting Mode 属性（VolumetricNonDirectional / VolumetricDirectional / Surface Forward Shading 等）在两种路径间选择。

5读 SceneColor / SceneDepth 做折射

半透明材质常见需求：扭曲/折射背景（如玻璃、水面）。材质在 PS 里可以读 SceneColor 缓冲（背景不透明物体的最终颜色）和 SceneDepth（背景深度，用于软粒子深度融合）。

由于半透明 Pass 正在写 SceneColor，读写同一个 RT 会有问题——UE 在开始半透明 Pass 前先把 SceneColor 复制一份（SceneColorCopy），半透明 PS 读 copy，写原始 SceneColor。折射扭曲通过 Refraction 材质输入实现，着色器内部偏移 UV 从 copy 采样。

6Separate Translucency：独立缓冲

UE 4.26 的独立半透明（Separate Translucency）机制：标记 bUseSeparateTranslucency = true 的半透明物体（默认大多数 PP 相关半透明）渲染到一张独立缓冲，不直接写入 SceneColor——而是在后处理链完成后（DOF/Bloom 之后）才合入最终图像。

为什么需要这个？深度景深（DOF）和 Bloom 处理 SceneColor 时，若半透明物体已在 SceneColor 里，它们会和背景一起被景深模糊，造成半透明边缘双重模糊 artifact。Separate Translucency 把半透明内容"隔离"，等 DOF 完成后再合入，避免被 DOF 误伤。

检索练习

半透明物体为什么必须从后向前排序绘制，而不能任意顺序？

说出半透明渲染在帧内的完整路径：为什么用前向着色、两种取光机制各自的原理、Separate Translucency 解决的问题。

为什么前向着色：半透明不能写 GBuffer（会覆盖背景材质属性），且混合需要已有的 SceneColor（只有不透明光照后才有），因此用前向着色——PS 直接采样灯光数据、算光照、与 SceneColor 混合一步完成。

取光机制：

  • 半透明光照体积：Froxel 3D 纹理，预积分场景光照为 SH，PS 按世界坐标插值采样，开销低，适合粒子/烟雾。
  • 光照网格（0024）：按 Froxel 索引取动态灯光列表，PS 逐灯精确计算，精度高但开销随灯光数增加。

Separate Translucency：部分半透明物体渲染到独立缓冲，等 DOF/Bloom 后处理完成后才合入最终图像，避免被景深模糊误伤。

带走这几条

  • 半透明不能写 GBuffer——改用前向着色，在不透明光照后、按后到前排序绘制，PS 直接算光照+混合。
  • 取光两条路：半透明光照体积（Froxel SH，低开销）+ Light Grid（0024，逐灯精确）；材质 Translucency Lighting Mode 控制选择。
  • 折射通过 SceneColorCopy 实现：半透明 PS 读背景 copy，写原 SceneColor，避免读写冲突。
  • Separate Translucency：独立缓冲 + 后处理后合入，防止 DOF 误处理半透明边缘。
  • 排序是 painter's algorithm，非 OIT——自相交网格有排序错误是固有局限。
