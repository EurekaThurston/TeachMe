RHI 线程与 FRHICommandList

渲染线程录制的"平台无关命令"如何被翻译成真正的 D3D12 调用？答案是一条专职翻译线程——RHI 线程——以及连接两者的命令列表体系：FRHICommandListImmediate、FRHICommandList、FRHICommandListExecutor。

时长 ~25 分钟 前置 0005（三线程架构）、0012（命令队列与同步）、0003（图形 API 契约） 产出 能解释 RHI 线程的职责、Immediate 与普通 CommandList 的区别、bypass 模式的含义，以及并行命令列表的录制与合并

你在 0012 看到渲染线程调用 FRHICommandListImmediate 方法录制绘制命令。那些调用并不立刻变成 ID3D12GraphicsCommandList::DrawIndexedInstanced()——它们先被序列化成一个引擎内部的命令流，再由 RHI 线程取出、翻译、最终提交 GPU。理解这条翻译管道，才能读懂 D3D12RHI 的任何代码。

1三线程架构回顾：RHI 线程的位置

UE 4.26 标准桌面配置有三条核心线程，各自的工作是：

游戏线程Tick → 产出"这帧场景长什么样" → ENQUEUE_RENDER_COMMAND 入队
渲染线程消费游戏命令 → 可见性剔除 → 组织各 Pass → 向 FRHICommandListImmediate 录制 RHI 命令 → 命令自动转发到 RHI 线程
RHI 线程消费渲染线程投递的 FRHICommand* 链表 → 调用具体 API（ID3D12GraphicsCommandList、VkCommandBuffer…）→ 提交 GPU 执行

RHI 线程的核心价值：把"翻译"工作从渲染线程解耦出去。渲染线程专注高层组织（选着色器、算资源屏障、维护 FScene），翻译成具体 API 调用这件繁琐又阻塞的事，留给 RHI 线程异步处理——两者在时间轴上重叠，节省了大量墙钟时间。

GRHIThread — RHI 线程的标识

全局指针 GRHIThread（Engine/Source/Runtime/RHI/Private/RHICommandList.cpp）在引擎启动时由 FRHIThread::Get() 初始化。它是 TaskGraph 的一个具名线程（ENamedThreads::RHIThread），与渲染线程类似：有自己的优先队列，在专属 CPU 核上循环取任务执行。可以通过控制台变量 r.RHICmdBypass 1 完全禁用 RHI 线程，退化为同步模式（详见第 4 节）。

2命令列表家族：三个类型，三种语义

UE RHI 有三种命令列表，它们不是继承关系，而是用途分工：

类型使用场景执行时机
FRHICommandListImmediate 渲染线程的"主命令列表"；绑定于 ENQUEUE_RENDER_COMMAND 的 lambda 参数。 命令被自动"推送"到 RHI 线程队列；渲染线程无需手动提交。
FRHICommandList 用于并行 Pass 录制（如并行 Shadow Depth 绘制）；可在 TaskGraph Worker 线程上创建。 录制完成后通过 FRHICommandListImmediate::QueueAsyncCommandListSubmit() 汇入主队列，RHI 线程按顺序翻译。
FRHIComputeCommandList Compute Pass 的专用接口（仅含 Dispatch 类命令）；RDG 的 Compute Pass lambda 接收此类型。 同 FRHICommandList；可被调度到异步 Compute 队列（若平台支持）。

FRHICommandListImmediate 实际上继承自 FRHICommandList，"Immediate"只是语义标记——意味着提交路径是自动驱动的，不需要显式 "Execute"。

这三种列表的内部表示都是同一套机制：一条内存中的命令对象单向链表（FRHICommand*）。每次调用 RHICmdList.DrawIndexedPrimitive(...)，引擎就在内存池里分配一个 FRHICommandDrawIndexedPrimitive 节点并追加到链表尾部——没有虚函数调用，通过模板展开写好了 Execute 函数指针。

3FRHICommandListExecutor：翻译引擎

FRHICommandListExecutor（RHI/Public/RHICommandList.h）是 RHI 线程侧的执行引擎。它的核心逻辑：

// RHICommandList.cpp — 概念简化，非逐字引用
void FRHICommandListExecutor::ExecuteInner(FRHICommandListBase& CmdList)
{
    FRHICommandBase* Cmd = CmdList.Root;
    while (Cmd)
    {
        // 每个命令节点带一个静态 Execute 函数指针
        Cmd->ExecuteAndDestruct(CmdList, *GDynamicRHI);
        Cmd = Cmd->Next;
    }
    CmdList.Reset();
}

Engine/Source/Runtime/RHI/Private/RHICommandList.cpp — ExecuteInner 的遍历逻辑

GDynamicRHI 是运行时选择的具体后端实例（如 FD3D12DynamicRHI）。每个 FRHICommand 的 ExecuteAndDestruct 会调用 GDynamicRHI 对应的 D3D12/Vulkan/Metal 方法。这就是 UE"平台无关命令 → 平台有关 API"的核心枢纽。

关键洞见 · 为什么用链表而非数组？

RHI 命令节点的大小不一（DrawIndexedPrimitive 比 SetShaderTexture 参数多得多），用链表 + 小对象内存池可以避免对齐浪费，同时追加是 O(1)。缺点是遍历时 CPU 缓存命中率稍低，但 RHI 线程本来就是"翻译+提交"而非高频数学运算，瓶颈在 D3D12 API 本身，缓存压力相对次要。

4Bypass 模式：r.RHICmdBypass

控制台变量 r.RHICmdBypass（默认 0）若设为 1，渲染线程上的 FRHICommandListImmediate 调用会直接调用 GDynamicRHI 的对应方法，绕过 RHI 命令链表和 RHI 线程，退化为纯同步模式。

Bypass 仅用于调试，不是性能优化

Bypass 消除了渲染线程与 RHI 线程的并行性，帧率通常显著下降。但它让调试变得简单：崩溃的调用栈直接指向 D3D12 API，无需穿越两层命令队列。在追查"哪条绘制命令导致设备丢失"时非常有用。生产构建中务必保持关闭。

Bypass 模式的检测代码：

// RHICommandList.h — 简化示意
FORCEINLINE bool Bypass() const
{
    return GRHICommandList.BypassMode() != 0;
}

// 渲染线程侧的调用路径（非 bypass）
void FRHICommandListImmediate::DrawIndexedPrimitive(...)
{
    if (Bypass())
    {
        GDynamicRHI->RHIDrawIndexedPrimitive(...); // 直接调用
        return;
    }
    // 否则：分配命令节点，追加到链表
    new (AllocCommand<FRHICommandDrawIndexedPrimitive>())
        FRHICommandDrawIndexedPrimitive(...);
}

5并行命令列表录制

4.26 的阴影通道、BasePass 都支持并行录制：多个 TaskGraph Worker 线程同时创建 FRHICommandList，各自录制自己负责的一批网格，最后汇入 FRHICommandListImmediate。

关键约束：合并顺序必须确定性——并行录制虽然乱序，但 QueueAsyncCommandListSubmit 按预先分配的槽位索引（ListIndex）排序后再交给 RHI 线程，保证 GPU 执行顺序与录制顺序一致。

FParallelCommandListSet — 并行录制的管理器

FParallelCommandListSet（Renderer/Private/SceneRendering.h）管理一次并行录制任务：它持有所有子 FRHICommandList 的指针，在析构时自动调用 Dispatch 把所有子列表按序汇入主列表。渲染线程的 Shadow/BasePass 并行路径都以此类为基础。

6渲染线程到 RHI 线程的推送时机

FRHICommandListImmediate 不是每次录制一条命令就立刻推给 RHI 线程——那会导致大量细粒度同步。实际的推送发生在以下三个节点：

  1. Immediate Flush：渲染线程主动调用 RHICmdList.ImmediateFlush(EImmediateFlushType::FlushRHIThread)（例如在需要上传纹理数据、或确认 GPU 完成特定工作时）。
  2. 帧末尾：RHIEndFrame() / Present 路径会触发最后一次 Flush，确保本帧所有命令都已提交。
  3. 命令链表超过阈值：若录制的命令数量过多（防止内存溢出），会自动触发一次分批推送。

正常情况下渲染线程与 RHI 线程的时间线关系如下：

渲染线程录制 Pass1 → 录制 Pass2 → 录制 Pass3 → ImmediateFlush → 继续录制 Pass4…
RHI 线程空闲 …… → 收到链表 → 翻译 Pass1 → 翻译 Pass2 → 翻译 Pass3 → 提交 D3D12 → ……

渲染线程录制期间 RHI 线程可以并发翻译上一批命令——这就是两线程并行节省时间的来源。

解释 FRHICommandListImmediate 和 FRHICommandList 的区别，以及并行录制如何保证最终提交顺序确定性。

FRHICommandListImmediate 是渲染线程的主列表，命令推送由引擎自动管理（Flush 节点驱动），使用者无需手动提交。FRHICommandList 是用于并行录制的子列表，可以在 Worker 线程上创建，录制完毕后通过 QueueAsyncCommandListSubmit 以预分配的 ListIndex 槽位索引汇入主列表。RHI 线程按槽位顺序翻译，因此无论 Worker 哪个先跑完，最终 D3D12 调用顺序都与渲染线程预定的逻辑顺序一致。

r.RHICmdBypass 打开后，渲染线程的调用栈会有什么变化？适合什么调试场景？

Bypass 打开后，FRHICommandListImmediate 的每次 Draw/Dispatch/SetState 调用会直接调用 GDynamicRHI->RHI* 方法，不再追加命令节点、不再经过 RHI 线程。这使得崩溃时的调用栈完整地从渲染线程高层函数延伸到 D3D12 API，无需"穿越"两层异步队列才能找到出错的绘制命令。适合：追查设备丢失（Device Removed）、验证某条绘制是否触发 D3D12 验证层报错、单步调试资源状态转换。生产构建中必须关闭，否则丢失渲染/RHI 线程并行带来的大量帧率。

带走这几条

  • RHI 线程（ENamedThreads::RHIThread，由 GRHIThread 标识）负责把平台无关的 FRHICommand* 链表翻译成具体图形 API 调用并提交 GPU。
  • FRHICommandListImmediate 是渲染线程的主列表，录制完成后自动推给 RHI 线程；FRHICommandList 是并行 Pass 录制用的子列表，录制完后通过 QueueAsyncCommandListSubmit 按槽位顺序汇入主列表。
  • FRHICommandListExecutor::ExecuteInner 遍历命令链表，对每个节点调用 ExecuteAndDestruct(GDynamicRHI)，这是"抽象 RHI → 具体 API"的核心枢纽。
  • r.RHICmdBypass 1 让渲染线程直接调用 GDynamicRHI，跳过 RHI 线程——调试利器，生产环境不用。
  • 推送时机：ImmediateFlush、帧末尾、链表超阈值——渲染线程录制期间 RHI 线程可并发翻译上一批命令。
