GPU 硬件管线：从三角形到像素

在拆解 Unreal 之前，先把"GPU 拿到一堆三角形之后到底做了什么"搞透。整条 UE 渲染管线，无论多复杂，最终都坍缩成一句话：让 GPU 把三角形变成像素。

时长 ~15 分钟 前置 写过 HLSL（会顶点/像素着色器即可） 产出 能凭记忆复述光栅化管线的每个阶段及其固定/可编程属性

你写过像素着色器，所以你知道"管线末端"长什么样。这节课要补的是：那个 float4 PS(...) 被调用之前和之后，硬件替你做了什么。这套阶段顺序是后面理解 UE 每一个 Pass 的物理基底——Depth Prepass、Base Pass、阴影、后处理，全都是"换一组着色器和渲染目标，再把同一条硬件管线跑一遍"。

1一次绘制调用，走过的七个阶段

当 CPU 发出一次 draw call（"用这个 PSO，画这 1200 个顶点"），GPU 内部按固定顺序推进下面这条逻辑管线。绿色是可编程（你写着色器），灰色是固定功能（硬件电路，你只能配置不能改写）。

这里省略了可选的几何阶段（Hull/Tessellation/Domain/Geometry Shader）。它们插在 VS 和图元装配之间，绝大多数 UE 物体不用，本课先不展开。

记住这条线，因为它是不变量。UE 的 Renderer 再花哨，本质都是在反复配置"这次用哪个 VS+PS、读写哪些缓冲"，然后让上面这条线跑一遍。下面逐段拆。

2顶点的旅程：从模型空间到屏幕

IA 阶段把显存里的顶点缓冲（位置、UV、法线…）和索引缓冲取出来，按图元拓扑（通常是三角形列表）喂给顶点着色器。VS 对每个顶点跑一次，最核心的职责是把顶点位置从模型空间一路变换到裁剪空间（clip space）。

VS 输出的是裁剪空间的 float4 (x,y,z,w)。注意：透视除法（÷w）和视口变换不是你做的——你只负责输出到裁剪空间，剩下的 NDC 归一化、映射到屏幕像素，是图元装配/光栅化阶段的固定功能硬件做的。

为什么停在裁剪空间而不是直接给屏幕坐标

裁剪（clipping）必须在透视除法之前做。w 分量携带了透视信息，硬件要用它判断顶点是否在相机背后或视锥外、并对跨越视锥边界的三角形做正确切割。如果你提前除掉 w，相机背后的点会被错误地翻到屏幕上。所以管线约定：着色器交付裁剪空间，硬件负责裁剪与除法。

VS 还会输出一组插值变量（UV、世界法线、顶点色等，HLSL 里随 SV_Position 一起 return 的那些）。这些值会在光栅化阶段被插值后送进像素着色器——下一段就是它。

3光栅化：三角形如何变成一堆片元

光栅化是整条管线的心脏，也是"几何世界"和"像素世界"的分界线。它的输入是屏幕空间里的一个三角形，输出是这个三角形覆盖的所有片元（fragment）——可以理解成"候选像素"。1

片元 ≠ 像素：一个像素位置可能被多个三角形各产生一个片元（重叠时），最终谁写进帧缓冲由深度测试决定。MSAA 下一个像素还有多个覆盖采样点。

硬件怎么知道一个像素在不在三角形内？用边函数（edge function）：对三角形的三条边各算一个有符号距离，三个符号一致就说明像素在内部。GPU 把屏幕切成小块（tile），对每块并行测试所有像素，所以光栅化天然是大规模并行的。

关键洞见 · 属性是"透视校正插值"出来的

像素着色器拿到的 UV、法线，不是简单地在屏幕上线性插值三个顶点——那样会在透视下扭曲（想象铁轨贴图近大远小）。硬件用 1/w 加权做透视校正插值，才能让纹理在斜面上看起来正确。这是固定功能硬件免费帮你做的事，但理解它能解释很多采样 artifact。

光栅化还顺手算两样东西：每个片元的深度值（用于下一步的深度测试）和（在 PS 里可取的）屏幕坐标 SV_Position。到此，几何信息已经"拍扁"成屏幕上一格一格的片元，准备逐个着色。

4像素着色与输出合并：决定最终写什么

像素着色器对每个片元跑一次，算出要输出的颜色（或者——在 UE 的 Base Pass 里——一整组 GBuffer 通道）。这是你最熟悉的部分，但它后面还有一道固定功能关卡：输出合并（Output Merger, OM）。OM 决定 PS 的输出到底能不能、以及如何写进渲染目标，做两件事：

OM 子阶段做什么UE 里的体现
深度/模板测试拿片元深度和深度缓冲比较，没通过就丢弃该片元（PS 可能根本不运行）。Depth Prepass 建立的 Early-Z 让 Base Pass 省掉大量被遮挡片元的着色。
混合 (Blend)把 PS 输出和目标已有颜色按公式合成（如半透明 src·α + dst·(1−α)）。不透明物体关混合（直接覆盖）；半透明 Pass 开混合。
Early-Z 与 Overdraw —— 后面整条 UE 主线的动机

深度测试默认在 PS 之后。但若 PS 不改深度、不丢弃片元，硬件可把测试提前到 PS 之前（Early-Z），从而跳过被遮挡片元的昂贵着色。Overdraw（同一像素被反复着色）是延迟着色管线的头号敌人——这正是 UE 先跑一遍只写深度的 Depth Prepass 的全部理由（见 0022）。你现在就能理解那节课的动机了。

5多渲染目标：GBuffer 的硬件基础

一个像素着色器可以同时往多个渲染目标（Multiple Render Targets, MRT）写不同数据：SV_Target0、SV_Target1…… 这就是 UE GBuffer 的硬件机制——Base Pass 的 PS 一次性把基础色、法线、粗糙度、金属度分别写进 4~6 张 RT。延迟着色之所以可行，靠的就是 MRT 这个硬件能力。你在 0021/0023 会看到 UE 的具体 GBuffer 布局，但底层就是"一个 PS，多个 SV_Target"。

检索练习

顶点着色器输出的位置处于哪个空间，谁负责把它变成屏幕像素坐标？

不开混合的不透明物体，深度测试为什么能搬到像素着色器之前？这对性能意味着什么？
当像素着色器不修改深度、不做 clip/discard、不开 alpha-to-coverage 时，片元的最终深度在光栅化后就已确定，硬件可以在运行昂贵的 PS 之前先做深度测试（Early-Z），把被遮挡的片元直接丢弃、根本不着色。意义：消除 overdraw 的着色开销。代价是 UE 愿意先花一个只写深度的 Depth Prepass 把深度缓冲填满，换来 Base Pass 几乎零浪费。这是延迟着色性能模型的基石。

带走这几条

  • 一次 draw call 的硬件管线固定为：IA → VS → 图元装配 → 光栅化 → PS → OM。绿色（VS/PS）可编程，其余固定功能。
  • VS 只把顶点送到裁剪空间；透视除法与视口变换是硬件做的——为了让裁剪发生在除 w 之前。
  • 光栅化是几何与像素的分界线，用边函数判定覆盖，并对属性做透视校正插值。
  • Early-Z 让深度测试前移、跳过被遮挡片元的着色——这是 UE Depth Prepass 存在的理由。
  • MRT（一个 PS 写多个 SV_Target）是 GBuffer 与延迟着色的硬件基础。
