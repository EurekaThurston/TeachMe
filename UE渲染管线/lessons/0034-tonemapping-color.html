色调映射与颜色分级

ACES 胶片色调映射曲线把 HDR 线性颜色压缩到显示器可呈现的 LDR；CombineLUTs 预合并颜色分级操作产生一个 3D LUT；色调映射 Pass 在单次着色器调用中完成曝光缩放、泛光叠加、LUT 查表和输出 gamma 编码。

时长 ~22 分钟 前置 0033（泛光与曝光）、0031（后处理概述） 产出 能解释 ACES 曲线的作用、CombineLUTs 的工作机制、色调映射 Pass 的完整输入/输出，以及 sRGB 与 HDR 显示输出的区别

色调映射是整条渲染管线里最后一道"物理→感知"的门槛。到目前为止，SceneColor 持有的是物理上准确的 HDR 线性辐亮度——这些值中有超过 1.0 的高亮，有极小的暗部细节，跨越 5–6 个量级的动态范围。显示器只能显示 [0, 1] 的窄范围。色调映射的任务是把这个宽动态范围压缩成显示器可呈现的范围，同时在视觉上保持高光不焦糊、暗部不死黑、整体对比度自然——这正是 ACES 胶片曲线解决的问题。

1ACES 胶片色调映射曲线

UE 4.26 默认使用 ACES（Academy Color Encoding System）胶片色调映射器，它是一条 S 形曲线，在视觉上模拟电影胶片的曝光响应曲线：暗部保留细节（toe），中间调对比强烈（linear section），高光优雅过渡（shoulder）而不焦糊。

UE 4.26 实现的 ACES 曲线来自 Krzysztof Narkowicz 的拟合公式（在 PostProcessTonemap.usf 中）：

// PostProcessTonemap.usf — ACES 胶片色调映射拟合
float3 ACESFilm(float3 x)
{
    float a = 2.51f;
    float b = 0.03f;
    float c = 2.43f;
    float d = 0.59f;
    float e = 0.14f;
    return saturate((x*(a*x+b))/(x*(c*x+d)+e));
}

这是一个有理多项式拟合，输入输出均为 [0, ∞) → [0, 1]。系数由 Narkowicz 的 ACES RRT+ODT 近似拟合，在质量/性能间取得平衡。注意：这不是完整 ACES 标准的 RRT（Reference Rendering Transform），而是一个适合实时渲染的视觉近似。

ACES 完整标准 vs. UE 实现

完整 ACES 流水线包含 IDT（Input Device Transform）、RRT（Reference Rendering Transform）、ODT（Output Device Transform），其中 RRT 是一个复杂的颜色空间变换和 S 形压缩，ODT 根据目标显示设备做 gamma、色域、亮度调整。UE 4.26 的 ACES 实现是一个视觉上对齐但计算简化的近似——它提供了 ACES 的"胶片感"，但不是 AMPAS 标准的参考实现。UE 4.26 也支持配置 r.TonemapperGamut 选择色域，但这都属于近似实现范畴。

FPostProcessSettings 中的 ACES 参数允许美术调整曲线形状：

参数对应曲线部分默认值效果
FilmSlope中间调斜率（对比度）0.88增大 → 对比更强
FilmToe暗部曲线（toe）0.55增大 → 暗部更快压黑

2CombineLUTs：预合并颜色分级为 3D LUT

颜色分级（Color Grading）是美术对画面颜色做最终调整的步骤——饱和度、色调偏移、色调曲线、白平衡等。如果每帧在色调映射着色器里逐参数计算这些操作，性能开销较高。UE 4.26 的解决方案是预合并 LUT（CombineLUTs）：

在色调映射 Pass 之前（但频率比每帧低，通常每帧只跑一次），运行一个小的 CombineLUTs Pass（PostProcessCombineLUTs.cpp），它把所有颜色分级操作烘培成一张 32×32×32 的 3D LUT 纹理（PF_A2B10G10R10 或 PF_FloatRGBA）：

这张 LUT 把线性 HDR 域的颜色查询映射到最终 LDR 输出颜色。色调映射 Pass 主体 PS 里对每个像素只需做一次 3D 纹理查表（Volume.SampleLevel），就完成了所有颜色分级 + ACES 曲线 + gamma 编码——O(1) 查表替代了多步参数计算。

关键洞见 · LUT 的维度与精度权衡

32×32×32 = 32768 个查表点。每个颜色通道 5 bit 精度（32 级），三线性插值补偿中间值。这个精度对视觉无损：人眼对颜色的分辨率不高，3D LUT 的 5 bit 精度在绝大多数情况下看不出量化瑕疵，除非极端颜色分级（如高对比度调色）。如需更高精度，理论上可以升到 64³，但显存翻 8 倍，且 3D 纹理采样的带宽成本也随之增加。

3色调映射 Pass 的完整着色器逻辑

色调映射 Pass 是整条后处理链里工作最密集的单个 PS（PostProcessTonemap.usf）。它在一个像素着色器调用里完成：

// PostProcessTonemap.usf — 主 Pass 逻辑（结构示意）
float4 TonemapPS(...) : SV_Target0
{
    // 1. 读 SceneColor（已含曝光缩放）
    float3 HDR = SceneColorTexture.Sample(Sampler, UV).rgb;
    HDR *= ExposureScale; // 曝光倍数（来自 EyeAdaptation 1×1 缓冲）

    // 2. 加泛光
    float3 Bloom = BloomTexture.Sample(Sampler, UV).rgb;
    HDR += Bloom * BloomIntensity;

    // 3. 镜头光晕（可选）
    // HDR += LensFlare * LensFlareIntensity;

    // 4. 查 3D LUT（已包含 ACES + 颜色分级 + gamma）
    // LUT 的 UV：把 HDR 值映射到 LUT 的查表坐标（对数映射）
    float3 LUTCoord = LinearToLUTSpace(HDR);
    float3 LDR = ColorGradingLUT.SampleLevel(Sampler, LUTCoord, 0).rgb;

    // 5. 输出到 LDR 渲染目标（sRGB gamma 由 RT 格式的 sRGB flag 自动处理，或在 PS 里做）
    return float4(LDR, 1.0);
}

输出格式为 PF_B8G8R8A8（若渲染目标开启 sRGB flag，硬件在写入时自动做线性→sRGB gamma 转换）或 PF_A2B10G10R10（10bit/channel 用于 HDR10 输出）。

5输出设备：sRGB、Rec.709 与 HDR10

色调映射的输出目标格式取决于显示设备：

输出设备目标格式gamma / OETF色域
普通 SDR 显示器PF_B8G8R8A8（sRGB flag）sRGB（≈gamma 2.2）Rec.709
HDR10 电视/显示器PF_A2B10G10R10PQ（ST2084）Rec.2020（目标，可能裁剪）
scRGB（PC HDR）PF_FloatRGBA（R16G16B16A16F）线性（scRGB 约定）Rec.709 扩展

UE 4.26 通过 r.HDR.EnableHDROutput 和 r.HDR.Display.OutputDevice 控制输出路径：

  • OutputDevice = 0（默认）：SDR sRGB，CombineLUTs 包含 sRGB gamma 编码；
  • OutputDevice = 3：HDR10 PQ，CombineLUTs 改用 PQ OETF 替代 sRGB gamma；
  • OutputDevice = 5：scRGB，输出线性值（交换链格式为 16F）。
sRGB flag 与手动 gamma 的区别

D3D 的 sRGB 渲染目标（DXGI_FORMAT_B8G8R8A8_UNORM_SRGB）让硬件在 PS 写入时自动把线性值转换为 sRGB gamma——等效于 pow(x, 1/2.2) 的近似。如果在 PS 里手动做 LinearToSRGB，然后写入非 sRGB RT，效果相同但多一步 ALU 指令。UE 4.26 的 CombineLUTs 会根据目标 RT 格式是否带 sRGB flag 决定是否在 LUT 生成时内嵌 gamma 转换——查一次 LUT 就包含了 gamma，不用额外 PS 指令。

禁用 ACES 的代价

r.TonemapperFilm = 0 会禁用 ACES 胶片曲线，改用一个简单的 Reinhard 色调映射（x / (x + 1)）。这会让画面高光更"平"（没有胶片感的 shoulder），整体对比度下降，但对需要完全线性颜色输出的应用（如合成管线、可视化）有时是必要的。CombineLUTs 在无 ACES 模式下仍然运行，颜色分级参数仍然有效。

检索练习

CombineLUTs Pass 在 UE 4.26 中的作用是什么，它的输出是什么格式的数据？

说出色调映射 Pass 在一次 PS 调用里按顺序完成的所有操作，以及 CombineLUTs 为什么必须在色调映射主 Pass 之前运行。

色调映射 PS 的操作顺序：

  1. 读取 SceneColor HDR 像素颜色；
  2. 乘以 ExposureScale（来自 1×1 曝光缓冲）做曝光校正；
  3. 加上泛光纹理（BloomIntensity 权重），在线性 HDR 域叠加光晕；
  4. 把曝光后的 HDR 值映射到 3D LUT 的查表坐标（对数空间 UV）；
  5. 从 3D LUT 三线性插值采样，得到包含了 ACES + 颜色分级 + gamma 编码的最终 LDR 颜色；
  6. 输出到 LDR RT（sRGB 8bit 或 HDR10 10bit）。

CombineLUTs 为什么必须先跑：色调映射 PS 里的 LUT 查表假设 LUT 已经烘培完毕。CombineLUTs 把当前帧的颜色分级参数合并成 LUT，如果在色调映射之后跑，LUT 还未更新，PS 会读到上一帧的 LUT（或未初始化数据）。所以 CombineLUTs 是色调映射 Pass 的前置依赖，RDG 会把它放在更早的节点。

带走这几条

  • ACES 胶片曲线是 UE 4.26 默认色调映射器：有理多项式 S 形，保暗部细节（toe）、压高光（shoulder）；系数可通过 FilmSlope 等参数调整。
  • CombineLUTs 把所有颜色分级（白平衡、饱和度、曲线、用户 LUT、gamma）预合并为一张 32³ 3D LUT，使色调映射 PS 只需一次查表。
  • 色调映射 PS 在单次调用里完成：ExposureScale 乘法 → Bloom 加法 → LUT 查表 → 输出 LDR。
  • 输出格式取决于目标显示设备：SDR 用 PF_B8G8R8A8（sRGB），HDR10 用 PF_A2B10G10R10（PQ OETF），LUT 生成时已内嵌对应的 gamma/OETF 转换。
  • 色调映射是整条后处理链里 HDR→LDR 的唯一转换点；之后的 UI 合成和 Present 在 LDR 空间工作。
