Part 8 · 调试与性能 · Lesson 0040

RenderDoc 抓帧调试：从事件到 GBuffer

一次 RenderDoc 抓帧，就是把本课程 Part 4–7 讲过的所有 Pass 完整地铺在你面前。这节课教你如何把事件树里的每个节点对应回 UE 源码中的 Pass，如何读 GBuffer 验证材质数据，如何追到具体的 PSO 和着色器——是调试"为什么这个像素颜色不对"的实战闭环。

时长 ~25 分钟 前置 0005（帧全景）、0021–0030（各 Pass 原理）、0015/0016（RDG） 产出 能独立抓取并解读 UE 4.26 帧；能定位 GBuffer 通道、追踪 draw call 到着色器

你学完了整条管线：Depth Prepass、Base Pass、Light Grid、Shadow、Lighting、SSAO/SSR、Translucency、TAA、Tonemap……但在代码和文字之间，有一层需要"亲眼看"才能打通的理解——那正是 RenderDoc 给的。抓帧工具不是黑盒神器；它只是把 GPU 驱动执行的每一条命令记录下来，让你逐帧、逐 Pass、逐 draw call 地回放和检查。你已经知道"这些 Pass 是什么"，现在学的是"在工具里在哪里找到它们，以及怎么读"。

1附加 UE 进程并抓帧

UE 4.26 最干净的抓帧方式是让 RenderDoc 在启动时注入，而不是事后附加——这样确保交换链也被拦截。有三种路径：

方式步骤适用场景
RenderDoc 直接启动在 RenderDoc 的 "Launch Application" 填入 UE Editor 或 Standalone 可执行文件路径与命令行参数，然后 Launch。最可靠，PIE 和独立包均可
UE 内置 RenderDoc 插件在 Edit → Plugins 搜索并启用 "RenderDoc" 插件，重启编辑器。工具栏出现相机图标；点击抓取当前 PIE 帧。迭代快，无需切窗口
事后附加（Attach）UE 已在运行，在 RenderDoc 选 "Attach to Running Instance"。可行但有时拦截不到交换链

抓帧快捷键默认 F12（RenderDoc 窗口需在前台，或用插件图标）。抓到后 RenderDoc 会弹出捕获列表，双击打开。

注意 · 启动参数

调试时建议在 UE 启动命令行加 -d3ddebug（开启 D3D Debug Layer，会在 RenderDoc 的 Errors & Warnings 里显示更多校验信息）和 -norhithread（禁用 RHI 线程，命令提交变同步，事件顺序更直观）。生产性能测试时不要加这两个参数，它们会严重影响帧时间。

2事件树：Pass 名从哪里来

打开捕获文件，左侧是 Event Browser（事件浏览器）。你会看到一棵树，节点名形如 BeginPass、ShadowDepths、BasePass、Lights……这些名字完全来自 UE 源码里的作用域标记宏，与 Pass 实现一一对应。

原理 · SCOPED_DRAW_EVENT 与 RDG_EVENT_SCOPE

UE 4.26 中有两套打标签的机制。

非 RDG Pass 用 SCOPED_DRAW_EVENT(RHICmdList, SomePassName)（定义在 Engine/Source/Runtime/RenderCore/Public/DrawDebugInfo.h）。它在构造时向命令列表插入一条 BeginEvent、析构时插入 EndEvent；D3D12 RHI 把它翻译成 PIXBeginEvent/PIXEndEvent，RenderDoc 截到这对标记就生成对应的树节点。

RDG Pass 用 RDG_EVENT_SCOPE(GraphBuilder, "SomePassName")（定义在 Engine/Source/Runtime/RenderCore/Public/RenderGraphUtils.h）。RDG 编译执行时，Pass 执行 lambda 里会自动插入同样的 BeginEvent/EndEvent，所以 RDG Pass 的名字也出现在事件树里。

结论：事件树节点名 = UE 源码里 SCOPED_DRAW_EVENT/RDG_EVENT_SCOPE 的字符串参数。你在源码里 grep 节点名，就能精确找到对应代码。

以下是典型帧里你会看到的顶级节点，以及它们对应的课程章节：

事件树节点对应 UE Pass本课程章节

BasePassBase Pass，写 GBuffer（MRT）0023
ShadowDepthsCSM + 点/聚光投影阴影深度图0026
Lights / DirectLighting延迟光照，读 GBuffer 叠光照0025

UE 4.26 中部分 Pass 仍处于从传统 SCOPED_DRAW_EVENT 向 RDG RDG_EVENT_SCOPE 迁移的中间态。两种宏在事件树里看起来完全一样，区别在源码层面。

3检查 GBuffer：验证材质数据

在事件树里展开 BasePass，找到任意一个 draw call 并选中，切换到 Texture Viewer（纹理查看器）面板。此时右上角 "Outputs" 标签会显示当前 draw 绑定的多个 渲染目标——这正是 GBuffer 的各张 RT。

UE 4.26 默认 GBuffer 布局（见 0021）在抓帧里映射如下：

RT 槽位格式（典型）存储内容视觉特征
GBufferA / SV_Target0R8G8B8A8世界法线（RGB，编码）+ Shading Model ID（A）彩色法线图，背景区域全黑
GBufferB / SV_Target1R8G8B8A8金属度（R）、高光度（G）、粗糙度（B）、各向异性/特殊（A）通道分离看最清晰
GBufferC / SV_Target2R8G8B8A8 或 B8G8R8A8基础色（Base Color）RGB + AO（A）接近漫反射颜色的外观
GBufferD / SV_Target3R8G8B8A8自定义数据（毛发、次表面参数等）；不用时为 0通常较暗/均匀
GBufferE / SV_Target4R8G8B8A8预计算阴影系数（Precomputed Shadow Mask）白/黑区域明显
Depth BufferD32 或 D24S8场景深度（Prepass 已写入）越远越亮（取决于视角）
Velocity / Motion BufferR16G16屏幕空间运动向量（TAA 用）运动区域有颜色，静止为黑
关键洞见 · 为什么 BasePass 结束时看深度已经有内容

因为 Depth Prepass（0022）已经在 BasePass 之前把深度缓冲写满了。如果你选中 BasePass 第一个 draw 之前的帧状态，就会看到深度图里的完整场景；BasePass 的每个 draw 利用 Early-Z 跳过了被遮挡的片元——只向 GBuffer RT 写入通过深度测试的可见像素。这正是 0022 讲的"Prepass 的收益在 BasePass 兑现"。

在纹理查看器里，你可以：

  • 点击 "R/G/B/A" 按钮单独查看某个通道——这是检查粗糙度、金属度最直观的方式。
  • 右键选 "Save Texture" 导出为图片——用于 bug 截图留存。
  • 调整 "Range" 滑杆——深度图通常需要收窄范围才能看出细节。
  • 在 "Previous" / "After" 模式下对比 draw 执行前后 RT 变化，精确确认是哪个 draw 写入了特定区域。

4选中一个 draw call：读 Pipeline State

在事件树里选中任意一个 draw call，切换到 Pipeline State 面板。这个面板把当次绘制的完整 PSO 展示出来，分几个子页：

  • IA（Input Assembler）：顶点缓冲绑定、顶点步长、索引缓冲。可以验证网格的顶点布局（位置、法线、UV 各在哪个 slot）。
  • VS / PS：当前绑定的顶点/像素着色器。点击着色器名旁边的"View"按钮，可直接看反编译后的 DXBC/DXIL，也可看 HLSL 反汇编。
  • RS（Rasterizer）：Cull Mode、Fill Mode、深度偏差（ShadowDepth Pass 会有非零 depth bias）。
  • OM（Output Merger）：绑定的 RT 数量和格式、Blend State、Depth-Stencil State。BasePass 的 OM 应显示 4–6 个活跃 RT，ShadowDepth Pass 的 OM 只有一个深度 RT（无颜色 RT）。
  • 资源绑定（Resources）：Texture2D / Buffer / SamplerState 的完整列表。可以验证材质纹理是否正确绑定、是否错误引用了前一帧残留资源。
源码锚点 · UE 的着色器与 RenderDoc 的对应

UE 的着色器经由虚拟着色器路径系统（GetShaderFilename()，Shaders/Private/ 目录下 .usf 文件）编译成 DXBC。RenderDoc 显示的是 DXBC 反汇编或 DXIL，你无法直接看到 .usf 源码。但可以：① 在 UE Editor 里用 r.ShaderDevelopmentMode 1 + r.DumpShaderDebugInfo 1 让 UE 把编译后着色器及中间 HLSL dump 到磁盘（路径形如 Saved/ShaderDebugInfo/）；② 用 RenderDoc 里的 "Edit Shader" 功能测试 HLSL 修改，但这仅是调试用途，不影响引擎持久状态。此外，r.BasePassForceFullyRoughSurface 1 可以强制所有材质粗糙度=1（镜面为0），快速确认高光异常是否来自材质数据。

5顶点输入检查：网格数据验证

选中 draw call，点击 Pipeline State → IA 区域的 "Mesh View" 按钮（或 Mesh Viewer 面板），RenderDoc 会展示原始顶点数据和光栅化后的三角形预览。典型用途：

  • 验证蒙皮网格骨骼变换后的顶点位置是否正确（异常的 skinning 会产生顶点飞出/穿模）。
  • 检查法线切线方向——法线朝向错误会直接导致 GBuffer 法线通道数据错乱，进而影响光照。
  • 核实顶点工厂（Vertex Factory，见 0018）的输出：实例化绘制时，每个实例的位置/变换是否通过 GPUScene（见 0020）正确填充。在 IA 面板展开 Vertex Buffer 列表，实例数据通常在单独的 slot 里（步长为 per-instance）。

6追到具体着色器：HLSL 反汇编阅读要点

在 Pipeline State 的 PS 区域点击 "View"，RenderDoc 显示反汇编 HLSL（DXBC 阶段）。阅读技巧：

  • 寄存器映射：常量缓冲 cb0、cb1… 对应 UE 里的 FViewUniformParameters（cb0）、逐图元数据等。纹理 t0、t1… 对应 HLSL 的 Texture2D 绑定，在 Resources 面板里可以看到它们具体绑定了哪张贴图。
  • sample 指令：sample_indexable / sample_b 是纹理采样。如果你看到大量 sample 指令，说明该着色器纹理访问多——这往往是带宽瓶颈的信号。
  • 数学密度：大量 mad（Multiply-Add）指令说明着色器数学运算多，可能是 ALU-bound。
  • UE 的 Ubershader 机制在 4.26 里通过 SF_（Shader Feature）宏静态分支，最终编译出的 DXBC 不含 dynamic branch，只有 flat 指令流——反汇编看起来没有显式 if 语句，而是大量预展开的 MAD/条件 MOV。
关键洞见 · ShadowDepth Pass 的 PS 为空

在事件树里找到 ShadowDepths，选中其中一个 draw。切到 Pipeline State → PS 区域，你会发现 PS 处于 "Disabled" 或绑定了一个极简着色器（仅 return）。这正是 0026 讲的原理：纯深度通道不需要像素着色输出，只靠 OM 的深度写入——硬件用 Early-Z 跳过 PS 完全合法，大幅降低阴影图生成的 GPU 开销。

注意 · 4.26 与更新版本的差异

本课所有 Pass 名称和 GBuffer 布局严格对应 UE 4.26。若你用的是编辑器构建而非 Shipping，事件名会更丰富（Debug builds 下 SCOPED_DRAW_EVENT 始终激活；Shipping 下部分标记被编译掉）。如果在 UE5 里打开同样的工作流，会看到 Nanite / Lumen / VSM 等新节点，本课范围内不讨论。

检索练习

在 RenderDoc 事件树里，UE 的 Pass 名称（如"BasePass"、"ShadowDepths"）来自于源码中的什么机制？

不看表，依次说出 UE 4.26 GBuffer 的 A/B/C 通道各存什么，以及为什么 BasePass 开始前深度缓冲就已经有内容了。

GBufferA（SV_Target0）：世界空间法线（RGB，编码到 [0,1]）+ Shading Model ID（Alpha）。

GBufferB（SV_Target1）：金属度（R）、高光度 Specular（G）、粗糙度（B）、自定义/次表面数据（A）。

GBufferC（SV_Target2）：基础色 Base Color（RGB）+ AO（A）。

深度缓冲在 BasePass 开始前就有内容，是因为 Depth Prepass（0022）先于 BasePass 运行，用极简着色器对所有不透明物体写了一遍深度——这是 Early-Z 策略，让 BasePass 的材质着色器只在最前面的可见片元上跑，消除 overdraw。

带走这几条

  • RenderDoc 事件树名称 = UE 源码中 SCOPED_DRAW_EVENT / RDG_EVENT_SCOPE 的字符串参数，可直接 grep 到源码位置。
  • 事件树里的节点层级与本课程 Part 5–6 的 Pass 顺序一一对应：PrePass → BasePass → ShadowDepths → Lights → SSAO/SSR → Translucency → PostProcessing。
  • 纹理查看器里可逐通道读取 GBuffer RT，验证法线/粗糙度/基础色是否从材质正确写入。
  • Pipeline State 面板暴露完整 PSO：RT 绑定、Blend State、Depth-Stencil、着色器反汇编——是追踪"为什么输出不对"的直接证据。
  • ShadowDepth Pass 的 PS 为空（或 disabled）是设计如此，不是 bug——深度通道不需要颜色输出。
