渲染模块地图：Engine / Renderer / RHI / RenderCore / D3D12RHI

UE 4.26 的渲染相关代码分散在五个以上的模块里，职责各异、依赖有向无环。弄清楚"这个类在哪个模块、这个模块能被谁 include"，是读源码、写插件、调试崩溃的基础。

时长 ~18 分钟 前置 0005（一帧生命周期）、0006（三线程流水线） 产出 能说出 Engine / Renderer / RHI / RenderCore / D3D12RHI 各自的职责、关键类放在哪个模块、以及依赖方向；理解为何不能直接 include Renderer

你刚看完三线程如何接力，现在要落地到源码目录：游戏线程的 UWorld 在 Engine 模块，FDeferredShadingSceneRenderer 在 Renderer 模块，FRHICommandList 在 RHI 模块，FShader 在 RenderCore 模块，真正的 D3D12 调用在 D3D12RHI 模块。这五个模块的依赖方向是单向有向图，打破任何一条箭头都会编译失败。这节课把这张地图印进大脑。

1五大模块一览

Engine Engine/Source/Runtime/Engine/ 游戏世界的对象模型。持有 UWorld、AActor、UPrimitiveComponent，以及把场景推送给渲染线程的接口（UGameViewportClient::Draw、FSceneInterface）。这里是游戏线程的地盘。 UWorld、AActor、UStaticMeshComponent、FSceneInterface

RenderCore：横向共享层

RenderCore 不在主依赖链上，而是被多个模块共同引用。FShader/FGlobalShader 在这里，FRenderResource（顶点缓冲、索引缓冲的基类）在这里，RDG 的 FRDGBuilder/FRDGTexture/FRDGBuffer 也在这里。如果你在写插件时想用 RDG 但不想依赖 Renderer，RenderCore 就是正确的依赖目标。

3Renderer 是 Private 模块：这意味着什么

Renderer 模块在其 Renderer.Build.cs 中没有导出公开头文件（不设 PublicIncludePaths），即声明为 private。这意味着：

  • 外部代码（包括你自己的游戏模块、插件）无法直接 #include "DeferredShadingRenderer.h" 或调用 FDeferredShadingSceneRenderer 的任何方法。
  • 扩展渲染行为通常要走引擎提供的钩子接口（如 IRendererModule、Scene Extension、自定义 Pass Processor），而不是直接调渲染器内部类。
  • 若你确实需要修改 Renderer 内部（如添加新 Pass），必须修改引擎源码——这正是为什么很多引擎级渲染特性都要拉 source build 的根本原因。
为什么不把 Renderer 做成公开模块

渲染逻辑高度内聚，内部类之间互依赖复杂，FDeferredShadingSceneRenderer 的 Render() 在一个函数里调用了数十个内部子系统。若对外暴露，任何外部代码都能依赖这些实现细节，版本间重构就会破坏大量下游——Epic 选择"完全私有 + 有限公开接口"来保持模块内聚和演进自由。UE5 中 Renderer 扩展性进一步通过 ISceneExtension 等机制增强，但 private 原则未变（UE5 特性，超纲）。

4着色器系统在 RenderCore：FShader 与 .usf

HLSL 着色器在 UE 里有一套完整的 C++ 封装，全部住在 RenderCore：

C++ 类职责对应 .usf 用法
FShader所有着色器的基类，持有编译产物的引用、参数绑定信息每个 .usf 入口函数对应一个 FShader 子类
FGlobalShaderFShader 的子类，单例式：整个引擎只有一个实例，不依赖 Material 参数后处理、屏幕空间 Pass（TAA、Bloom、Tonemap）的着色器全用 FGlobalShader
FMaterialShader依赖 Material 的排列，每个 Material×VertexFactory×Pass 有各自的编译实例Base Pass 的材质着色器；在 Renderer 模块使用，但基类在 RenderCore

实际的 HLSL 代码写在 Engine/Shaders/Private/*.usf（.usf = Unreal Shader File，本质是带 UE 宏扩展的 HLSL）。编译流程在 RenderCore 的着色器编译器（ShaderCompiler.cpp）里管理，最终产物被缓存到 DerivedDataCache。

关键洞见 · FGlobalShader vs FMaterialShader

后处理链所有的 Pass（TAA、Bloom、Tonemap、SSAO…）都用 FGlobalShader——因为它们不依赖材质参数，整个引擎只编译一份。而 Base Pass 每个材质要编译 N 个排列（不同 VertexFactory × 光照模型 × 着色模式的叉积），代价高、量大，这就是 UE 着色器编译慢的根本原因。你在 HLSL 里写的每一个 if defined() 宏分支，都可能倍增排列数。

5平台 RHI 如何在运行时插入

D3D12RHI（以及 VulkanRHI、D3D11RHI）并不被 Renderer 或 RHI 静态链接。它们在引擎启动时根据平台/配置动态加载，调用 RHIInit 把一个 FDynamicRHI 的派生实例（如 FD3D12DynamicRHI）注册为全局 RHI 实现。此后所有 FRHICommandList 上的命令，最终都通过这个全局实例分发到平台代码。

这就是为什么切换 D3D11/D3D12/Vulkan 不需要重新编译游戏代码——抽象层（RHI）和实现层（平台 RHI）在运行时才绑定。源码锚点：Engine/Source/Runtime/D3D12RHI/Private/D3D12RHI.cpp 中的 FD3D12DynamicRHI::Init()。

不看表格，说出 Renderer 模块为何是 private、以及这对想扩展渲染管线的开发者意味着什么。
Renderer 是 private 模块，意味着外部代码（游戏模块、插件）无法直接 #include 其头文件或调用 FDeferredShadingSceneRenderer 等内部类。原因是渲染逻辑高度内聚、内部互依赖复杂，公开会导致任何重构都破坏外部依赖。对开发者的影响：扩展渲染行为要走 IRendererModule、Scene Extension、自定义 Pass Processor 等公开钩子，而不能直接调内部函数；若必须修改 Renderer 内部（如新增 Pass），需要拉取引擎源码（source build），是真正的引擎修改而非插件扩展。

带走这几条

  • Engine：游戏对象层（UWorld/Actor/Component），游戏线程的地盘；路径 Runtime/Engine/。
  • Renderer：场景渲染逻辑（FDeferredShadingSceneRenderer、Pass、可见性），private 模块，外部不能 include；路径 Runtime/Renderer/。
  • RHI：平台无关图形 API 抽象（FRHICommandList、FRHIResource）；路径 Runtime/RHI/。
  • RenderCore：着色器基础设施（FShader/FGlobalShader）+ 渲染资源（FRenderResource）+ RDG（FRDGBuilder）；横向共享层，路径 Runtime/RenderCore/；着色器 .usf 在 Engine/Shaders/Private/。
  • D3D12RHI：RHI 接口的 D3D12 具体实现，运行时动态注入，不是编译时静态依赖；路径 Runtime/D3D12RHI/。
  • 依赖方向：Engine → Renderer → RHI ← D3D12RHI（运行时）；RenderCore 横向被多方依赖。
