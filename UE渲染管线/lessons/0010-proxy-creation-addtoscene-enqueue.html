代理的创建与更新：AddToScene 与渲染命令入队

一个图元如何从游戏线程的 UPrimitiveComponent 走过 ENQUEUE_RENDER_COMMAND，最终在渲染线程侧被 FScene::AddPrimitiveSceneInfo_RenderThread 注册、并在 AddToScene 时缓存绘制命令。

时长 ~20 分钟 前置 0009（UWorld/FScene/Proxy 三件套） 产出 能描述图元注册/更新如何经 ENQUEUE_RENDER_COMMAND 跨线程到达 FScene，并说清 AddToScene 时缓存绘制命令的时机

0009 建立了概念地图：游戏侧 Component、渲染侧 Proxy、渲染器记账 SceneInfo。这节课看机制：这三个对象是怎么被创建的，变换更新怎么跨线程传递，以及 AddToScene 里藏着的一步——CacheMeshDrawCommands——为什么对静态网格性能至关重要。

1图元注册：从 AddPrimitive 到 AddToScene

当一个 UPrimitiveComponent 被注册到世界（RegisterComponent / Actor 的 BeginPlay 期）时，游戏线程调用：

// 游戏线程（PrimitiveComponent.cpp 节选，逻辑简化）
World->Scene->AddPrimitive(this);   // FSceneInterface::AddPrimitive，this = UPrimitiveComponent*

FScene::AddPrimitive（Scene.cpp）在游戏线程上做两件事：

  1. 调用 Component->CreateSceneProxy()，创建对应的 FPrimitiveSceneProxy 子类实例。
  2. 分配一个新的 FPrimitiveSceneInfo，然后通过 ENQUEUE_RENDER_COMMAND 把它连同 Proxy 一起投到渲染线程：
// FScene::AddPrimitive 末尾（简化）
ENQUEUE_RENDER_COMMAND(AddPrimitiveCommand)(
    [Scene = this, SceneInfo, Proxy](FRHICommandListImmediate& RHICmdList)
    {
        Scene->AddPrimitiveSceneInfo_RenderThread(RHICmdList, SceneInfo);
    });

渲染线程消费这条命令后，AddPrimitiveSceneInfo_RenderThread 把 SceneInfo 插入 FScene::Primitives 数组，为它分配可见性 ID，并调用：

SceneInfo->AddToScene(RHICmdList, bUpdateStaticDrawLists);   // 核心步骤见第 2 节

2AddToScene 与 CacheMeshDrawCommands

FPrimitiveSceneInfo::AddToScene（ScenePrivate.h / Scene.cpp）是注册时最繁重的一步。对于支持静态绘制路径的图元（如 UStaticMeshComponent），它会调用：

Scene->CacheMeshDrawCommands(RHICmdList, {SceneInfo});
// 调用 Proxy->DrawStaticElements → 生成 FMeshBatch
// → FMeshPassProcessor 转换为 FMeshDrawCommand
// → 缓存到 FScene::CachedMeshDrawCommands

缓存的 FMeshDrawCommand 是完整的"一次绘制"描述：PSO、顶点缓冲绑定、着色器绑定——可以在之后每帧不重新生成、直接提交给 RHI。这是 UE 4.22 之后新 Mesh Draw Pipeline 对静态网格的关键优化：注册时付出一次较高的构建成本，换取运行时每帧接近零开销的提交。

静态路径 vs 动态路径的缓存决策

并非所有图元都走静态路径。判据在 FPrimitiveSceneProxy::GetPrimitiveType() 等接口：若 Proxy 声明支持静态绘制（SupportsStaticLighting、ShouldRenderInMainPass 等），则 AddToScene 时缓存；否则每帧走 GetDynamicMeshElements 动态生成。骨骼网格（每帧骨骼矩阵变化）走动态路径；静态网格默认走静态路径。

3变换更新：UpdatePrimitiveTransform

注册后，若图元移动了（如角色携带的物件），游戏线程调用：

// FScene::UpdatePrimitiveTransform（Scene.cpp，游戏线程）
ENQUEUE_RENDER_COMMAND(UpdateTransformCommand)(
    [Proxy, NewLocalToWorld, WorldBounds, LocalBounds](FRHICommandListImmediate&)
    {
        Proxy->SetTransform(NewLocalToWorld, WorldBounds, LocalBounds, ...);
        // 同时更新 SceneInfo 的包围盒，刷新加速结构
    });

变换更新不需要重建 Proxy，只需更新 Proxy 持有的矩阵与包围盒。但如果材质改了、网格资产换了，通常需要销毁并重建 Proxy（游戏线程调用 MarkRenderStateDirty 后，SendAllEndOfFrameUpdates 会触发 RecreateRenderState）。

4ENQUEUE_RENDER_COMMAND 宏机制

ENQUEUE_RENDER_COMMAND（RenderingThread.h）是游戏线程向渲染线程通信的唯一正式通道。它的展开逻辑：把传入的 lambda 包装成一个 TEnqueueUniqueRenderCommandType，投入渲染线程的命令队列（底层是 TaskGraph 任务）。

// 宏展开示意（RenderingThread.h）
ENQUEUE_RENDER_COMMAND(MyCommandName)(
    [CapturedData](FRHICommandListImmediate& RHICmdList)
    {
        // 这段代码在渲染线程上执行，CapturedData 已安全跨线程
    });

几条使用规则：

  • 捕获的数据必须是值拷贝或渲染线程安全的指针（如 FPrimitiveSceneProxy*，渲染线程管理其生命周期）——不能捕获 UObject*，GC 随时回收。
  • 命令按入队顺序执行，不重排——这保证"先注册再更新变换"的顺序不乱。
  • 游戏线程不等待渲染线程执行完才继续——入队即可（异步），这是两线程流水差一帧的原因之一。
不能捕获 UObject 指针

Lambda 在渲染线程上执行时，游戏线程可能已经进行了下一帧甚至触发 GC。UObject 指针可能已经失效。正确做法：只捕获 FPrimitiveSceneProxy*（渲染线程管理生命周期）、基础类型、或者用 TRefCountPtr/TSharedPtr 维护生命周期。这是 UE 跨线程编程的高频陷阱。

5FRenderResource 生命周期：InitRHI / ReleaseRHI

GPU 资源（顶点缓冲、纹理、索引缓冲）通过 FRenderResource 体系管理（RenderResource.h）。关键接口：

  • InitRHI()：在渲染线程上调用，创建 RHI 侧的实际 GPU 对象（如 RHICreateVertexBuffer）。
  • ReleaseRHI()：释放 GPU 资源，由 BeginReleaseResource 触发（同样经 ENQUEUE_RENDER_COMMAND）。

FStaticMeshVertexBuffers、FRawStaticIndexBuffer 等都继承 FRenderResource。Proxy 构造时把顶点工厂（Vertex Factory）与这些缓冲关联；缓冲的 InitRHI 才是真正往 GPU 上传数据的时机。

游戏线程 RegisterComponent → FScene::AddPrimitive → CreateSceneProxy() → 分配 FPrimitiveSceneInfo → ENQUEUE_RENDER_COMMAND(AddPrimitive)

移动物件 → MarkRenderTransformDirty → 帧末 SendAllEndOfFrameUpdates → ENQUEUE_RENDER_COMMAND(UpdateTransform)
渲染线程 消费 AddPrimitive 命令 → AddPrimitiveSceneInfo_RenderThread → AddToScene → CacheMeshDrawCommands（静态路径）

消费 UpdateTransform 命令 → Proxy::SetTransform → 更新 SceneInfo 包围盒
检索练习

CacheMeshDrawCommands 在图元注册时被调用，它缓存的是什么，目的是什么？

完整说出一个 UStaticMeshComponent 从注册到渲染线程建立缓存绘制命令的调用链（含文件锚点）。
游戏线程： UPrimitiveComponent::RegisterComponent → FScene::AddPrimitive（Scene.cpp）→ UStaticMeshComponent::CreateSceneProxy() 产出 FStaticMeshSceneProxy（StaticMeshRender.cpp）→ 分配 FPrimitiveSceneInfo → ENQUEUE_RENDER_COMMAND(AddPrimitive)（RenderingThread.h）。
渲染线程： 消费命令 → FScene::AddPrimitiveSceneInfo_RenderThread → 插入 Primitives 数组 → FPrimitiveSceneInfo::AddToScene → FScene::CacheMeshDrawCommands（调 Proxy->DrawStaticElements → FMeshBatch → FMeshPassProcessor → FMeshDrawCommand → 缓存）。

带走这几条

  • 注册流：游戏线程 FScene::AddPrimitive → 创建 Proxy + SceneInfo → ENQUEUE_RENDER_COMMAND → 渲染线程 AddPrimitiveSceneInfo_RenderThread → AddToScene。
  • AddToScene 时为静态网格调用 CacheMeshDrawCommands，把 FMeshBatch 经 FMeshPassProcessor 编译为可复用的 FMeshDrawCommand，运行时每帧零开销提交。
  • 变换更新：FScene::UpdatePrimitiveTransform 同样经 ENQUEUE_RENDER_COMMAND，只更新 Proxy 矩阵与 SceneInfo 包围盒，不重建 Proxy。
  • ENQUEUE_RENDER_COMMAND（RenderingThread.h）是唯一正式跨线程通道，异步投递，禁止捕获 UObject*。
  • FRenderResource::InitRHI（渲染线程）是 GPU 资源真正创建的时机；ReleaseRHI 是释放时机。
