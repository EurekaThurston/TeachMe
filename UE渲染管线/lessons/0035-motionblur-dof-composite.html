Part 6 · 后处理 · Lesson 0035

动态模糊、景深与最终合成

动态模糊消费速度缓冲沿运动方向做定向模糊；Diaphragm DOF 用散景（Bokeh）gather 核模拟光圈失焦；最终上采样把内部分辨率恢复到输出分辨率；UI 合成叠在最顶层。这是后处理链在 Present 之前的最后一段。

时长 ~22 分钟 前置 0031（后处理概述）、0032（TAA / 速度缓冲） 产出 能解释动态模糊的速度缓冲消费方式、Diaphragm DOF 的 gather 核原理、上采样时机，以及 UI 合成如何收尾后处理链

在 TAA、泛光、曝光、色调映射这四大核心 Pass 之外，动态模糊和景深是两个"可选但高影响力"的后处理效果——它们模拟的是相机镜头的两种物理特性：快门打开期间传感器记录的运动轨迹（动态模糊），以及镜头焦平面以外物体因弥散圆（Circle of Confusion）产生的散焦（景深）。理解它们的实现有助于排查常见的画面问题（模糊拖尾、散景形状异常等）。

1动态模糊：消费速度缓冲的定向模糊

动态模糊 Pass（PostProcessMotionBlur.cpp，着色器 PostProcessMotionBlur.usf）工作在 HDR 线性域，在 TAA 之后、泛光/曝光之前运行。

速度缓冲回顾

动态模糊的数据来源是 Base Pass 写入的 Velocity GBuffer（与 TAA 相同，见 0032），它存储每个像素的屏幕空间速度矢量（像素/帧）。相机运动引起的速度可以从 View 矩阵差分计算；物体运动（骨骼动画、物体旋转）产生的速度由 VS 计算并写入。

速度展平（Velocity Flatten）：对每个 N×N 的屏幕分块（tile），找出该块内速度最大的像素，生成一张低分辨率的"最大速度贴图"。这样在后续模糊步骤中，每个 tile 内的像素都能知道该区域最大速度，避免运动物体边界处的欠模糊（刀削边）。

模糊重建（Blur Reconstruction）：在全分辨率的像素着色器里，对每个像素沿其速度方向做多 tap 加权采样：

// PostProcessMotionBlur.usf — 运动方向模糊（示意）
float2 PixelVelocity = VelocityTexture.Sample(Sampler, UV).xy;
// 速度向量决定采样方向和范围（单位：像素）
float  BlurRadius = length(PixelVelocity) * MotionBlurAmount;
float3 AccumColor = 0;
float  AccumWeight = 0;

for (int i = 0; i < NumSamples; i++)
{
    // 在速度方向上均匀分布采样偏移
    float  t = ((float)i / (NumSamples - 1)) - 0.5;
    float2 SampleUV = UV + PixelVelocity * t;
    float3 SampleColor = SceneColorTexture.SampleLevel(Sampler, SampleUV, 0).rgb;
    // 可选：按与中心距离做高斯权重
    AccumColor  += SampleColor;
    AccumWeight += 1.0;
}
float3 Result = AccumColor / AccumWeight;

采样数（r.MotionBlurSampleCount，默认 4–12）和强度（MotionBlurAmount，0–1）控制质量与性能。采样数越多，快速运动物体的模糊越平滑（无条纹），但 PS 开销线性增长。

动态模糊的遮挡问题与深度感知采样

沿速度方向采样时，运动物体的边界附近会出现"穿越遮挡"：采样到被运动物体临时遮挡的背景区域，产生不正确的模糊。UE 4.26 的实现用深度比较（采样点的深度 vs. 中心像素深度）来权重调节——背景采样点权重低，前景采样点权重正常——减少这类 artifact。

2Diaphragm DOF：光圈聚散焦散景模拟

UE 4.26 的主要景深实现是 Diaphragm DOF（光圈景深，代码在 DiaphragmDOF.cpp，着色器在 DiaphragmDOF/ 目录下）。它模拟真实摄影光圈的散焦散景（Bokeh），比早期的圆形高斯 DOF 视觉上更接近真实镜头。

弥散圆（Circle of Confusion）

散焦的核心概念是弥散圆（CoC，Circle of Confusion）：焦平面上的一个点在传感器上是一个点，离焦面越远，这个点就扩散成越大的圆盘（直径 = CoC 半径）。UE 4.26 先计算每个像素的 CoC 大小：

// CoC 计算（示意）
// FocalDistance: 焦距（cm），FocalRegion: 清晰区域宽度
// SceneDepth: 像素的线性视图空间深度
float DepthDiff = SceneDepth - FocalDistance;
// 线性 CoC（正=前景失焦，负=背景失焦）
float CoCRadius = (DepthDiff / FocalRegion) * MaxBokehRadius;
CoCRadius = clamp(CoCRadius, -MaxBokehRadius, MaxBokehRadius);

CoC 半径以像素为单位，值越大散焦越模糊。正值通常代表前景（近于焦平面），负值代表背景（远于焦平面），但取决于约定。

Gather-Based DOF：以输出像素为中心"聚集"采样

Diaphragm DOF 采用 gather（聚集）方式——对输出的每个像素，从其周围 CoC 范围内收集多个输入样本，模拟真实光圈将周围光汇聚的过程：

关键洞见 · Gather vs. Scatter

Scatter 方式是：对每个输入像素，把它的颜色"散播"到 CoC 范围内的所有输出像素（类似粒子渲染）。这在 GPU 的光栅化管线里效率低（写入多个输出位置），且容易产生覆盖 artifact。Gather 方式反过来：对每个输出像素，从输入图像的 CoC 范围内"聚集"多个采样点，用加权平均产生散焦效果。Gather 的并行性完美适合 GPU 的像素着色器模型，每个输出像素独立计算，没有写入冲突。Diaphragm DOF 的核心就是在 gather 核中模拟真实光圈形状（圆形、六边形等散景形状）。

Diaphragm DOF 的四阶段实现

这个系统把景深分为前景和背景分别处理（因为两者的遮挡关系不同），完整链如下：

下采样 + CoC半分辨率提取
计算每像素 CoC
›
前景 GatherCoC>0：前景散景
光圈形状核采样
›
背景 GatherCoC<0：背景散景
分层处理遮挡
›
重建合成前景+背景+清晰区
按 CoC 混合回全分辨率

前景和背景分开处理的原因：前景物体（近于焦平面）的散焦效果会覆盖在其背后物体之上，而背景物体的散焦会在前景清晰物体边缘产生"渗出（Bleed）"效果。简单地用同一个 gather 核会导致两者混乱，必须按遮挡层次分开处理，再在重建阶段合成。

Diaphragm DOF 在 ScreenPercentage < 100 时的精度损失

DOF 在内部渲染分辨率（ScreenPercentage 指定的尺寸）下工作。如果 ScreenPercentage = 50%，DOF 的 gather 核是在半分辨率 SceneColor 上操作，散景形状会损失精度（边缘锯齿，特别是多边形光圈形状）。这与 TAA 后置的上采样（TAAU）配合时尤其明显。权衡：低 ScreenPercentage 节省带宽，但 DOF 和 TAA 的质量下降；高质量影视场景需要 100% 以上。

3空间上采样：恢复输出分辨率

如果 r.ScreenPercentage < 100，整个后处理链（TAA、DOF、动态模糊）都在低分辨率下工作。在色调映射附近（具体位置取决于 AA 模式），需要做空间上采样（Spatial Upscale）把图像恢复到最终输出分辨率。

UE 4.26 的主要上采样方案：

  • Lanczos 上采样：标准的高质量空间滤波，适用于大部分 SDR 场景；
  • Catmull-Rom 上采样：略锐化，高对比度边缘清晰；
  • TAAU（Temporal Upsampling）：把 TAA 的历史累积与上采样结合，用时间维度补偿空间分辨率损失，对低 ScreenPercentage 效果最好（需开启 r.TemporalAA.Upsampling = 1）。

上采样的位置关键：它在色调映射之后或之前都有可能，取决于引擎配置。UE 4.26 默认 Secondary Screen Percentage（二级屏幕百分比）在色调映射之后、UI 合成之前执行上采样——这保证了 UI 始终在全分辨率渲染，不受 ScreenPercentage 影响。

Primary vs. Secondary Screen Percentage

Primary Screen Percentage（r.ScreenPercentage）控制 GBuffer 和大部分后处理 Pass 的内部分辨率，是主要的分辨率缩放旋钮。Secondary Screen Percentage（r.SecondaryScreenPercentage.GameViewport）是一个额外的上采样比例，在主上采样之后、UI 合成之前应用——它主要用于某些需要在不同分辨率下渲染不同阶段的特殊场景（如 VR）。大多数游戏只需关注 Primary。

4UI/HUD 合成：Slate 渲染与叠加

色调映射和上采样完成后，SceneColor 是 LDR 全分辨率图像，准备接受 UI 叠加。UE 的 UI 系统 Slate 在独立的渲染路径里绘制 HUD、菜单、按钮等 2D 元素，最后以一次 Blit 或混合 Pass 叠加到场景图像上。

UI 渲染的几个关键特点：

  • UI 在全分辨率渲染，不受 ScreenPercentage 影响——这正是上采样在 UI 合成之前完成的原因；
  • UI 渲染的像素格式通常是 PF_B8G8R8A8，它使用标准 alpha 混合叠加到 SceneColor；
  • Slate 的绘制命令由游戏线程生成，在渲染线程通过 FSlateDrawBuffer 传入，最终通过 RHI 绘制到后缓冲；
  • 这一步是整条渲染管线里最后一个写后缓冲的步骤，之后是 Present。

5后处理链的完整闭环：Present 之前的最后一步

下面是从 TAA 到 Present 的完整后处理链总视图（含本课涉及的 DOF 和动态模糊的位置）：

渲染线程
TAA（速度缓冲 → 历史累积） → 动态模糊（速度缓冲 → 定向模糊，可选） → Diaphragm DOF（CoC → gather 散景，可选） → 泛光（下/上采样金字塔，产生泛光纹理） → 自动曝光（直方图 → 曝光缓冲） → CombineLUTs（颜色参数 → 3D LUT） → 色调映射（SceneColor×Exposure + Bloom → LUT → LDR） → 空间上采样（内部分辨率 → 输出分辨率） → UI/HUD 合成（Slate 叠加） → Present（RHI 提交交换链）

检索练习

Diaphragm DOF 采用 gather 方式而非 scatter 方式的主要原因是什么？

说出 Diaphragm DOF 的四个处理阶段（下采样+CoC计算、前景 gather、背景 gather、重建合成），以及为什么前景和背景要分开处理。
  1. 下采样 + CoC 计算：把 SceneColor 下采样到半分辨率，对每个像素根据其深度与焦距的差计算 CoC 半径（正=前景，负=背景）。这一步奠定后续 gather 的"模糊半径"数据。
  2. 前景 Gather（CoC > 0）：对近于焦平面的像素，在其 CoC 范围内以光圈形状核（圆形或多边形）聚集采样，产生前景散焦效果。
  3. 背景 Gather（CoC < 0）：对远于焦平面的像素，同样做 gather，但需要处理被前景清晰物体"遮住"的区域——背景的散焦光晕不应该穿透前景清晰边缘。
  4. 重建合成：把前景散焦层、背景散焦层和清晰中间层按 CoC 权重混合，上采样回全分辨率，输出最终带景深的 SceneColor。

为什么分开：前景物体（近）会覆盖在背景散焦光晕之上；背景散焦光晕在前景清晰物体的边缘会产生"渗出"现象（前景不透明但背景光晕扩散到其轮廓外）。如果用同一个 gather 核处理，前景遮挡和背景渗出两种情况会互相干扰，产生错误的颜色混合。分层处理后在重建阶段按遮挡关系合成才能正确。

带走这几条

  • 动态模糊（PostProcessMotionBlur.cpp）消费 Velocity GBuffer，先提取 tile 最大速度，再在全分辨率沿速度方向做定向多 tap 采样；工作在 HDR 线性域，在 TAA 之后、泛光之前。
  • Diaphragm DOF：计算每像素 CoC 半径（深度差/焦距范围），用 gather 方式分前景/背景分别在散景形状核内聚集采样，最后重建合成。Gather 天然适合 GPU 的输出像素并行模型。
  • 空间上采样在色调映射后、UI 合成前执行，把内部渲染分辨率（ScreenPercentage）恢复到输出分辨率，保证 UI 始终在全分辨率。
  • UI/HUD 合成（Slate）是后处理链最后一步，在全分辨率 LDR SceneColor 上以 alpha 混合叠加，之后即 Present 到交换链。
