RDG 入门：为什么要图、如何构图

RDG（Render Dependency Graph）把整帧 Pass 的录制和执行拆成两个阶段：先"描述"、后"执行"。这个延迟模型让引擎得以自动化一切屏障、资源分配和异步计算调度。理解这一对比，是读懂 4.26 之后任何渲染代码的门槛。

时长 ~25 分钟 前置 0003（资源屏障）、0014（Render() 总览） 产出 能说出 RDG 的"延迟录制→编译→执行"模型，并读懂一个最小 Pass 的声明

0003 讲了手动管理资源屏障的痛苦——每次资源用途改变（RT → ShaderReadOnly）都要精确插入 ResourceBarrier，漏插崩溃、多插性能损耗。RDG 的设计目标正是：让你描述"Pass 读写什么资源"，由图编译器自动推导并放置所有屏障。这不是语法糖，是根本性的抽象层变换。

1命令式 vs 声明式：为什么需要"图"

命令式（旧方式）Pass A → 手动插屏障(A→B) → Pass B → 手动插屏障(B→C) → Pass C → … 每个屏障由程序员在代码里精确放置，错位即 GPU 乱序读脏数据或 TDR。Pass 引用但实际不需要的资源也无法自动跳过。
声明式（RDG）向 FRDGBuilder 声明 Pass：Pass 读哪些资源、写哪些资源。RDG 在 Execute() 编译时：① 推导所有屏障位置；② 合并相邻屏障；③ 推导资源生命周期；④ 对生命周期不重叠的资源做内存别名；⑤ 剔除无引用的 Pass。程序员只描述"做什么"，RDG 决定"怎么安全高效地做"。
RDG 的三步模型

1. 构图阶段（Recording）：渲染线程调用 GraphBuilder.AddPass() 把 Pass 描述加入图，此时 lambda 不运行，真实 GPU 命令不录制。2. 编译阶段（Compile）：GraphBuilder.Execute() 被调用，RDG 遍历图，推导依赖关系、资源生命周期、屏障位置、Pass 剔除、内存别名。3. 执行阶段（Execute）：按拓扑序依次调用各 Pass 的 lambda，向 FRHICommandList 录制真实命令，最终提交。

2核心 API 逐一拆解

2.1 FRDGBuilder

定义在 Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h。构造时绑定一条 FRHICommandListImmediate：

FRDGBuilder GraphBuilder(RHICmdList);

GraphBuilder 的生命周期就是"描述整帧所有 Pass"的窗口。析构或 Execute() 时结束构图、触发编译执行。

2.2 CreateTexture / CreateBuffer

// 声明一张临时纹理——此刻不分配真实显存
FRDGTextureDesc Desc = FRDGTextureDesc::Create2D(
    FIntPoint(Width, Height),
    PF_FloatRGBA,
    FClearValueBinding::Black(),
    TexCreate_RenderTargetable | TexCreate_ShaderResource);

FRDGTextureRef SceneColorTexture =
    GraphBuilder.CreateTexture(Desc, TEXT("SceneColor"));

FRDGTextureRef 是一个虚拟句柄（类似"图里的变量名"），不持有真实 FRHITexture*。真实显存在 Execute 的编译期才根据生命周期分配（或从瞬态池复用）。

2.3 BEGIN_SHADER_PARAMETER_STRUCT

每个 Pass 的输入输出用结构体宏声明：

BEGIN_SHADER_PARAMETER_STRUCT(FMyPassParameters, )
    SHADER_PARAMETER_RDG_TEXTURE(Texture2D, InputTexture)
    SHADER_PARAMETER_RDG_TEXTURE_UAV(RWTexture2D<float4>, OutputTexture)
    RENDER_TARGET_BINDING_SLOTS()
END_SHADER_PARAMETER_STRUCT()

宏里每个 RDG_TEXTURE_ACCESS、RDG_TEXTURE_UAV 都告诉 RDG "这个 Pass 对这个资源的访问类型"。RDG 据此推导：这个资源在这个 Pass 之前应处于什么状态、之后变成什么状态——屏障就这样自动算出来。

2.4 AddPass

FMyPassParameters* Parameters =
    GraphBuilder.AllocParameters<FMyPassParameters>();
Parameters->InputTexture  = InputTexRef;
Parameters->OutputTexture = GraphBuilder.CreateUAV(OutputTexRef);

GraphBuilder.AddPass(
    RDG_EVENT_NAME("MyPass"),
    Parameters,
    ERDGPassFlags::Compute,
    [Parameters, LocalData](FRHICommandList& RHICmdList)
    {
        // 这里才真正录制 RHI 命令——Execute() 时才运行
        TShaderMapRef<FMyComputeShader> ComputeShader(GetGlobalShaderMap(/* FeatureLevel */));
        SetComputePipelineState(RHICmdList, ComputeShader.GetComputeShader());
        SetShaderParameters(RHICmdList, ComputeShader, ComputeShader.GetComputeShader(), *Parameters);
        DispatchComputeShader(RHICmdList, ComputeShader, GroupsX, GroupsY, 1);
    });
AddPass 的 lambda 里不能访问 RDG 资源的 RHI 句柄——除非用 GetPooledRDGTexture 先解包

在 AddPass 的 lambda 内，FRDGTextureRef 虚拟句柄已被 RDG 解析成真实 FRHITexture* 并绑定到 Parameters 里——直接使用 Parameters 里的字段即可。不要在 lambda 里对 FRDGTextureRef 直接调用 RHI 方法，那是构图阶段的对象，Execute 时它的"意义"已完成。

2.5 Execute

GraphBuilder.Execute();
// 之后 GraphBuilder 析构，所有 RDG 资源句柄失效

Execute() 做三件事：① 拓扑排序 + Pass 剔除；② 推导资源生命周期 + 分配/别名真实显存；③ 插入所有屏障并按序调用每个 Pass 的 lambda 向 RHICmdList 录制命令。从 Execute 返回后，命令已录入 FRHICommandListImmediate，等待 RHI 线程提交。

3一个最小完整示例：SceneColor 后处理 Pass

// 构图阶段（在 Render() 或某个 AddXxxPasses 函数里）
FRDGBuilder GraphBuilder(RHICmdList);

// 声明输入输出纹理
FRDGTextureRef SceneColor = GraphBuilder.RegisterExternalTexture(
    GSceneRenderTargets.GetSceneColor());   // 把已有 RHI 纹理注册进图

FRDGTextureDesc OutDesc = SceneColor->Desc;
OutDesc.Flags |= TexCreate_UAV;
FRDGTextureRef OutTexture = GraphBuilder.CreateTexture(OutDesc, TEXT("PP_Out"));

// 填充参数
auto* Params = GraphBuilder.AllocParameters<FMyPostProcessPassParams>();
Params->InputTexture  = SceneColor;
Params->OutputTexture = GraphBuilder.CreateUAV(OutTexture);

// 声明 Pass（此时不运行 lambda）
GraphBuilder.AddPass(
    RDG_EVENT_NAME("MyPostProcess"),
    Params,
    ERDGPassFlags::Compute,
    [Params](FRHICommandList& RHICmdList)
    {
        // Execute() 时才运行，此处是真正录制 GPU 命令
        // ... Dispatch compute shader ...
    });

GraphBuilder.Execute();   // 编译+执行整图
检索练习

调用 GraphBuilder.CreateTexture() 时，GPU 显存实际在何时分配？

不看笔记，解释 RDG 的三步模型（Recording、Compile、Execute），并说明 FRDGTextureRef 是什么、什么时候变成真实 FRHITexture*。

Recording：渲染线程调用 CreateTexture、AddPass 等 API，向 FRDGBuilder 描述本帧所有 Pass 及其输入输出资源。此时 lambda 不运行，GPU 命令不录制，显存不分配。FRDGTextureRef 是图里的"逻辑变量"，不持有真实显存句柄。

Compile：Execute() 被调用，RDG 遍历图，推导每个资源的生命周期（首次写→最后一次读），对不重叠的资源做内存别名，推导所有屏障插入点，剔除无引用的 Pass。这时 FRDGTextureRef 被解析为真实的 FRHITexture*（从瞬态池分配或别名复用）。

Execute：按拓扑序依次调用每个 Pass 的 lambda，向 FRHICommandList 录制真实 GPU 命令（Dispatch/Draw）以及编译期确定的屏障。录制完后提交给 RHI 线程。

带走这几条

  • RDG（RenderCore/Public/RenderGraphBuilder.h）的核心：先描述、后执行——构图不录命令，Execute() 时才编译图、分配显存、插屏障、录命令。
  • FRDGTextureRef / FRDGBufferRef 是虚拟句柄，编译期才解析为真实 FRHITexture*——这是内存别名的前提。
  • BEGIN_SHADER_PARAMETER_STRUCT 里声明的 RDG 资源访问告诉 RDG"谁读谁写"——屏障位置由此自动推导，呼应 0003 手动屏障之痛。
  • AddPass(name, params, flags, lambda)：flags 指定 Pass 类型（Raster/Compute/AsyncCompute/Copy），lambda 在 Execute 阶段才运行。
  • GraphBuilder.Execute()：拓扑排序 + 资源分配 + 屏障 + 按序执行 lambda，是整帧 GPU 工作的真正"起跑枪"。
