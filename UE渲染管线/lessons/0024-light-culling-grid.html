光照剔除：Light Grid / 聚簇

场景里可能有几百盏动态点光和聚光，但每个像素只受其中几盏影响。Light Grid 用计算着色器把光源分配进屏幕 3D 网格，着色时按像素位置 O(1) 取回光源列表，把光照循环从"所有光"压缩到"相关光"。

时长 ~15 分钟 前置 0002（计算着色器 / GPU 并行）、0023（BasePass / GBuffer） 产出 能解释为什么要把光源分配进屏幕 3D 网格，以及它服务于哪些着色路径

延迟着色里，方向光全屏跑一次 pass 即可；但局部光（点光、聚光）每盏都有影响范围，若对每盏光遍历所有可见像素代价太高。另一个极端是在 PS 里对所有光循环并跳过不相关的——条件分支让 warp 里的线程分歧，浪费 GPU 利用率。Light Grid 的思路：在着色之前，用一个独立的计算 Pass 把每盏光的包围体投影到屏幕 3D 网格，把光源 ID 写进对应的簇（cluster）。着色时，任何像素只需查自己所在的簇，取到一个短列表——O(1) 查找，无分歧，无浪费。

1ComputeLightGrid：入口与数据结构

计算 Light Grid 的入口是 ComputeLightGrid（LightGridInjection.cpp）。它在 RDG 里注册一个计算 Pass，用 LightGridInjection.usf 里的 compute shader 执行以下步骤：

  1. 划分网格：把视图空间深度范围按对数切片分成若干 Z 层（froxel = frustum voxel），屏幕 XY 方向按固定 tile 大小（默认 64px）划格，得到三维网格。
  2. 光源注入：对每盏局部光，计算其在视图空间中覆盖哪些格（包围球/锥体与 froxel AABB 相交测试），把光源 ID 写入这些格的光源链表。
  3. 构建索引缓冲：把每个格的光源列表压缩成连续的索引数组，并在每个格的头部存（offset, count）。着色时按 offset/count 随机访问索引数组，取出光源 ID。
// LightGridInjection.cpp（架构示意）
ComputeLightGrid(GraphBuilder, Scene, Views, ...);
// 输出：
//   ForwardLightingViewResources.ForwardLocalLightBuffer  — 光源参数数组
//   ForwardLightingViewResources.NumCulledLightsGrid      — 每格光源数
//   ForwardLightingViewResources.CulledLightDataGrid      — 光源索引列表
// 打包到 FForwardLightData uniform buffer，绑定给多个 Pass
froxel 的 Z 分层为什么用对数

透视投影下，近处深度范围密集、远处稀疏。线性分层会把大量 Z 层堆在近处而远处过于粗糙。对数分层让每层在视图空间里的"厚度"随深度增加，视觉上均匀分布，兼顾近处精度与远处覆盖。cvar r.Forward.LightGridSizeZ 控制 Z 层数（默认 32），r.Forward.LightGridPixelSize 控制 XY tile 大小（默认 64）。

2Light Grid 服务哪些路径

前向 / 聚簇着色前向路径（Forward Renderer）或延迟路径里的"Clustered Deferred"（r.* cvar 开启）：PS 直接查 Light Grid，在一个 Pass 内对所有相关局部光着色，无需逐光 Pass。
半透明光照（0029）半透明物体无法写 GBuffer，直接用 Light Grid 在前向 PS 里查询影响自身的光源，实现多光照射。
体积雾（0030）体积雾的计算 Pass 按 froxel 逐格求光照贡献，直接复用 Light Grid 的分层结构。
纯延迟直接光（0025）纯延迟路径（RenderLights）仍按光源逐个画全屏/包围体 Pass，不靠 Light Grid——但 Light Grid 数据对调试和混合路径仍有参考价值。

重点：Light Grid 是为前向/半透明/体积路径准备的共享资源，并非延迟直接光照的必要依赖。纯延迟路径里，RenderLights（0025）用模板+裁剪自行限制每盏光的影响区域，无需 Grid。

聚簇着色 vs. 分块着色（Tiled vs. Clustered）

分块延迟（Tiled Deferred，2D tile + 深度范围裁剪）是更早的方案，Light Grid 是其 3D 扩展（Clustered Deferred / Clustered Forward）。3D 版本在深度不连续（窗口、走廊）的场景里更准确：2D tile 若包含多个深度层（如近处遮挡物和远处窗外）会把所有层的光都纳入，而 3D froxel 精确区分深度层，不会把远处的灯光错误分配给前景像素。

检索练习

UE 4.26 的纯延迟直接光照 Pass（RenderLights）是否直接依赖 Light Grid 来决定哪些像素需要着色？

不看图，解释 Light Grid 的三个构建步骤（划分→注入→索引），并说出它为什么在前向着色 PS 里能做到 O(1) 查询。
① 划分：把视图空间按对数 Z 切片 + 屏幕 XY tile 划成三维网格（froxel）。② 注入：Compute Shader 对每盏局部光做包围体与 froxel 的相交测试，把光源 ID 写入所有相交格的链表。③ 索引：把每格链表压缩成连续数组，每格头部存（offset, count）。O(1) 的原因：着色时，任何像素根据自己的屏幕坐标和深度，直接计算出所在 froxel 的下标，用 offset 直接跳到光源索引数组对应位置，读 count 个条目——整个过程是固定代价的随机访问，不受场景总光源数影响。

带走这几条

  • Light Grid 把光源的影响范围预分配进屏幕 3D froxel 网格，着色时 O(1) 查表取相关光源列表，消除"全光源循环"的分歧开销。
  • 入口：ComputeLightGrid（LightGridInjection.cpp）；输出绑定为 FForwardLightData uniform buffer。
  • 主要服务：前向着色（主路径）、半透明光照（0029）、体积雾（0030）；纯延迟直接光照（0025）仍按光源逐 Pass，不依赖 Grid。
  • 关键 cvar：r.Forward.LightGridSizeZ（Z 层数）、r.Forward.LightGridPixelSize（tile 大小）、r.Forward.MaxCulledLightsPerCell。
  • 3D 聚簇比 2D 分块更准确：froxel 精确区分深度层，避免不同深度的光源相互污染。
