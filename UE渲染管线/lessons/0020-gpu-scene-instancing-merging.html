GPU Scene、实例化与合批

FGPUScene 把图元的变换与包围盒常驻 GPU 结构化缓冲，着色器按 PrimitiveId 索引——绕过每帧 CPU 上传。在此基础上，绑定相同的多条 FMeshDrawCommand 可合并为一次实例化绘制，大幅降低 CPU 提交开销。

时长 ~18 分钟 前置 0018（FMeshDrawCommand）、0019（合批基础）、0002（GPU 并行性） 产出 能解释 GPU Scene 把图元数据常驻 GPU 的意义，以及绘制命令合批/自动实例化如何降低 CPU 开销

传统做法是每帧把每个图元的变换矩阵从 CPU 上传到 GPU（SetShaderParameter 或更新常量缓冲）——物体越多，CPU 带宽和调用开销越高。FGPUScene（UE 4.23 引入，4.26 逐步成熟）改变了这一点：图元数据一次写进 GPU 端的结构化缓冲（PrimitiveSceneData），只在数据实际变化时增量更新，着色器用 PrimitiveId 索引——消除了大量重复上传。合批（Draw Call Merging）则从另一方向压缩开销：多个绑定一致的绘制命令合成一次 instanced draw。

2FGPUScene 的数据布局

FGPUScene（GPUScene.cpp / GPUScene.h）维护一个 GPU 端的结构化缓冲 PrimitiveSceneData，每个图元占固定大小的槽（当前为一组 float4x4 矩阵 + 包围盒 + 标志位等，共约 36 个 float4）。代码里通过 GPUScene.PrimitiveBuffer（FRDGBuffer）在 RDG 里传递，着色器端定义为：

// SceneData.ush（Engine/Shaders/Private/）
StructuredBuffer<float4> GPUSceneInstanceSceneData;
StructuredBuffer<float4> GPUScenePrimitiveSceneData;

// 顶点着色器里读取当前图元变换（PrimitiveId 来自 DrawInstanced 的实例 ID）
FPrimitiveSceneData PrimitiveData = GetPrimitiveData(PrimitiveId);
float4x4 LocalToWorld = PrimitiveData.LocalToWorld;

着色器端通过 PrimitiveId 从结构化缓冲取变换，无需单独常量缓冲

3增量更新机制

每帧游戏线程把发生变化的图元标记为 Dirty（移动、材质变更、新增/删除）。FGPUScene::Update 在渲染线程收集所有 Dirty 图元，通过 Compute Shader 或直接写操作把更新数据批量上传，覆盖对应槽位。未变化的图元数据一帧都不碰。

相关 cvar：

  • r.GPUScene.UploadEveryFrame — 调试用，强制每帧全量上传（验证增量是否正确）
  • r.GPUScene.MaxPooledUploadBufferSize — 控制上传缓冲池大小
  • r.GPUScene.UseGPUSceneForDynamicPrimitives — 是否对动态图元也走 GPU Scene 路径（4.26 部分支持）

4合批与自动实例化：从命令到一次 Draw

当多个 FMeshDrawCommand 满足以下条件时，SubmitMeshDrawCommands 会把它们合并为一次 DrawIndexedPrimitiveInstanced：

  • 相同的 PSO（着色器 + 混合 + 光栅状态）
  • 相同的顶点流与索引缓冲（同一静态网格资源）
  • 相同的着色器绑定（除 PrimitiveId 外）——依赖 GPU Scene 使得所有图元共享同一 PrimitiveSceneData 缓冲

合并条件中，"相同着色器绑定"在传统常量缓冲模式下几乎不可能满足（每个物体的变换都是独立绑定）；GPU Scene 把变换移进结构化缓冲后，绑定变得一致，合批变成可能。

GPU Scene 使合批成为系统性能力而非特例

旧版本里，合批需要美术手动把多个 Mesh 打包成一个（Mesh Merge），有局限。GPU Scene + PrimitiveId 方案让任意相同静态网格的多个实例在 Mesh Draw Pipeline 里自然满足合批条件，无需特殊处理。这是 Draw Call Merging / Instancing 从"技巧"变成"默认机制"的关键原因。

6UE5 的演进方向（超纲标注）

UE 4.26 的 GPU Scene 以动态实例化合批为主，CPU 可见性剔除结果才决定哪些图元参与绘制。UE5 在此基础上引入了 Instance Culling（实例化 GPU 端剔除）和 Nanite（GPU 端完全接管几何提交与可见性）——这些在 UE 4.26 中均不存在，属超纲内容，此处仅作演进方向说明。本课程严格锁 4.26，以动态实例化合批为终点。

检索练习

GPU Scene 让合批（自动实例化）变得容易的根本原因是什么？

解释"GPU Scene 常驻" vs "传统 CPU 上传"在每帧 CPU 开销上的区别，以及合批能发生的三个必要条件。
CPU 开销对比：传统方式每帧对所有可见图元调用 SetShaderParameter 上传变换，O(可见图元数)；GPU Scene 只在图元数据实际变化时增量更新脏数据，O(脏图元数)——场景越静态，节省越多。

合批三条件：① 相同 PSO（着色器+混合+光栅状态）；② 相同顶点流和索引缓冲（同一 Mesh 资源）；③ 相同着色器绑定（依赖 GPU Scene——变换已在共享结构化缓冲中，绑定无差异，只有 PrimitiveId 不同，而 PrimitiveId 用实例 ID 传递，不需要单独绑定）。

带走这几条

  • FGPUScene（GPUScene.cpp）把所有图元的变换/包围盒等数据常驻 GPU 结构化缓冲 PrimitiveSceneData，着色器通过 PrimitiveId 索引——绕过每帧全量 CPU 上传。
  • 增量更新：只有 Dirty 图元才走上传路径，静态场景的 GPU Scene 更新开销接近零。
  • 合批需要：PSO 相同 + 顶点流相同 + 着色器绑定相同。GPU Scene 把变换移进共享缓冲，使"绑定一致"成为常态，让合批从特例变成默认机制。
  • 合批后 VS 用 SV_InstanceID 作 PrimitiveId 读取对应变换——N 个物体一次 draw call，GPU 并行 N 个实例。
  • UE5 的 Instance Culling 和 Nanite 是 GPU Scene 的演进方向，在 UE 4.26 中不存在，属超纲。
