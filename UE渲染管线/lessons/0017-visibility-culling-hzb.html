可见性与剔除：视锥、距离、遮挡(HZB)

InitViews 是渲染线程最重要的准备阶段：它决定哪些图元会被画、会被送到哪些 Pass。四层剔除逐步过滤，最终结果写进 FViewInfo::PrimitiveVisibilityMap。剔除做得好，后续所有 Pass 都省力。

时长 ~20 分钟 前置 0005（帧生命周期）、0011（FScene 与代理）、0004（CPU/GPU 回读延迟） 产出 能说出 InitViews 里四层剔除的顺序与原理，理解遮挡查询为何延迟一帧

FDeferredShadingSceneRenderer::InitViews（SceneVisibility.cpp）在每帧 Render() 的最开头调用。它的职责是：对场景里的每个 FPrimitiveSceneProxy，判断当前视图是否需要渲染它。被剔除的图元不会生成任何 FMeshBatch，不会走后续 Pass，不占 GPU 时间。这节课把四层剔除的逻辑、顺序、源码入口逐一拆开。

1剔除的总体结构：漏斗模型

四层剔除从粗到细依次执行，每一层只处理上一层"活下来"的图元。越早剔除成本越低——视锥剔除只是 AABB vs 平面的数学，遮挡查询则需要 GPU 渲染一遍低分辨率场景。

最终写入 FViewInfo::PrimitiveVisibilityMap（一个按图元索引的位域）和 StaticMeshVisibilityMap（静态网格专用）。只有置位的图元才会参与后续的 GetDynamicMeshElements 或缓存绘制命令查找。

2第一层：视锥剔除（FrustumCull）

视锥剔除检测图元的包围体是否与相机视锥相交。UE 用轴对齐包围盒（AABB）对视锥的六个平面做快速 SAT（分离轴测试）。代码入口：FSceneRenderer::ComputeViewVisibility → FrustumCull<true, true>(...（SceneVisibility.cpp）。

视锥剔除完全在 CPU 上运行，可以多线程并行（每个 View 独立）。它是成本最低、效果最稳的一层——完全在视锥外的物体，后续所有 GPU 工作全省掉。

为什么用 AABB 而非精确的 Convex Hull

AABB 对平面的 SAT 是 6 次点积 + 6 次比较，可以 SIMD 化，一次批量测试几百个包围盒只需微秒级。精确形状测试精度更高但计算量翻数倍，对于视锥剔除而言过度精确——"包围盒漏网、精确形状其实剔掉"的概率不高，不值得额外开销。真正的精准剔除留给后面的遮挡层。

3第二层：距离剔除（Min/Max Draw Distance）

每个 UPrimitiveComponent 有两个属性：MinDrawDistance 和 LDMaxDrawDistance（关卡细节最大绘制距离）。FPrimitiveSceneInfo 存储它们的平方值，ComputeViewVisibility 把图元到相机的距离平方与之比对，超出范围直接剔除。

距离剔除还承载另一个功能：LOD 选择。FPrimitiveLODLevel 根据距离选出当前应渲染的 LOD，后续 GetDynamicMeshElements 和命令缓存查找都用这个 LOD 结果。距离剔除本质上是 LOD 系统的极端情况（LOD 为"不画"）。

4第三层：预计算可见性（Precomputed Visibility）

UE 的预计算可见性系统（PrecomputedVisibilityHandler）在编辑器中把关卡划分为网格单元（Cell），对每个 Cell 离线计算哪些图元可见，结果编码成压缩的 PVS 位域存进关卡包。运行时只需查找当前相机所在 Cell，直接取出位域做 AND 运算。

PVS 的代价：离线时间换运行时零开销

PVS 查找的 CPU 开销几乎为零，但它依赖关卡几何静止。动态物体（角色、可破坏物体）不能参与 PVS，只能依赖遮挡查询。室内走廊、城堡内部等遮挡关系稳定的场景最适合 PVS——开放世界基本不用，因为 Cell 数量会爆。cvar r.PrecomputedVisibilityWarning 可检查覆盖情况。

5第四层：遮挡剔除——两种机制并存

遮挡剔除是代价最高但效果最强的一层，UE 4.26 有两种机制交替使用：

5a. 硬件遮挡查询（FOcclusionQueryHelpers）

对每个候选图元提交一次 GPU 遮挡查询：用包围盒在已有深度缓冲上做深度测试，统计通过的像素数。关键约束：查询结果必须等 GPU 完成后才能读回 CPU，而这至少需要等到下一帧（呼应 0004 回读延迟）。所以这帧提交的查询，下一帧才用结果决定是否剔除——结果始终落后一帧。

延迟一帧的后果

如果相机快速移动，前一帧被遮挡的物体在新视角下可能已经露出，但查询结果还显示"遮挡"——这导致一帧的鬼影（popping）：物体先消失，下帧重新出现。UE 通过保守估计（扩大查询包围盒、保留缓冲帧数）缓解，但无法完全消除。这是硬件遮挡查询的根本限制，不是 bug。

5b. HZB 遮挡（FHZBOcclusionTester）

Hierarchical Z-Buffer（层级 Z 缓冲）是另一种遮挡思路：从上一帧的深度缓冲生成 mip 链（每级取最大深度），然后把图元包围盒投影到屏幕，在 HZB 的合适 mip 级别上采样一次深度值做比较。整个过程在 GPU 的 Compute Shader 里跑，比硬件查询更并行、延迟更可预测。

HZB 生成入口：FDeferredShadingSceneRenderer::RenderHzb（HZBOcclusion.cpp）。HZB 纹理用上一帧的 SceneDepth 构建，同样有一帧延迟，但因为在 GPU 端执行，回读开销小得多。

6相关性计算：FPrimitiveViewRelevance

通过所有剔除层的图元还需要做相关性计算（FPrimitiveViewRelevance）：FPrimitiveSceneProxy::GetViewRelevance 告诉渲染器这个图元需要参与哪些 Pass（深度 Pass？BasePass？半透明？速度缓冲？）。结果存入 FRelevancePacket，决定图元被送到哪些 Pass 的绘制命令队列。

这一步是"可见"与"参与 Pass"之间的桥梁：图元可见不等于它参与所有 Pass，例如纯透明物体不写 GBuffer，不透明的 DepthOnly 材质不写 BasePass。

// FPrimitiveViewRelevance 的关键字段（SceneManagement.h）
struct FPrimitiveViewRelevance
{
    uint32 bDrawRelevance      : 1;  // 这帧要画吗
    uint32 bStaticRelevance    : 1;  // 走静态路径（缓存命令）
    uint32 bDynamicRelevance   : 1;  // 走动态路径（GetDynamicMeshElements）
    uint32 bRenderInMainPass   : 1;  // 进 BasePass / DepthPass
    uint32 bTranslucentRelevance : 1;// 进半透明 Pass
    uint32 bShadowRelevance    : 1;  // 参与阴影投射
    // ... 更多 Pass 标志
};

7结果汇总：PrimitiveVisibilityMap

四层剔除 + 相关性计算完成后，FViewInfo 持有两张关键位图：

字段类型含义
PrimitiveVisibilityMapTBitArray<>通用可见性，按图元全局索引
StaticMeshVisibilityMapTArray<uint64>静态网格专用，用于快速查找缓存绘制命令

后续 FMeshPassProcessor 和缓存命令系统都以这两张位图为索引入口，只处理置位的图元。

检索练习

硬件遮挡查询的结果为什么总是延迟一帧才能用？

不看图，按顺序说出 InitViews 里四层剔除的名称、执行位置（CPU/GPU）、结果时效，以及最终结果写进哪个数据结构。
① 视锥剔除（CPU，FrustumCull，即时）——AABB vs 视锥六平面。
② 距离剔除（CPU，ComputeViewVisibility，即时）——MinDrawDistance/LDMaxDrawDistance 平方比较。
③ 预计算可见性（CPU，PrecomputedVisibilityHandler，即时）——离线 PVS 位域 AND 运算。
④ 遮挡剔除（GPU，FOcclusionQueryHelpers + FHZBOcclusionTester，延迟一帧）——硬件遮挡查询统计通过像素数；HZB 用上帧深度 mip 链比较包围盒深度。
最终结果写入 FViewInfo::PrimitiveVisibilityMap（通用）和 StaticMeshVisibilityMap（静态网格）。

带走这几条

  • InitViews 的核心函数是 FDeferredShadingSceneRenderer::ComputeViewVisibility（SceneVisibility.cpp），四层剔除在此串联。
  • 剔除漏斗从粗到细：视锥→距离→PVS→遮挡，每层只处理上层存活的图元，越早剔除越省。
  • 硬件遮挡查询和 HZB 都依赖上一帧结果，天然延迟一帧——这是 GPU 回读延迟（0004）的直接体现，会导致快速移动时的一帧 popping。
  • FPrimitiveViewRelevance（GetViewRelevance）在可见性判定之后进一步决定图元参与哪些 Pass——可见不等于进所有 Pass。
  • 最终可见性写入 FViewInfo::PrimitiveVisibilityMap 和 StaticMeshVisibilityMap，后续所有 Pass 以此为入口。
