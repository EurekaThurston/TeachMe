ProfileGPU 与 GPU 时间统计

ProfileGPU 命令和 stat GPU 把 GPU 上每个 Pass 的耗时直接打到屏幕上。这节课教你读懂这些数字的含义，理解 UE 4.26 内部的计时机制（FRealtimeGPUProfiler、RDG_GPU_STAT_SCOPE），以及如何把时间开销归因到之前学过的具体管线阶段。

时长 ~20 分钟 前置 0005（帧全景）、0040（RenderDoc 事件树）、0015/0016（RDG） 产出 能用 ProfileGPU/stat GPU 读出各 Pass 耗时，并将时间热点归因到具体管线阶段

RenderDoc 给你结构——哪个 Pass 输出了什么。GPU Profiler 给你时间——哪个 Pass 花了多少。两者配合才能做完整的性能分析：先用 stat GPU 找最贵的 Pass，再用 RenderDoc（0040）深入那个 Pass 看是哪个 draw call 在拖时间。本节专注于"时间"这一半。

1ProfileGPU 命令与 GPU Visualizer

在 UE 编辑器或 PIE 的控制台（~ 键）输入：

ProfileGPU

引擎会在下一帧采集一次完整的 GPU 时间树，然后弹出 GPU Visualizer 窗口（也叫 GPU Profile 窗口）。这个窗口是一个时间轴视图：横轴是时间（毫秒），每一行是一个 GPU 事件（与 RenderDoc 事件树名称相同，来自同样的 SCOPED_DRAW_EVENT/RDG_GPU_STAT_SCOPE 标记），长度代表 GPU 耗时，颜色按层级区分。

原理 · GPU 时间戳查询

UE 4.26 里，FRealtimeGPUProfiler（定义在 Engine/Source/Runtime/RenderCore/Public/RealtimeGPUProfiler.h，实现在对应 .cpp）在每个被标记的 Pass 前后向命令列表插入一对 GPU 时间戳查询（D3D12 里是 QueryHeap 的 EndQuery(Timestamp)）。GPU 执行到那条命令时写入自己的计数器值；一帧结束后 CPU 读回查询结果，换算成毫秒。这就是"GPU 时间"的来源——不是 CPU 侧的计时，而是 GPU 时钟本身的差值。

注意：GPU 时间戳的精度受 GPU 频率影响（典型精度在微秒级），且 读回需要等 GPU 完成——所以 FRealtimeGPUProfiler 的查询结果默认延迟一帧可用，和遮挡查询同理（见 0004）。

2stat GPU 与持续帧统计

ProfileGPU 是单帧快照。想持续监控 GPU 各 Pass 耗时，用：

stat GPU

这会在屏幕上叠加一个持续刷新的统计面板，列出各个 GPU 事件名及其当前帧耗时（毫秒）。关闭用同一命令再输入一次，或输入 stat none 清除所有统计。

典型输出（数字仅示意）：

// stat GPU 输出示例（括号内为典型中端 GPU 参考值）
[GPU]              12.3 ms
  PrePass           0.8 ms
  BasePass          3.1 ms
  ShadowDepths      1.4 ms
  Lights            2.2 ms
  Translucency      0.6 ms
  PostProcessing    1.8 ms
  // …（其余 Pass 依次列出）

stat GPU 的层级与 GPU Visualizer 一致；各 Pass 耗时之和可能小于总 GPU 时间，因为部分操作未被标记或存在并行执行。

注意 · 并行执行与时间叠加

GPU 可以并行执行图形队列和异步计算队列（见 0016 RDG 异步计算）。stat GPU 显示的是每个 Pass 自身的 GPU 时间，不是墙钟时间叠加。两个并行 Pass 的时间可能"重叠"在同一时钟段——这意味着 stat GPU 各行求和可以大于总帧 GPU 时间，这是正常的，并不是 bug。

3RDG_GPU_STAT_SCOPE 与 Pass 时间归属

上一节介绍的 RDG_EVENT_SCOPE 控制 RenderDoc 里的事件名；而 GPU 时间统计用的是另一个宏：

// 在渲染函数里，声明一个 RDG GPU 统计作用域
RDG_GPU_STAT_SCOPE(GraphBuilder, Foo);
// 或用 DECLARE_GPU_STAT 宏系统（非 RDG Pass 也可用）：
DECLARE_GPU_STAT(Foo)
SCOPED_GPU_STAT(RHICmdList, Foo);

宏定义在 Engine/Source/Runtime/RenderCore/Public/RealtimeGPUProfiler.h。RDG_GPU_STAT_SCOPE 是 RDG-aware 版本，自动与 Pass 生命周期绑定。

DECLARE_GPU_STAT(Foo) 在全局注册一个统计槽（类似 DECLARE_CYCLE_STAT），SCOPED_GPU_STAT/RDG_GPU_STAT_SCOPE 在它前后插入时间戳。这套系统让 stat GPU 能按自定义粒度显示统计。

UE 4.26 里已有大量内置统计项，你在 stat GPU 里看到的所有行名都对应源码里的一个 DECLARE_GPU_STAT 注册。用这些名字在源码里搜索，就能找到相应 Pass 的实现入口。

4读懂 GPU Visualizer：各 Pass 时间含义

打开 GPU Visualizer 后，关注几件事：

4.1 时间块的宽度与嵌套

宽度越大 = 耗时越长。嵌套关系代表 Pass 层级：外层是父作用域（如 BasePass），内层是子 draw call 或子 Pass。如果看到外层 BasePass 很宽，但内部有很多细碎的窄块，说明 draw call 数量多但单次较廉价——CPU-bound 信号（0042 详解）。如果内层少数块特别宽，说明某个着色器或某批物体极贵。

4.2 把时间热点映射回管线阶段

时间最贵的 Pass可能的根因对应课程章节
BasePass材质着色器复杂度高、纹理采样多（带宽）、overdraw（Depth Prepass 未生效）0022（Prepass）、0023（BasePass）
ShadowDepths动态阴影投射物多、多个 CSM 级联、多个点光源阴影0026（阴影）
Lights / DirectLighting动态光源多、Light Grid 填充率高、不透明物体 overdraw 叠光照0024（Light Grid）、0025（光照）
AmbientOcclusionSSAO 半径/采样数配置高、分辨率未降频0027
Reflections / ScreenSpaceReflectionsSSR 步进步数多、屏幕空间覆盖率大0027
PostProcessingTAA 开销（速度缓冲、历史帧重投影）、Bloom、Tonemap0031–0035
Translucency半透明物体多、overdraw 严重（无 Early-Z）0029
VolumetricFogFroxel 网格分辨率高、体积雾范围大0030
关键洞见 · BasePass 贵未必是材质的问题

新手常见误判：BasePass 耗时高 → 立刻去简化材质。但 BasePass 的时间来自三个完全不同的来源：①着色器 ALU（材质数学）②纹理带宽（采样数×贴图分辨率）③ overdraw（Depth Prepass 没有有效剔除）。正确做法是先用 r.BasePassForceFullyRoughSurface 1 强制最简材质——如果 BasePass 时间没有显著下降，说明瓶颈不在材质着色器，而在带宽或 overdraw。再用 viewmode overdraw（编辑器 Viewport → View Mode → Shader Complexity）目测 overdraw 区域。

5stat SceneRendering 辅助统计

stat GPU 看的是纯 GPU 耗时；stat SceneRendering 补充了渲染线程侧的 CPU 统计，包括 draw call 数量、三角形数量、动态网格数等：

stat SceneRendering

关注以下几行：

统计项含义调优方向
DrawPrimitiveCalls渲染线程实际发出的 draw call 总数高 → 检查合批（0020）、GPU Scene 实例化
MeshDrawCalls通过 FMeshDrawCommand 提交的网格绘制数（包含各 Pass）高 → 查各 Pass 是否有冗余 draw
DynamicPathMeshDrawCalls走动态路径（不可缓存）的 draw 数，这些每帧重新构建开销更高尽量减少、改走静态缓存路径（0019）
Primitives当前场景里的图元总数（FScene 里）高本身不是问题，关键看剔除后有多少可见
VisibleStaticMeshElements剔除后的可见静态网格数量应远小于 Primitives；若接近则剔除效率低（0017）

›

补充工具 · r.ProfileGPU.ShowLeafEvents

控制台变量 r.ProfileGPU.ShowLeafEvents 1 让 GPU Visualizer 显示更细粒度的叶节点（单个 draw call 级别）。默认为 0 只显示 Pass 级别。打开后时间轴会变得很密，但能精确定位到哪个网格的 draw 最贵。

7常见误读与陷阱

陷阱 1：GPU 时间 vs 帧时间
stat GPU 的 GPU 总时间不等于帧率对应的帧时间（1000/FPS）。帧时间由 CPU 和 GPU 里最慢的那条决定（见 0004）。GPU 总时间反映的是纯 GPU 工作量，与 CPU 侧无关。

陷阱 2：Pass 顺序不等于时间起点
在 GPU Visualizer 的时间轴上，Pass 的起点是 GPU 实际开始执行的时刻。由于 GPU 可能在等待之前 Pass 的资源屏障（见 0003、0015），两个 Pass 的时间块可能有间隙——这些间隙是 GPU 空闲等待，也是优化目标（RDG 的异步计算调度试图填满这些间隙）。

陷阱 3：VSync 把 GPU 时间"撑大"
如果开了 VSync，GPU 在完成所有工作后会等下一个 VBlank。这段等待时间会出现在 stat GPU 的总 GPU 时间里（以一个宽 Present 块的形式），让你误以为 GPU 工作量很重。诊断时应先关 VSync（r.VSync 0）再测量。

检索练习

stat GPU 里各 Pass 耗时之和大于总 GPU 时间，最可能的原因是什么？

说出 UE 4.26 里 GPU 时间统计的两套宏（RDG Pass 用的和非 RDG Pass 用的），以及为什么 GPU 时间结果总是延迟一帧可用。

RDG Pass 用 RDG_GPU_STAT_SCOPE(GraphBuilder, Name)；非 RDG Pass 用 DECLARE_GPU_STAT(Name) 注册 + SCOPED_GPU_STAT(RHICmdList, Name) 标记。两者最终都通过 FRealtimeGPUProfiler 在命令列表里插入 GPU 时间戳查询对。

延迟一帧的原因：GPU 时间戳查询结果必须等 GPU 实际执行完那段命令后才写入查询堆；CPU 读取查询结果需要等 GPU 完成，而 CPU 与 GPU 异步流水线化运行（0004），所以读取只能在下一帧（或更晚）进行，否则会强制 CPU 等待 GPU 完成——打断流水线。延迟一帧使用查询结果避免了这个同步点。

带走这几条

  • ProfileGPU 触发单帧 GPU Visualizer；stat GPU 持续刷新各 Pass 耗时——两者结合是 GPU 性能分析的主入口。
  • GPU 时间来自 FRealtimeGPUProfiler 插入的时间戳查询对，读取结果延迟一帧，与遮挡查询同理。
  • RDG_GPU_STAT_SCOPE（RDG Pass）和 SCOPED_GPU_STAT（非 RDG）是 Pass 时间出现在统计里的源码机制——grep 名字可定位实现。
  • 各 Pass 时间热点与本课程 Part 5–6 的 Pass 直接对应：BasePass贵→看材质/overdraw；ShadowDepths贵→看动态阴影投射物；Lights贵→看动态光源数量。
  • 诊断时先关 VSync（r.VSync 0），再用 stat GPU 确定热点 Pass，再用 RenderDoc 深入细节——是标准三步流程。
