Part 5 · 延迟着色 · Lesson 0025

直接光照：延迟光照通道

延迟着色的承诺在这里兑现：RenderLights 逐光源读 GBuffer、做 BRDF，用模板/裁剪把着色开销严格锁在光照影响区域内，阴影遮罩整合进来，最终把直接光贡献累加到 SceneColor。

时长 ~20 分钟 前置 0021（GBuffer / ShadingModelID）、0024（光照剔除）、0026（阴影，前置可弱） 产出 能描述 RenderLights 如何逐光源读 GBuffer 做 BRDF，并理解模板/裁剪优化与阴影遮罩整合

GBuffer 里已有每像素的 WorldNormal、BaseColor、Roughness、Metallic 和 ShadingModelID；SceneDepth 可以反推世界坐标。现在只需对每盏动态光执行一次"读 GBuffer → 按 ShadingModel 求 BRDF → 写 SceneColor"的 Pass，就能完成直接光照。光照成本 ∝ 光源数 × 受影响像素数，与几何三角形数无关——这正是延迟着色在 0021 里许下的承诺。

1代码入口：RenderLights

FDeferredShadingSceneRenderer::RenderLights（LightRendering.cpp）按光源类型分发：方向光全屏、局部光按包围体。核心着色器是 FDeferredLightVS + FDeferredLightPS（DeferredLightPixelShaders.usf）。

// LightRendering.cpp（架构示意）
void FDeferredShadingSceneRenderer::RenderLights(
    FRDGBuilder& GraphBuilder, ...)
{
    for (auto& LightSceneInfo : Scene->Lights)
    {
        if (LightSceneInfo.LightType == LightType_Directional)
            RenderDirectionalLight(GraphBuilder, LightSceneInfo, ...);
        else
            RenderLocalLight(GraphBuilder, LightSceneInfo, ...); // 点/聚光
    }
}

2FDeferredLightPS 的执行流程

DeferredLightPixelShaders.usf 里的 PS 对每个覆盖片元做：

// DeferredLightPixelShaders.usf（核心路径，简化）
void DeferredLightPixelMain(... out float4 OutColor : SV_Target0)
{
    // 1. 从 SceneTextures 读 GBuffer
    FGBufferData GBuffer = GetGBufferDataFromSceneTextures(ScreenUV);

    // 2. 从 SceneDepth 重建世界坐标
    float3 WorldPosition = ReconstructWorldPosition(ScreenUV, SceneDepth);

    // 3. 按 ShadingModelID 分发 BRDF
    FDirectLighting Lighting = SurfaceShading(
        GBuffer,          // 材质属性
        LightData,        // 当前光源参数
        WorldPosition,
        CameraDirection);
    // SurfaceShading 内部 switch(GBuffer.ShadingModelID) → ShadingModels.ush

    // 4. 乘以阴影遮罩（来自 Shadow Pass，0026）
    Lighting.Diffuse  *= ShadowMask;
    Lighting.Specular *= ShadowMask;

    // 5. 累加进 SceneColor（Additive blend）
    OutColor = float4(Lighting.Diffuse + Lighting.Specular, 0);
}

SurfaceShading（ShadingModels.ush）根据 ShadingModelID switch 到正确分支：DefaultLit 调用 GGX_Specular + Lambert_Diffuse（BRDF.ush），ClearCoat 在此基础上多一层 clear coat 项，SubsurfaceProfile 走屏幕空间次表面散射……每种模型读取的 GBufferD 字段也不同。

3模板 + 裁剪：局部光的精确开销控制

如果对一盏影响范围 2 米的点光全屏跑 PS，99% 的像素都会做无用功。UE 用两级机制把 PS 调用锁在有效区域：

Scissor 裁剪把光源包围球/锥体在屏幕上的 2D AABB 设为 scissor rect。光源包围体屏幕外的矩形区域直接跳过光栅化，无 PS 调用。
Stencil 模板先用 stencil pass 在深度测试下把"位于光源包围体内部"的像素标记（stencil write），再用 stencil test 在光照 PS 里只处理被标记的像素。精确到每像素，排除同一 scissor 矩形内却在包围体外的像素。

对于极大或极近的光源（包围体覆盖大部分屏幕），stencil 的 pass 本身开销不划算——UE 会切换到全屏 Pass（bDrawSimpleLights 路径），直接在 PS 里做距离衰减提前 discard。

4方向光：全屏 Pass

方向光无衰减范围，所有可见像素都受影响，走全屏 quad 直接 PS，不需要 stencil。方向光同时是阴影遮罩（CSM，0026）最主要的消费者——阴影 Pass 把每像素阴影因子写进屏幕空间阴影遮罩纹理，RenderLights 读这张纹理乘进漫反射/镜面反射结果。

5附加特性：光照函数与 IES

UE 4.26 支持每盏光关联一个光照函数材质（Light Function）——在光照 PS 里额外采样一张纹理或执行材质逻辑，对光照结果做 mask/调制（如窗户投影）。IES Profile 是另一种修饰：从 .ies 文件烘焙成 1D/2D 纹理，描述光源的真实角度分布，光照 PS 按出射方向采样此纹理作为强度系数。两者都在 FDeferredLightPS 里作为 permutation 编译进去，非零成本但不影响主流程。

6Tiled / Clustered Deferred 路径

UE 4.26 也支持 Tiled Deferred 路径（r.TiledDeferredShading=1）：用计算着色器把屏幕分 tile，每 tile 内对所有光源做剔除+着色，省去逐光 Pass 的 draw call overhead。这是对局部光数量极多时（数百盏）的优化路径，核心数学与逐光 Pass 相同，差别是批量执行。

为什么说"延迟在此兑现"

0021 讲过延迟着色的核心：光照成本与几何无关。这里是物理体现：RenderLights 对每盏光只遍历受影响的像素，不管场景里有多少三角形——哪怕场景有一百万个多边形，只要它们的 GBuffer 已经写好，光照 Pass 的代价完全由"光源数 × 受影响像素数"决定。

检索练习

延迟光照 Pass 里，FDeferredLightPS 如何确定某像素应该用哪种 BRDF（如 DefaultLit 还是 Subsurface）？

说出延迟光照 Pass 完成一盏局部光着色的完整步骤（从 stencil 到 SceneColor），并解释阴影遮罩是如何整合进来的。
① Scissor：用光源包围体屏幕 AABB 设裁剪矩形，排除屏幕外区域。② Stencil Pass：在深度测试下标记位于包围体内部的像素。③ Deferred Light PS：读 GBufferA/B/C/D + SceneDepth 重建世界坐标，调用 SurfaceShading 按 ShadingModelID switch 到正确 BRDF（漫反射+镜面），计算光源贡献。④ 阴影整合：Shadow Pass（0026）已把每像素阴影因子写入屏幕空间阴影遮罩纹理，光照 PS 采样此纹理直接乘进漫反射/镜面结果。⑤ Additive Blend：把光照贡献叠加到 SceneColor——每盏光叠加一次，最终 SceneColor 包含所有直接光之和（加上 BasePass 写入的 Emissive）。

带走这几条

  • 入口：FDeferredShadingSceneRenderer::RenderLights（LightRendering.cpp）；核心着色器：FDeferredLightPS（DeferredLightPixelShaders.usf）。
  • 每盏光：读 GBuffer → 重建世界坐标 → SurfaceShading（按 ShadingModelID switch）→ 乘阴影遮罩 → Additive 累加 SceneColor。
  • 局部光双级优化：Scissor（2D AABB）+ Stencil（逐像素包围体测试），把 PS 调用严格锁在影响区域。
  • BRDF 实现在 ShadingModels.ush 和 BRDF.ush；阴影遮罩来自 Shadow Pass（0026），以屏幕空间纹理传入。
  • 延迟着色在此兑现：光照成本 ∝ 光源数 × 受影响像素，与场景几何复杂度彻底解耦。
