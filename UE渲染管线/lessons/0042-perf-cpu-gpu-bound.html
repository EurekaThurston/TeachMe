性能方法论：CPU-bound vs GPU-bound

你已经走完了整条渲染管线。这节课是终章：从 stat unit 开始，建立一套严格的诊断流程——先定位瓶颈线程，再根据线程方向找根因，最后对应到本课程每一节课学过的系统上开处方。

时长 ~25 分钟 前置 整门课程（尤其 0004、0005、0014–0020、0041） 产出 能独立走完诊断流程：stat unit → 定位瓶颈 → stat scenerendering/stat initviews/stat gpu → 找根因 → 开处方

性能优化最大的陷阱不是"不知道怎么优化"，而是"在错误的地方优化"。把 GPU 着色器简化了三天，结果瓶颈根本在 CPU 的 draw call 提交上——帧率纹丝不动。这节课的核心是一张诊断决策树：用几条 stat 命令把瓶颈精确定位到线程和系统，再调出对应章节的知识开处方。

1起点：stat unit 读三线程计时

0004 讲过 UE 的三级流水线：游戏线程（Game）、渲染线程（Draw）、GPU。stat unit 把这三条传送带各自的耗时打到屏幕上：

stat unit
四个数的确切含义

stat unit 显示四行：

  • Frame：从游戏线程帧开始到帧结束的实际墙钟时间（约等于 1000/FPS ms）。
  • Game：游戏线程（UWorld::Tick、Actor Tick、物理、动画、Gameplay）的耗时。
  • Draw：渲染线程的耗时——从接到"渲染这帧"的命令，到把 RHI 命令全部提交完毕（包括 InitViews、RDG 构建与执行录制）。
  • GPU：GPU 实际执行这帧所有命令的耗时（来自 GPU 时间戳，见 0041；结果延迟一帧）。

还有第五行 RHIT（RHI Thread）——RHI 线程将平台无关命令翻译成 D3D12 调用的耗时。大多数场景下 RHIT 不是瓶颈，但在 D3D12 下 PSO 编译、描述符堆操作偶尔会使其突出。

判定规则：

Game 最大游戏线程瓶颈
Gameplay / Tick / 物理
›
Draw 最大渲染线程瓶颈
Draw Call / 剔除 / 材质编译
›
GPU 最大GPU 瓶颈
着色器/带宽/overdraw

Frame ≈ max(Game, Draw, GPU)（见 0004）。如果 Frame 明显大于三者，说明存在线程等待——游戏线程在等渲染线程、或渲染线程在等 GPU 完成（围栏，见 0004）。

关键洞见 · 降分辨率实验

不确定是 Draw-bound 还是 GPU-bound？在控制台输入 r.ScreenPercentage 50（降到 50% 分辨率）观察帧率变化：帧率明显上升 → GPU-bound（像素工作减少了）；帧率几乎不动 → Draw-bound 或 Game-bound（瓶颈不在像素）。反向实验：r.ScreenPercentage 200 强制超采样，若帧率显著下降则更确认 GPU-bound。

2渲染线程瓶颈（Draw-bound）：stat SceneRendering 与 stat InitViews

若 stat unit 里 Draw 最大，进一步用：

stat SceneRendering    // draw call 数量、三角形数、动态 mesh 数
stat InitViews         // 可见性计算（视锥剔除、HZB 遮挡剔除）的 CPU 耗时

2.1 draw call 过多

stat SceneRendering 里的 DrawPrimitiveCalls 或 MeshDrawCalls 异常高（典型问题阈值因平台而异，PC 上几千是正常的，几万就要开始担心）。

根因与对应章节：

根因对应章节处方
材质变体太多，FMeshDrawCommand 无法合批0019（MeshPassProcessor 与命令缓存）减少材质变体、复用材质实例
GPU Scene / 实例化未启用，同类网格反复单独提交0020（GPU Scene 与实例化）确认 r.GPUScene.UploadEveryFrame 等开关、用实例化静态网格
DynamicPathMeshDrawCalls 占比高（每帧重新构建命令）0019（缓存 vs 动态路径）让更多 draw 走缓存路径，避免强制动态路径的材质特性
场景里小网格碎片化过多0017（可见性剔除）合并网格（Merge Actors）、HLOD 聚合远处物体

2.2 可见性 / 剔除开销高

stat InitViews 里的 FrustumCull、OcclusionCull 耗时高，说明场景图元数量多或 HZB 遮挡查询开销大：

根因对应章节处方
视锥剔除通过率低（大量物体在视锥内但实际遮挡）0017（HZB 遮挡剔除）确认 HZB 剔除已启用（r.HZBOcclusion 1）、设置合理的 MinDrawDistance
PrecomputedVisibility 未烘焙（室内场景）0017为室内/封闭空间烘焙 Precomputed Visibility Volume
场景 Primitive 数量本身过大0009（FScene 与 SceneProxy）按可见距离分层、用 LOD、用 HLOD

3GPU 瓶颈（GPU-bound）：stat GPU + 二次细分

若 stat unit 里 GPU 最大，进入 0041 讲的 GPU 时间统计体系：

stat GPU

找到时间最重的 Pass，然后按下面的决策树往下钻：

3.1 BasePass 过重

Base Pass（0023）是主要的材质着色 Pass，贵可能来自三个不同地方，要逐一排除：

  1. ALU 瓶颈（着色器数学）：r.BasePassForceFullyRoughSurface 1 强制最简材质，若 BasePass 时间显著下降 → 材质着色器复杂是根因 → 简化节点数、减少 Texture Sample 个数。
  2. 带宽瓶颈（纹理采样）：时间在 r.BasePassForceFullyRoughSurface 1 后下降有限，但纹理分辨率高/采样次数多 → 检查贴图 MIP 设置、纹理压缩格式（BC7 vs BC1）、LOD Bias。
  3. Overdraw（Depth Prepass 效果差）：用 viewmode overdraw 查看过度绘制热图。过度绘制高说明 Depth Prepass 没能有效剔除被遮挡片元 → 检查是否有大量使用 Masked 材质（Mask discard 阻止 Early-Z 优化，见 0022）。
警告 · Masked 材质的 Depth Prepass 代价

Masked 材质（带 Opacity Mask、调用 clip() / discard）会阻止 GPU 的 Early-Z 优化（见 0001 / 0022）——因为 PS 执行前不知道这个片元是否会被 clip 掉，不能提前做深度测试。UE 4.26 为 Masked 材质单独在 Depth Prepass 里跑一遍（用 EarlyZ/TwoSidedSign 变体），但如果 Masked 物体很多，Prepass 本身也会变贵，且 BasePass 的 Early-Z 效益减弱。如果场景大量使用 Masked 材质且性能差，考虑改用 Dithered LOD Transitions 或减少 Masked 数量。

3.2 ShadowDepths 过重

对应 0026（CSM 与阴影深度图）。动态阴影的开销主要来自：

  • 每个动态投影光源都需要额外的 ShadowDepth Pass——减少动态投影光源数量或缩小其影响半径。
  • CSM 级联数量（r.Shadow.CSM.MaxCascades）和分辨率（r.Shadow.MaxResolution）——降低可显著减少开销，代价是阴影质量。
  • 远距离物体仍在投影——设置 Dynamic Shadow Distance 限制阴影投影范围。

3.3 Translucency 过重

对应 0029（半透明前向渲染）。半透明物体无 Early-Z，天然 overdraw 严重：

  • 减少屏幕上半透明物体覆盖的像素面积。
  • 粒子特效：降低粒子数量、用 r.ParticleLODBias 强制 LOD、把粒子贴图改为更简单的 Unlit 材质。
  • 启用 r.SeparateTranslucency 0 禁用独立半透明缓冲（会有质量代价，但省一次全分辨率 Pass 开销）。

3.4 PostProcessing 过重

对应 0031–0035。常见重点：

  • TAA（0031）：r.TemporalAA.Upsampling 0 禁用 TAA 上采样，或切换为 FXAA（质量下降但极快）。
  • Bloom（0032）：r.Bloom.Quality 1 或 0 降低 Bloom 质量，大幅降低 PostProcess 开销。
  • AmbientOcclusion（0027）：r.AmbientOcclusion.MaxQuality 0 禁用或 r.AmbientOcclusionSampleCount 降低采样数。
  • SSR（0027）：r.SSR.Quality 0 禁用或降级。

4stat RHI：命令翻译层的成本

若 RHIT 线程时间突出，用：

stat RHI

关注 DrawPrimitiveCalls（RHI 层实际 draw call 数）和 TrianglesDrawn（三角形数量）。RHI 线程瓶颈的典型根因是：

  • PSO 缓存未命中（首次使用的材质/着色器组合触发实时 PSO 编译）——预热 PSO 缓存（r.ShaderPipelineCache.Mode 2）。
  • 描述符堆操作过多（D3D12）——通常由 draw call 数量驱动，处方同 2.1。

关键洞见 · 性能预算而非事后救火

这套诊断流程的真正价值不在于"项目快上线了帧率不够再来救"，而在于建立性能预算意识：早期就用 stat unit + stat GPU 量化每个系统的成本，并与目标平台的帧时间对比。一旦某个 Pass 的 GPU 时间超过预算（例如：目标 30fps 的手机项目，BasePass 超过 4ms），立刻追查，而不是等到所有系统堆完再整体优化。每个系统的成本模型，你在对应章节里都学过——这节课只是把量尺拿出来。

7终章：一帧的闭环

从第 0001 课的"GPU 拿到三角形之后做了什么"，到现在用 stat unit 看着三个数字判断今天的帧率问题出在哪里——你走过的是同一条路，只是方向从"原理→工具"变成了"工具→原理"的反向溯源。

再走一遍这条路，这次每个环节都有名字：

游戏线程FEngineLoop::Tick → UWorld::Tick → ENQUEUE_RENDER_COMMAND → BeginRenderingViewFamily ｜ 对应：stat Game，Part 2
渲染线程FDeferredShadingSceneRenderer::Render → InitViews（剔除/FMeshDrawCommand/GPU Scene）→ RDG 构建 → 逐 Pass 录 RHI 命令 ｜ 对应：stat Draw，Part 3–4
RHI 线程把 RHI 命令翻成 D3D12 CommandList → Submit Queue ｜ 对应：stat RHIT，Part 7
GPUPrePass → BasePass（GBuffer）→ ShadowDepths → Light Grid → Lighting → SSAO/SSR → Translucency → PostProcess → Present ｜ 对应：stat GPU，Part 5–6
检索练习

stat unit 显示 Game=8ms、Draw=22ms、GPU=11ms、Frame=23ms。最可能的瓶颈是什么？下一步应该用哪个 stat 命令深入？

不看表，从 stat unit 的四个数出发，描述完整的诊断决策路径：Draw-bound 时下一步是什么，GPU-bound 时下一步是什么，以及把 BasePass 耗时高进一步细分的三步排查顺序。

Draw-bound（Draw 最大）：用 stat SceneRendering 查 DrawPrimitiveCalls（draw call 数量）和 DynamicPathMeshDrawCalls（动态路径占比）；用 stat InitViews 查视锥/遮挡剔除的 CPU 耗时。若 draw call 高 → 看 0019/0020 合批与 GPU Scene；若剔除 CPU 高 → 看 0017 HZB 配置与 Precomputed Visibility。

GPU-bound（GPU 最大）：用 stat GPU 找最贵的 Pass；用 ProfileGPU 获取单帧精确时间树；用 r.ScreenPercentage 50 确认确实是 GPU-bound（帧率应提升）。找到热点 Pass 后用 RenderDoc（0040）深入该 Pass 的 draw call 和着色器。

BasePass 耗时高的三步排查：① 用 r.BasePassForceFullyRoughSurface 1——若时间大幅下降则是 ALU/材质复杂度问题，简化材质节点；② 若时间无明显变化，用 viewmode overdraw 检查过度绘制——若有明显红色区域，查 Masked 材质数量（0022）或 Depth Prepass 是否有效；③ 若 overdraw 不严重，考虑带宽瓶颈——检查纹理分辨率、MIP 设置、压缩格式。

带走这几条

  • stat unit 的 Game/Draw/GPU 三个数是诊断的入口；Frame ≈ max(三者)，瓶颈就是最大的那条线程。
  • Draw-bound → stat SceneRendering（draw call 数）+ stat InitViews（剔除 CPU 开销），根因在 Part 4（0017–0020）。
  • GPU-bound → stat GPU + ProfileGPU 找热点 Pass，每个 Pass 的根因都在 Part 5–6 对应章节。
  • BasePass 高要区分 ALU/带宽/overdraw 三个不同原因——r.BasePassForceFullyRoughSurface 1 是快速区分 ALU vs 其他的方法。
  • 性能优化的最大陷阱是在错误的地方优化。先 stat unit 定位，再向下钻，最后开处方——顺序不能反。
  • 这节课是整门课程的闭环：每一个性能问题最终都可以追溯到这门课某一节讲过的系统和机制。
