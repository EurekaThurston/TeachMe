Part 0 · 基础 · Lesson 0004

CPU 与 GPU 的协作：异步、围栏、多缓冲

CPU 录命令、GPU 放命令，两者在各自的时钟上异步奔跑。这节课讲它们如何不打架地协作——以及由此引出的那个贯穿全课程的性能问题：到底是 CPU 拖后腿还是 GPU 拖后腿？

时长 ~15 分钟 前置 0003 命令列表/队列 产出 能解释帧流水线、围栏同步、CPU-bound vs GPU-bound 的判定逻辑

把 CPU 和 GPU 想成两条独立的传送带。CPU 这条不停生产"命令列表"，GPU 那条不停消费它们。它们速度不同、不同步。两个工程问题立刻冒出来：① GPU 还在用上一帧的数据时，CPU 能不能就去改它？② CPU 怎么知道 GPU 到底做完没？答案分别是多缓冲和围栏。把这两件事理顺，你就理解了帧的节奏。

1两条时间线，落后一帧的流水线

如果 CPU 提交一帧后傻等 GPU 画完再做下一帧，两条传送带就总有一条在空转，吞吐减半。正确做法是流水线化：CPU 准备第 N+1 帧时，GPU 同时在画第 N 帧。

CPU帧 N  |  帧 N+1  |  帧 N+2  |  帧 N+3
GPU       |  帧 N     |  帧 N+1  |  帧 N+2

2多缓冲：让两帧的数据不打架

流水线化立刻引爆问题①：GPU 还在读第 N 帧的常量缓冲（相机矩阵、逐物体数据），CPU 已经想往里写第 N+1 帧的值了。直接写 = 改坏正在用的数据（数据竞争）。

解法是多缓冲（double / triple buffering）：每帧会被 CPU 改写的资源，准备 N 份副本，第 N 帧用第 N % 副本数 份。这样 CPU 写"下一份"时，GPU 还在读"这一份"，互不干扰。

// 概念：每帧轮换写哪一份
ringIndex = frameNumber % 3;        // 三缓冲
perFrameCB[ringIndex].Update(viewMatrix, time…);
cmd.SetConstantBuffer(perFrameCB[ringIndex]);

UE 的动态/瞬态常量缓冲、上传堆都基于这种环形多缓冲

交换链的后缓冲同样是多缓冲的：GPU 画进一张后缓冲时，显示器可能正在扫描另一张。后缓冲数量直接关系撕裂与延迟，0038 详谈。

3围栏：CPU 唯一能"看见" GPU 进度的窗口

问题②：提交 ≠ 完成。CPU 怎么确认第 N-2 帧的 GPU 工作真的结束、那份缓冲可以安全复用？靠围栏（fence）——一个 GPU 能写、CPU 能读的整数。

CPU提交帧 N
记 fence=N
→
GPU执行完帧 N
Signal(fence)=N
→
CPU要复用帧 N 的缓冲
先 Wait(fence≥N)

围栏是单向的进度信号：GPU 走到某点就把 fence 设成某值，CPU 通过比较 fence 值判断"GPU 是否已越过那一点"。三缓冲下，CPU 通常只在"领先 GPU 超过 3 帧"时才会真的等 fence——正常情况下根本不阻塞。围栏也用于队列间同步（如异步计算队列与图形队列，见 0002/0016）。

回读为什么贵 · 强制同步

想把 GPU 算的结果读回 CPU（遮挡查询结果、GPU 统计、截屏）必然要等 GPU 真的算完——这是一次强制 CPU↔GPU 同步，会打断流水线、让 CPU 干等。所以 UE 的硬件遮挡查询结果总是延迟一帧使用（这一帧发查询、下一帧才读结果），就是为了不打断流水线。凡是"GPU→CPU 回读"的设计，都要假定它慢且滞后。

4谁拖后腿：CPU-bound vs GPU-bound

这是你的"性能优化"目标最核心的一张思维图。在理想流水线下，一帧的实际耗时约等于两条传送带里慢的那条：

帧时间 ≈ max(CPU 一帧耗时, GPU 一帧耗时)
瓶颈症状典型原因优化方向
CPU-boundGPU 常在等 CPU 喂命令；降分辨率不提帧率Draw Call 太多、可见性/剔除、游戏逻辑、Tick合批、剔除、降低 draw 数、并行录制
GPU-boundCPU 常在等 GPU；降分辨率能提帧率过度绘制、像素着色重、带宽、后处理降 overdraw、简化材质、降分辨率/特效
关键洞见 · 一个免费的诊断实验

怀疑瓶颈在哪？把渲染分辨率砍一半看帧率变化：帧率明显上升 → GPU-bound（像素工作量减少了）；帧率几乎不动 → CPU-bound（瓶颈不在像素）。反过来，把场景物体数量砍一半主要影响 CPU（Draw Call）。UE 里用 stat unit 看 Game / Draw / GPU 三个数，谁最大谁是瓶颈——这是 Part 8 性能实战的起点，但原理就在本节。

5映射到 UE：三级流水线预览

本课讲的是 CPU↔GPU 两级。UE 把 CPU 侧又拆成两级线程，于是是三级流水：

游戏线程帧 N：Tick、产出渲染数据
渲染线程帧 N-1：组织场景、录 RHI 命令
GPU帧 N-2：执行、上屏

所以 stat unit 里的 Game（游戏线程）、Draw（渲染线程）、GPU 三个数，正是这三条传送带各自的耗时，帧率由最慢的决定。下一课（0005）正式展开这三条线程的全景。

检索练习

你把游戏的渲染分辨率从 1080p 降到 540p，帧率几乎没变。最可能的结论是？

为什么需要"多缓冲"？如果只用一份逐帧常量缓冲、又想让 CPU 与 GPU 流水线并行，会发生什么？
流水线并行意味着 GPU 还在画第 N 帧时，CPU 已在准备第 N+1 帧。若逐帧常量缓冲只有一份，CPU 写入新值时会覆盖掉 GPU 正在读取的旧值，造成数据竞争——画面错乱或闪烁。要么 CPU 必须等 GPU 读完才敢写（退化成串行、吞吐减半），要么准备多份副本轮换：CPU 写"下一份"，GPU 读"这一份"，互不干扰。后者就是双/三缓冲，UE 的动态常量缓冲与上传堆都基于它。

带走这几条

  • CPU 与 GPU 异步并行、流水线化：GPU 通常落后 CPU 一帧——这是吞吐设计，不是 bug。
  • 多缓冲让被逐帧改写的资源有多份副本，避免 CPU 写 / GPU 读同一份的数据竞争。
  • 围栏是 CPU 观察 GPU 进度的唯一窗口，用于安全复用资源与队列间同步。
  • GPU→CPU 回读强制同步、打断流水线——所以遮挡查询结果延迟一帧用。
  • 帧时间 ≈ max(CPU, GPU)；用降分辨率/降物体数实验 + stat unit 判定 CPU-bound vs GPU-bound。
