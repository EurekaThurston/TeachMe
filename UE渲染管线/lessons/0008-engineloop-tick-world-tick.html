FEngineLoop::Tick 深入：World Tick 与组件 Tick

游戏线程一帧的完整旅程：从引擎主循环入口，经 UWorld::Tick、Tick 分组调度，到帧末把脏渲染状态刷入 FScene，最终触发渲染。这条链路是 Part 2 其余所有课的地基。

时长 ~20 分钟 前置 0005（三线程总览） 产出 能描述从 FEngineLoop::Tick 到 UWorld::Tick、Tick 分组、再到帧末把渲染状态变更发往渲染线程的全过程

0005 告诉你游戏线程负责"决定场景内容"——这节课把这句话拆细。FEngineLoop::Tick 是整条链的第一个函数调用；它的职责不是算渲染，而是驱动世界里所有可能改变场景的逻辑——物理、动画、Gameplay——再把这些变化打包投递给渲染线程。理解这条链，你才能知道"一个组件移动了，渲染线程什么时候能看见"。

1调用链顶层：三层入口

主循环驱动是从 FEngineLoop::Tick()（LaunchEngineLoop.cpp）开始的。它做了很多杂活（性能采样、帧率平滑、热重载），但最核心的一步是把帧驱动权交给引擎对象：

// LaunchEngineLoop.cpp（精简示意）
GEngine->Tick(DeltaSeconds, bIdleMode);   // 进入 UGameEngine::Tick 或 UEditorEngine::Tick

GEngine 的类型在 PIE 下是 UGameEngine，在编辑器下是 UEditorEngine，但都重写了 Tick。UGameEngine::Tick 最终调用：

// UEngine::Tick 节选（Engine.cpp），简化
World->Tick(LEVELTICK_All, DeltaSeconds);   // LevelTick.cpp，驱动所有 Actor/Component

这三层——FEngineLoop::Tick → GEngine->Tick → UWorld::Tick——是游戏线程一帧真正的"主干道"。所有后面要讲的事都发生在 UWorld::Tick 的内部或其返回之后。

文件定位

FEngineLoop::Tick → Engine/Source/Runtime/Launch/Private/LaunchEngineLoop.cpp
UWorld::Tick → Engine/Source/Runtime/Engine/Private/LevelTick.cpp
这两个文件是游戏线程侧最值得反复翻阅的源码，4.26 调用关系在此都能追到。

2Tick 分组：ETickingGroup 与 FTickTaskManager

UWorld::Tick 不是把所有 Actor 线性 tick 一遍——它分成多个Tick 组，按依赖顺序执行，且组内可以并行。Tick 分组由枚举 ETickingGroup 定义（TickTaskManagerInterface.h）：

Tick 组含义典型用途
TG_PrePhysics物理模拟之前移动角色控制器、应用输入、动画预处理
TG_DuringPhysics物理模拟期间读物理但不干涉——物理子步在此并行
TG_PostPhysics物理模拟之后布娃娃、IK、物理读回结果
TG_PostUpdateWork最晚更新相机、后处理、需要其他组件最终状态的 Tick

调度由 FTickTaskManager（TickTaskManager.cpp）和 FTickFunction 驱动。每个可 Tick 的 Component/Actor 注册一个 FTickFunction，并声明它属于哪个组、依赖哪些其他 Tick Function。FTickTaskManager 在每帧建任务图，组内 Task 通过 TaskGraph 并行执行（在工作线程上，最终由游戏线程等待）。

关键洞见 · 并行 Tick 与线程安全

同一个 ETickingGroup 内，互不依赖的 FTickFunction 会在工作线程上并发执行——不是只有游戏线程在跑。这意味着：若你在 Tick 里读写同组其他组件的数据而没有声明依赖，就会数据竞争。UE 的 Tick 依赖系统（AddTickPrerequisite）正是为此设计。骨骼网格动画更新（USkeletalMeshComponent::TickComponent → TickPose → RefreshBoneTransforms）默认在 TG_PrePhysics 并行执行。

3骨骼网格动画在 Tick 里更新

动画不在渲染线程跑——它是游戏线程侧 Tick 的一部分。USkeletalMeshComponent 在 TickComponent 里调用 TickPose，然后 RefreshBoneTransforms 把 AnimInstance 运算出的骨骼矩阵写进组件的本地 bone array。这些结果随后在帧末被"刷"进渲染线程侧的代理——细节见第 4 节。

4帧末同步：SendAllEndOfFrameUpdates 与 MarkRenderStateDirty

一帧 Tick 结束后，游戏侧有一大批组件"脏了"——变换变了、材质参数变了、骨骼姿势变了——但渲染线程还不知道。这时 UWorld::Tick 在帧末执行：

// LevelTick.cpp，帧末
World->SendAllEndOfFrameUpdates();   // 把所有 MarkRenderStateDirty 的组件刷到 FScene

SendAllEndOfFrameUpdates 遍历帧内所有调用过 MarkRenderStateDirty() / MarkRenderTransformDirty() 的组件，对每个组件调用相应的更新函数（如 FScene::UpdatePrimitiveTransform），内部通过 ENQUEUE_RENDER_COMMAND 把变更投到渲染线程的命令队列。

随后，UGameViewportClient::Draw → FRendererModule::BeginRenderingViewFamily 把"渲染这一帧"踢给渲染线程——这是 0005 讲过的那个入口，渲染线程从此接管（见 0013）。

5MarkRenderStateDirty 的语义

组件并不在每次 setter 调用时立即通知渲染线程——那样一帧内多次修改就会多次推送。UE 的模式是：修改游戏侧数据时调用 MarkRenderStateDirty()（PrimitiveComponent.h），把组件加入"脏列表"。帧末 SendAllEndOfFrameUpdates 统一处理，保证同一帧内只推送一次、且在所有 Tick 完成后推送。

常见误区 · 直接读 UPrimitiveComponent 的渲染数据

渲染线程不能直接访问 UPrimitiveComponent。游戏线程的 setter → MarkRenderStateDirty → SendAllEndOfFrameUpdates → ENQUEUE_RENDER_COMMAND 这条链是强制路径。如果你在渲染线程里想读组件数据，必须通过它的 FPrimitiveSceneProxy 副本（见 0009）。跳过这条路会数据竞争。

检索练习

一个组件在 TG_PostPhysics Tick 里改了位置，渲染线程最早何时能看到这个变化？

不看笔记，说出 ETickingGroup 的四个分组和各自典型用途，以及为何同组 Tick 能并行而无需加锁。
四组：TG_PrePhysics（输入/移动，物理前）→ TG_DuringPhysics（读物理，物理并行）→ TG_PostPhysics（IK/布娃娃，物理后）→ TG_PostUpdateWork（相机，最晚）。并行安全的原因：同组内并行的 FTickFunction 之间不允许直接依赖（若有依赖需通过 AddTickPrerequisite 声明），因此彼此操作的数据不重叠，不需要加锁。声明了依赖的 Tick 会在依赖项完成后才被调度，保证顺序。

带走这几条

  • 游戏线程一帧的主干：FEngineLoop::Tick（LaunchEngineLoop.cpp）→ GEngine->Tick → UWorld::Tick(LEVELTICK_All, DeltaSeconds)（LevelTick.cpp）。
  • Tick 按 ETickingGroup 四组串行，组内 FTickFunction 可并行（TaskGraph）；声明依赖（AddTickPrerequisite）控制组内顺序。
  • 骨骼动画（USkeletalMeshComponent::TickPose → RefreshBoneTransforms）在游戏线程侧、TG_PrePhysics 内完成，结果帧末送往渲染线程。
  • 帧末 SendAllEndOfFrameUpdates 统一把 MarkRenderStateDirty 标记的组件通过 ENQUEUE_RENDER_COMMAND 推入渲染线程，游戏线程不直接写 FScene。
  • 这条链路是理解 0009–0011 中"游戏态如何变成渲染态"的前提。
