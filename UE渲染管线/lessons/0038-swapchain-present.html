交换链与 Present

后处理链结束后，最终的像素还在 GPU 内存里。Present() 才是让它出现在显示器上的那个调用。本课拆开 UE 4.26 里 FD3D12Viewport、DXGI 翻转模型（Flip Model）、VSync、RHIEndDrawingViewport 的完整路径。

时长 ~20 分钟 前置 0037（D3D12 RHI 后端）、0003（图形 API 契约） 产出 能说清交换链后缓冲轮转逻辑、翻转模型与旧 Blit 模型的区别、VSync 对 Present 的影响，以及 RHIEndDrawingViewport 在 UE 代码里的位置

渲染管线的最后一个 CPU 动作是 IDXGISwapChain::Present()。在此之前，你需要理解：GPU 写的是哪块缓冲、DXGI 的翻转模型如何实现零拷贝、VSync 标志如何改变 Present 的阻塞行为。这一切都封装在 UE 的 FD3D12Viewport 里。

1FD3D12Viewport：UE 对 IDXGISwapChain 的封装

FD3D12Viewport（Engine/Source/Runtime/D3D12RHI/Private/D3D12Viewport.h）是 UE 4.26 中一个可显示视口的完整表达，持有：

  • IDXGISwapChain3*——DXGI 交换链接口（3 版本支持 HDR 颜色空间查询）。
  • 后缓冲数组 BackBuffers[]——每个元素是一个 FD3D12Texture2D，对应交换链的一个 ID3D12Resource。
  • 当前后缓冲索引 CurrentBackBufferIndex，由 IDXGISwapChain3::GetCurrentBackBufferIndex() 查询。
  • Present 计数器和最后一次 Present 的 VBlank 时间戳（用于帧率统计）。
为什么是 IDXGISwapChain3 而不是 1 或 4

SwapChain1 引入了 Flip Model 和 Present1()（Windows 8+）。SwapChain3 在此基础上增加了 GetCurrentBackBufferIndex() 和 HDR 颜色空间支持（SetColorSpace1()），UE 4.26 的 HDR 输出路径依赖后者。SwapChain4 的 HDR Metadata API 在 4.26 中为可选扩展，核心路径仍以 SwapChain3 为基础。

2后缓冲轮转：三缓冲的逻辑

DXGI Flip Model 下交换链持有 N 个后缓冲（UE 默认 3 个，对应 NumSwapChainBuffers）。三缓冲的轮转状态：

后缓冲 0GPU 正在写
（当前帧）
›
后缓冲 1已 Present
等待扫描输出
›
后缓冲 2已显示完毕
可被下帧写入

每次 Present() 后，DXGI 把"当前后缓冲"推进显示队列，并把"下一个可用后缓冲"的索引告知应用。UE 通过 GetCurrentBackBufferIndex() 在每帧开始时拿到新索引，把对应的 FD3D12Texture2D 设为当前帧的渲染目标。

3翻转模型：DXGI_SWAP_EFFECT_FLIP_DISCARD

UE 4.26 在 Windows 10 上使用 DXGI_SWAP_EFFECT_FLIP_DISCARD（Flip Discard 模型）创建交换链。与旧的 DXGI_SWAP_EFFECT_DISCARD（Blit 模型）相比：

特性旧 Blit 模型翻转模型（Flip Discard）
Present 操作 把后缓冲内容复制到前缓冲（或 DWM 合成缓冲）；复制有 GPU 开销和延迟。 直接翻转缓冲指针——把后缓冲"提升"为前缓冲，零拷贝，延迟更低。
缓冲所有权 Present 后应用仍拥有缓冲，可继续写；但 DWM 可能同时读——需要额外同步。 Present 后该缓冲归 DXGI/DWM 所有直到显示完毕（"Discard"语义）；不得再写。
Windows 10 DWM 集成 DWM 每帧强制合成，独占全屏也走 DWM（Vista+），额外延迟一个 VBlank。 独占全屏 Flip 绕过 DWM 合成，直接写显示控制器扫描缓冲——最低延迟路径。
关键洞见 · 为什么"独占全屏"延迟最低

窗口模式下，DXGI 翻转的后缓冲仍须经过 DWM（Desktop Window Manager）与其他窗口合成；独占全屏时，DWM 把桌面合成权交给游戏，翻转直接到达显示控制器的扫描缓冲——跳过了 DWM 合成的一个 VBlank 延迟。这是竞技游戏追求"独占全屏"的核心原因。UE 的 r.FullscreenMode 0（独占全屏）与 1（无边框窗口）之间的延迟差异，本质就来自这里。

4VSync 与 Present 的阻塞语义

DXGI 的 IDXGISwapChain::Present(SyncInterval, Flags) 第一个参数 SyncInterval 控制 VSync 行为：

  • SyncInterval = 0：立即呈现，不等 VBlank——可能产生撕裂，但延迟最低。
  • SyncInterval = 1：等待下一个 VBlank 后呈现——无撕裂，帧率上限为刷新率，延迟增加 0–16ms（60Hz 时）。
  • SyncInterval = 2：每两个 VBlank 才呈现一次——相当于 30fps 锁定（60Hz 显示器）。

UE 4.26 的控制台变量 r.VSync（默认 0）映射到这个参数：

// D3D12Viewport.cpp — 概念简化
uint32 SyncInterval = RHIConsoleVariables::bSyncWithDWM ? 1 : 0;
// r.VSync=1 → SyncInterval=1；r.VSync=0 → SyncInterval=0
HRESULT hr = SwapChain->Present(SyncInterval, PresentFlags);

Engine/Source/Runtime/D3D12RHI/Private/D3D12Viewport.cpp — Present 调用的 SyncInterval 选择

r.VSync=0 + 无帧率上限 = CPU/GPU 空转

关闭 VSync 且不设帧率上限时，游戏和渲染线程会以 CPU 最高速度提交帧，可能达到数百帧/秒。GPU 工作量极重、温度上升，且输入延迟并未因为"帧率高"而随之等比降低（受限于显示器刷新率和扫描时序）。实践中配合 t.MaxFPS 或 NVIDIA Reflex / AMD CHILL 做帧率封顶更合理。

5RHIEndDrawingViewport：UE 侧的 Present 入口

UE 4.26 渲染线程在帧末调用 RHIEndDrawingViewport()（通过 FRHICommandListImmediate 录制成 RHI 命令，RHI 线程执行），最终到达 FD3D12DynamicRHI::RHIEndDrawingViewport()。其内部流程：

  1. 确保当前后缓冲处于 D3D12_RESOURCE_STATE_PRESENT 状态（插必要的 ResourceBarrier）。
  2. 刷新命令队列（提交当前帧所有录制的命令列表到 D3D12 命令队列）。
  3. 调用 IDXGISwapChain::Present(SyncInterval, PresentFlags)。
  4. 更新 CurrentBackBufferIndex（GetCurrentBackBufferIndex()），准备下帧。
  5. 在命令队列上 Signal 一个围栏值，供下帧开始时检查"上帧 GPU 工作是否完成"（防止写入 GPU 还在使用的后缓冲）。
渲染线程录制最后一帧命令 → RHIEndDrawingViewport(录制成命令节点)
RHI 线程执行命令链表 → 插 Present Barrier → 提交 D3D12 命令队列 → IDXGISwapChain::Present() → Signal Fence
GPU执行命令队列（异步）→ 完成后 Signal 围栏 → DXGI 把后缓冲送显示队列

6帧等待与后缓冲复用保护

三缓冲理论上允许 CPU 比 GPU 超前 2 帧提交工作，但必须在写入新一轮后缓冲之前确认 GPU 已经用完上一轮该缓冲。UE D3D12RHI 在每帧开始时（RHIBeginDrawingViewport()）等待对应后缓冲的上一次使用围栏：

// D3D12Viewport.cpp — 概念示意
void FD3D12Viewport::BeginDrawing()
{
    uint32 BackBufferIdx = SwapChain->GetCurrentBackBufferIndex();
    // 等待此后缓冲对应的上帧围栏（若 GPU 尚未完成则 CPU 在此阻塞）
    BackBufferFences[BackBufferIdx].WaitForFence(
        BackBufferFenceValues[BackBufferIdx]);
}

检索练习

DXGI_SWAP_EFFECT_FLIP_DISCARD 和旧 Blit 模型（DXGI_SWAP_EFFECT_DISCARD）的核心区别是什么？

r.VSync=0 时 Present 的 SyncInterval 是多少？这会带来哪些利弊？

r.VSync=0 时 SyncInterval=0，Present 立即返回不等 VBlank。好处：帧率不受显示器刷新率限制，当 GPU 帧时间<刷新周期时，输入延迟降低（新帧立刻呈现，不用排队等下一个 VBlank）。坏处：若帧切换发生在扫描输出进行中，显示器上半部分显示旧帧、下半部分显示新帧——即撕裂（Tearing）。此外，无帧率上限时 CPU/GPU 空转、功耗和温度上升，需配合 t.MaxFPS 等机制封顶。NVIDIA G-Sync 和 AMD FreeSync 可以在关闭 VSync 的同时避免撕裂（下节课 0039 详述）。

UE 的 RHIEndDrawingViewport 在 GPU 侧的后缓冲"资源状态"要求是什么？如果不满足会发生什么？

Present 前，后缓冲必须处于 D3D12_RESOURCE_STATE_PRESENT（也写作 D3D12_RESOURCE_STATE_COMMON）状态。整个渲染过程中后缓冲被用作渲染目标（D3D12_RESOURCE_STATE_RENDER_TARGET），最后一条绘制完成后，D3D12RHI 的 RHIEndDrawingViewport 会插入一个 ResourceBarrier(RT → Present) 过渡命令，然后才调用 IDXGISwapChain::Present。若不做这个过渡，D3D12 调试层会报验证错误，驱动行为未定义（可能崩溃或产生黑屏）。

带走这几条

  • FD3D12Viewport 持有 IDXGISwapChain3* 和后缓冲数组；每帧通过 GetCurrentBackBufferIndex() 确定本帧写入哪个后缓冲。
  • UE 4.26 Windows 10 默认使用 DXGI_SWAP_EFFECT_FLIP_DISCARD——指针翻转、零拷贝、独占全屏可绕过 DWM，延迟最低。
  • r.VSync 1 让 SyncInterval=1（等 VBlank，无撕裂）；r.VSync 0 让 SyncInterval=0（立即呈现，可能撕裂，延迟更低）。
  • RHIEndDrawingViewport：插 Present Barrier（RT→Present）→ 刷新命令队列 → 调用 IDXGISwapChain::Present() → Signal Fence。
  • 每帧开始时等待上一轮同一后缓冲的 GPU Fence，防止 CPU 超前覆写 GPU 还在使用的缓冲。
